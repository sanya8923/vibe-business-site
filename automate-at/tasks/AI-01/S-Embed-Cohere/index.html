Что меняем и зачем

Когда агент ищет ответ в базе знаний, он сначала превращает вопрос пользователя в вектор — цифровое представление смысла. Embedding-модель — это и есть тот инструмент, который делает превращение. По умолчанию мы используем модель OpenAI.

Проблема: OpenAI text-embedding-3-small обучалась преимущественно на английских текстах. Русский язык со всей его морфологией — «стол», «стола», «столу», «столы» — для неё сложнее. Cohere embed-v4.0 специально создавалась для 100+ языков и понимает русский так же хорошо, как английский.

⚠
Когда это проявляется
Агент находит не то или «не знает» ответа — хотя он есть в базе

Пользователь спрашивает «где находятся наши склады» — в базе есть документ про «склад» и «хранение». OpenAI embedding иногда не свяжет «склады» и «склад» как близкие по смыслу. Cohere embed-v4 с этим справляется — она понимает морфологию и синонимы на русском языке.

На бенчмарках: на русскоязычных запросах Cohere embed-v4 показывает на 15–20% лучший recall, чем OpenAI text-embedding-3-small.

🎓 Cohere — что это, какие модели, зачем разный input_type статья · ~3 мин ▾

Cohere — компания, которая делает языковые модели для бизнеса. Главный конкурент OpenAI и Anthropic. Их сильная сторона — мультиязычность: модели понимают 100+ языков примерно одинаково хорошо. У OpenAI русский всегда «второй сорт» после английского, у Cohere — наравне.

У Cohere три семейства моделей:

  • Embed — превращает текст в вектор (числовое представление смысла). Используется в RAG для поиска по смыслу. Это то, что мы будем менять в задаче.
  • Rerank — берёт топ-50 результатов поиска и переранжирует их по точности. Отдельный стретч S-Retrieval-Rerank.
  • Chat — генеративная модель для ответов агента (аналог gpt-4). Мы её сейчас не используем, у нас OpenAI.

Почему модель embed-v4.0

Это текущая флагманская embedding-модель Cohere для мультиязычного поиска. Что нам в ней важно:

  • +: Понимает русский на уровне с английским. Морфология, синонимы, разговорные формы — всё ок.
  • +: Поддерживает Matryoshka embeddings — можно укоротить вектор до 256/512/1024 чисел без сильной потери точности.
  • +: Отдельные режимы для документов и запросов (см. ниже про input_type).
  • −: Размерность 1024 (у OpenAI 1536) — нельзя смешивать векторы. Старую коллекцию придётся пересоздать.
  • −: Свободный лимит — 1000 запросов в месяц. Для тестов хватает, для прода нужен платный план.

Через 6–12 месяцев Cohere выпустит новую версию (v5, v6…). Логика выбора останется той же: лучший мультиязычный recall + поддержка input_type. Конкретный номер версии меняется, принцип нет.

Зачем разный input_type в WF1 и WF2

Cohere специально обучила модель так, чтобы документы и вопросы к ним «тяготели» друг к другу в векторном пространстве — но при условии, что они обрабатываются через правильный режим:

  • search_document — для индексации документов. Используется в WF1 (когда грузим файлы в KB).
  • search_query — для поисковых запросов. Используется в WF2 (когда пользователь спрашивает агента).

Если поставить везде search_document — вопрос «попадает» в другое облако векторного пространства, и расстояния до документов будут хуже. Recall падает на 5–10%.

Мнемоника: WF1 загружает документы → search_document. WF2 ищет по запросу пользователя → search_query.

Что меняется в воркфлоу

Два воркфлоу содержат sub-ноду Embeddings OpenAI. Меняем её на Embeddings Cohere — с важным отличием в параметре между WF1 и WF2.

В WF1 и WF2 параметр Input Type разный

В WF1 ставим search_document — это режим для индексации документов. В WF2 ставим search_query — режим для поискового запроса пользователя. Если поставить везде одинаково — качество поиска упадёт на 5–10%. Разбор причины — в статье «Cohere — что это» выше.

🔑 Как добавить Header Auth credential в n8n статья · ~3 мин ▾

Header Auth — универсальный тип credential в n8n для сервисов, которые принимают API-ключ через HTTP-заголовок Authorization (Cohere, Voyage, OpenRouter и многие другие).

  1. В левом меню n8n нажми Credentials.

  2. Нажми Add credential (кнопка в правом верхнем углу).

  3. В поиске набери Header Auth → выбери Header Auth.

  4. Заполни поля:

    • Credential name: <имя_сервиса> — задача-носитель скажет конкретное имя (например, Cohere API). Ноды будут искать credential по этому имени.
    • Name: Authorization — это имя HTTP-заголовка, его не меняй.
    • Value: Bearer <твой-api-key> — подставь ключ от сервиса. Пробел между Bearer и ключом обязателен.

  5. Нажми Save. Credential появится в списке и станет доступен во всех нодах.

Где взять API-ключ? Возможны два варианта:

  • Школьные сервисы (LiteLLM-прокси, школьный Qdrant Master, общие OpenAI/Anthropic) — ключ лежит в личном кабинете школы, раздел «Мои Credentials». Открой её — скопируй нужный ключ.
  • Внешние сервисы под твою задачу (Cohere, Voyage, своя OpenAI-подписка) — регистрируешься сам, ключ получаешь в дашборде сервиса. Это описано в шагах задачи.

Почему именно Header Auth? Это универсальный способ передать Authorization: Bearer ключ в любую HTTP-ноду. В некоторых версиях n8n есть встроенный тип под конкретный сервис (например, Cohere API) — если он есть, используй его, удобнее. Если нет — Header Auth работает одинаково хорошо.

🗝 Раздел «Мои Credentials» в личном кабинете школы статья · ~1 мин ▾

В личном кабинете школы (этой платформе) есть раздел «Мои Credentials». Там лежат школьные ключи и параметры доступа, которые ты можешь использовать в своих воркфлоу:

  • Qdrant Master — API-ключ для школьного Qdrant (для DELETE-запросов, прямых HTTP-вызовов).
  • OpenAI / Anthropic / DeepSeek — школьные ключи через LiteLLM-прокси.
  • student_id — твой идентификатор (например, c01), нужен для имени Qdrant-коллекции.
  • Postgres — реквизиты школьной БД для логов и chat memory.

Внешние сервисы (Cohere, Voyage, Mem0 и т.п.) под свои задачи студент регистрирует сам — их ключи в школьный раздел не попадают.

Если нужного ключа нет в «Мои Credentials» или ключ не работает — пиши в чат @profy_conveyor.

Шаги внедрения

1 Пересоздай Qdrant-коллекцию делать ПЕРВЫМ

Почему первым: Qdrant хранит коллекцию с фиксированной размерностью. OpenAI делал векторы на 1536 чисел, Cohere — на 1024. Эти два формата несовместимы. Если сначала поменяешь ноды, а потом попытаешься загрузить файл — Qdrant вернёт ошибку. Поэтому сначала удаляем старую коллекцию, создаём новую, а потом меняем ноды.

🧭 Что такое Qdrant и как с ним работать в n8n статья · ~3 мин ▾

Qdrant — векторная база данных. В обычной БД ты ищешь по точному совпадению («где имя = Иван»). В Qdrant ты ищешь по смыслу — «найди документы, похожие на этот текст».

Чтобы это работало, документы и запросы предварительно прогоняются через embedding-модель (OpenAI, Cohere, bge…). Модель превращает текст в вектор — список из 1024 / 1536 / 768 чисел. Похожий смысл = близкие векторы. Qdrant умеет быстро находить «ближайшие» векторы к заданному.

Структура хранения

  • Коллекция — это «таблица» в Qdrant. У неё фиксированная размерность векторов и метрика расстояния. Поменять размерность нельзя — только пересоздать коллекцию.
  • Точка (point) — одна запись внутри коллекции: id + вектор + произвольный JSON-payload (например, { "filename": "manual.pdf", "page": 4 }).
  • Search — отдай вектор запроса, получи топ-N ближайших точек с их payload'ами.

Ноды Qdrant в n8n

В стандартной поставке n8n работает с Qdrant через три «AI-родителя»:

  • Qdrant Insert — кладёт чанки текста в коллекцию (используется в WF ingestion). Подключает sub-ноды Embeddings (откуда брать векторы) и Document Loader (откуда брать текст).
  • Qdrant KB (как Tool у AI Agent) — даёт агенту инструмент «искать по базе знаний». Тоже подключает sub-ноду Embeddings — обязательно ту же модель, что и при Insert.
  • HTTP Request — для всего остального: создать/удалить коллекцию, посмотреть статистику, очистить точки. Прямые REST-вызовы Qdrant API.

Школьная инфра

Школа держит общий Qdrant-инстанс на https://qdrant.school.vibe-business.site. У каждого студента свои коллекции — имена изолированы префиксом student_id. Master-ключ для прямых HTTP-вызовов лежит в разделе «Мои Credentials» в личном кабинете.

🏷 Правила именования Qdrant-коллекций в школе статья · ~1 мин ▾

Все коллекции в школьном Qdrant именуются по шаблону:

<student_id>-<product>-<что_хранится>

Примеры:

  • c01-kb-agent — у студента c01, продукт AI-01 (RAG-агент), хранится база знаний.
  • c12-faq-bot — студент c12, продукт FAQ-бот.

Зачем: на одном Qdrant-инстансе живут коллекции всех студентов. Префикс student_id исключает пересечения. Никогда не используй чужой prefix — увидишь чужие данные или сломаешь чужую коллекцию.

Свой student_id найдёшь в «Мои Credentials» в личном кабинете. Имя коллекции, которое ты создал в базовом агенте AI-01, дублируется в трёх местах: WF1 (Qdrant Insert), WF2 (Qdrant KB sub-нода), Product Config. При смене embedding-провайдера достаточно сохранить то же имя коллекции — пересоздать её с новой размерностью.

📐 Почему нельзя смешивать векторы разной размерности статья · ~3 мин ▾

Когда ты создаёшь коллекцию в Qdrant, ты один раз говоришь: «в этой коллекции векторы будут размером 1536». Это прошивается в структуру БД навсегда — как тип колонки в SQL-таблице.

Embedding-модели выдают векторы конкретного размера. Это не случайность — размер определяет «сколько смысла» помещается в вектор:

  • OpenAI text-embedding-3-small → 1536 чисел
  • Cohere embed-v4.0 → 1024 числа (или меньше через Matryoshka)
  • bge-m3 → 1024 числа

Если ты попытаешься записать 1024-мерный вектор в коллекцию, созданную под 1536 — Qdrant откажет с ошибкой Wrong input: expected dim 1536, got 1024.

Что делать:

  1. Удалить старую коллекцию (все векторы пропадут).
  2. Создать новую с правильной размерностью. Это делается автоматически при первом insert'е через ноду Qdrant Insert.
  3. Перезалить все файлы KB через WF ingestion — теперь они запишутся с правильными векторами.

Бояться потери данных? Нет. KB — это производное от твоих исходных файлов. Файлы у тебя есть, WF1 умеет их грузить. Удаление коллекции = «очисти индекс», а не «удали документы».

🗑 Как удалить Qdrant-коллекцию через HTTP Request (с Import from cURL) статья · ~2 мин ▾

В n8n нет отдельной ноды «Qdrant Delete Collection» — используем универсальный HTTP Request. Сделать это можно за минуту через функцию Import from cURL.

  1. В любом sandbox-воркфлоу добавь ноду HTTP Request (или открой существующую). В правом верхнем углу настроек ноды нажми Import from cURL.

  2. Вставь в открывшееся поле curl-команду:

    curl -X DELETE \
  https://qdrant.school.vibe-business.site/collections/<имя_твоей_коллекции> \
  -H "api-key: <Qdrant Master ключ>"

    n8n автоматически распарсит curl и заполнит метод (DELETE), URL и заголовок.

  3. Подставь свои значения:

    • <имя_твоей_коллекции> — имя коллекции, которую ты создал в базовом агенте AI-01 (например, c01-kb-agent). Скопируй из своего Product Config.qdrant.collection.
    • <Qdrant Master ключ> — лежит в личном кабинете школы, раздел «Мои Credentials». Можно использовать сохранённый в n8n credential Qdrant Master (Generic Header Auth) — тогда поле Header не нужно заполнять вручную.

  4. Нажми Execute step. Ожидаемый ответ — JSON {"result": true, "status": "ok"}.

  5. Удали HTTP Request ноду или оставь в sandbox-воркфлоу — она нужна была одноразово, в продакшн-воркфлоу её не нужно.

Безопасность: DELETE действительно удаляет коллекцию — вместе со всеми векторами. Сохранённые в Qdrant payload'ы пропадут. Файлы-источники, из которых строилась KB, остаются у тебя — просто перезалей их.

Что конкретно сделать

Удали свою текущую коллекцию через HTTP Request — детали в раскрывайке выше.

Имя коллекции возьми из своего Product Config.qdrant.collection (то самое имя, которое ты задал в базовом агенте AI-01-02). После удаления оно останется тем же — просто следующий запуск WF1 пересоздаст коллекцию автоматически с новой размерностью.

2 WF1: Chat Trigger Ingestion Pipeline — замени Embeddings OpenAI → Cohere

Что делаем: в нодах Qdrant Insert TXT и Qdrant Insert PDF есть sub-нода Embeddings OpenAI. Заменяем её на Embeddings Cohere.

Embeddings OpenAI (text-embedding-3-small) → Embeddings Cohere (embed-v4.0)
🔀 Что такое sub-нода в n8n и как её заменить статья · ~2 мин ▾

В n8n AI-ноды устроены как «родитель + дети». Родитель — основная нода (Qdrant Insert, AI Agent, Vector Store Tool…). Дети — sub-ноды, которые подключаются к кружкам на нижней стороне родителя. Каждая sub-нода — это «поставщик ресурса» для родителя:

  • Embeddings * — поставляет векторы (OpenAI / Cohere / Voyage / bge…).
  • Document Loader — поставляет «нарезанные» куски документа.
  • Text Splitter — внутри Document Loader, разбивает на чанки.
  • Memory — поставляет историю диалога для AI Agent.
  • Tool — поставляет инструменты, которые AI Agent может вызвать.

Когда родитель работает (например, Qdrant Insert получил файл и собирается записать в БД), он по очереди обращается к sub-нодам: «дай мне чанки», «переведи каждый чанк в вектор». Sub-ноды отвечают, родитель собирает результат.

Как заменить sub-ноду

  1. Открой родительскую ноду (двойной клик).
  2. В правой панели найди sub-ноду нужного типа (например, существующая Embeddings OpenAI).
  3. Нажми × на sub-ноде — она отсоединится от родителя.
  4. Нажми + на том же кружке (например, Add document embedding) → выбери новую sub-ноду из списка.
  5. Заполни параметры новой sub-ноды (credential, модель, режим).

Важно: если родитель используется в нескольких местах (одна и та же sub-нода вшита в две разные ноды Qdrant Insert) — заменить нужно в каждом месте отдельно. Sub-ноды локальны для родителя, не разделяются.

Что конкретно сделать в WF1

4 Обнови Product Config DataTable

Что делаем: в DataTable [AI-01] Product Config у тебя одна строка product_id = ai-01 с JSON-конфигом продукта. Внутри секции qdrant меняем три поля — провайдер, модель и размерность.

Embedding-настройки относятся к продукту, а не к каналу — у одного продукта одна база знаний и одна embedding-модель. Если позже добавишь web-чат или MAX-канал, embedding не нужно дублировать в их конфиги — он берётся из Product Config.

Что меняем в поле config

Открой DataTable [AI-01] Product Config → найди строку product_id = ai-01 → отредактируй JSON в поле config. В секции qdrant:

  • embedding_provider "openai" → "cohere"
  • embedding_model "text-embedding-3-small" → "embed-v4.0"
  • embedding_dimensions 1536 → 1024

Остальные поля (collection, kb_description, chat_memory, escalation_tool, log_table) не трогай.

Типичные ошибки
Три места где обычно застревают
  1. input_type одинаковый в WF1 и WF2. Самая частая ошибка. WF1 = search_document, WF2 = search_query. Разные.
  2. Не пересоздали коллекцию перед сменой нод. Получишь Wrong input: expected dim 1536, got 1024 при первой же загрузке файла.
  3. Native ноды Cohere нет в n8n. Зависит от версии. Если в списке sub-нод нет Embeddings Cohere — нужен HTTP Request, это отдельный более сложный сценарий (~2 часа работы). Уточни у ментора.
