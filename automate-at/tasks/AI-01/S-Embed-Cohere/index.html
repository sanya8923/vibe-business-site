Что меняем и зачем

Когда агент ищет ответ в базе знаний, он сначала превращает вопрос пользователя в вектор — цифровое представление смысла. Embedding-модель — это и есть тот инструмент, который делает превращение. По умолчанию мы используем модель OpenAI.

Проблема: OpenAI text-embedding-3-small обучалась преимущественно на английских текстах. Русский язык со всей его морфологией — «стол», «стола», «столу», «столы» — для неё сложнее. Cohere embed-v4.0 специально создавалась для 100+ языков и понимает русский так же хорошо, как английский.

⚠
Когда это проявляется
Агент находит не то или «не знает» ответа — хотя он есть в базе

Пользователь спрашивает «где находятся наши склады» — в базе есть документ про «склад» и «хранение». OpenAI embedding иногда не свяжет «склады» и «склад» как близкие по смыслу. Cohere embed-v4 с этим справляется — она понимает морфологию и синонимы на русском языке.

На бенчмарках: на русскоязычных запросах Cohere embed-v4 показывает на 15–20% лучший recall, чем OpenAI text-embedding-3-small.

🎓 Cohere — что это, какие модели, зачем разный input_type статья · ~3 мин ▾

Cohere — компания, которая делает языковые модели для бизнеса. Их сильная сторона — мультиязычность: модели понимают 100+ языков примерно одинаково хорошо. Большинство embedding-моделей оптимизированы под английский и теряют точность на русской морфологии — Cohere специально решает эту проблему.

У Cohere три семейства моделей:

  • Embed — превращает текст в вектор (числовое представление смысла). Используется в RAG для поиска по смыслу. Это то, что мы будем менять в задаче.
  • Rerank — берёт топ-50 результатов поиска и переранжирует их по точности. Отдельный стретч S-Retrieval-Rerank.
  • Chat — генеративная модель для ответов агента. В этой задаче мы её не задействуем — меняем только embedding.

Почему модель embed-v4.0

Это флагманская embedding-модель Cohere для мультиязычного поиска.

Плюсы:

  • Понимает русский на уровне с английским — морфология, синонимы, разговорные формы.
  • Поддерживает Matryoshka embeddings — можно укоротить вектор до 256 / 512 / 1024 чисел без сильной потери точности (когда нужно сэкономить место в БД).
  • Отдельные режимы для документов и запросов (см. ниже про input_type).

Минусы:

  • Размерность 1024 — нельзя смешивать векторы с коллекцией под другую размерность. Старую коллекцию придётся пересоздать.
  • Свободный лимит — 1000 запросов в месяц. Для тестов хватает, для прода нужен платный план.

Зачем разный input_type в workflow загрузки и workflow агента

В этой задаче ты меняешь embedding в двух местах: в workflow загрузки в базу знаний и в workflow агента, который ищет ответ на вопрос пользователя. Параметр input_type задаётся для каждого случая отдельно:

  • search_document — режим для индексации документов. Используется в workflow загрузки.
  • search_query — режим для поискового запроса. Используется в workflow агента.

Cohere обучила модель так, чтобы документы и вопросы «тяготели» друг к другу в векторном пространстве, но при условии что обрабатываются через правильный режим. Если поставить везде search_document — вопрос «попадает» в другое облако векторного пространства, и расстояния до документов будут хуже. Recall падает на 5–10%.

Мнемоника: при загрузке в базу знаний → search_document; при поиске ответа агентом → search_query.

Что меняется в воркфлоу

В базовом агенте у тебя два workflow:

  • workflow загрузки — кладёт файлы в базу знаний (ingestion).
  • workflow агента — отвечает на вопросы пользователей (retrieval).

Точные имена workflow могут отличаться в твоей версии курса — ориентируйся на функцию.

Оба workflow содержат sub-ноду Embeddings OpenAI. Меняем её на Embeddings Cohere — с важным отличием в параметре между workflow загрузки и workflow агента.

💡
Главная тонкость
В workflow загрузки и в workflow агента параметр Input Type разный

В workflow загрузки (где файлы кладутся в базу знаний) ставим search_document — это режим для индексации документов.

В workflow агента (где агент ищет ответ на вопрос пользователя) ставим search_query — режим для поискового запроса.

Если поставить везде одинаково — качество поиска упадёт на 5–10%. Разбор причины — в статье «Cohere — что это» выше.

Шаг 0 — Получи доступ к Cohere API

Cohere — внешний сервис. Нужно зарегистрироваться, получить ключ и проверить что он работает. Занимает 5 минут.

0.1
Регистрация
Открой dashboard.cohere.com → Sign Up. Email + подтверждение. Банковская карта не нужна — free tier даётся сразу.
0.2
Получи API key
В левом меню: API Keys → + New Trial key. Имя любое, например n8n-school. Скопируй ключ — он показывается один раз.
0.3
Проверь что ключ работает
Самый простой способ — отправить тестовый запрос к Cohere API через HTTP Request ноду в n8n. Используй готовый curl-запрос ниже: его не нужно запускать в терминале, n8n умеет автоматически импортировать curl в HTTP Request ноду.
curl-запрос для проверки ключа 
curl https://api.cohere.com/v2/embed \
  -H "Authorization: Bearer ВАШ_КЛЮЧ" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "embed-v4.0",
    "input_type": "search_document",
    "embedding_types": ["float"],
    "texts": ["тестовая фраза для проверки"]
  }'
Замени ВАШ_КЛЮЧ на свой API key из шага 0.2.
Ожидаемый ответ: JSON с полем embeddings.float — массив из 1024 чисел.
Если 401 — ключ скопирован неверно. Если 429 — trial исчерпан (1000 req/мес).
⚡ Как быстро настроить HTTP Request ноду через Import from cURL статья · ~2 мин ▾

cURL (читается «кёрл») — это команда для отправки HTTP-запросов из терминала. Документация большинства API устроена так: «вот пример запроса в виде curl-команды — отправь его и получишь ответ».

Не обязательно запускать curl в терминале. В n8n у HTTP Request ноды есть кнопка Import from cURL — она забирает готовую curl-команду и автоматически разбирает её на части: метод, URL, заголовки, тело запроса. Это самый быстрый способ настроить вызов любого API.

Пошагово

  1. Скопируй curl-команду из задачи или документации API.

  2. В n8n создай новый workflow (или открой sandbox). Добавь ноду HTTP Request и открой её настройки (двойной клик).

  3. В правом верхнем углу панели настроек ноды (под кнопкой Execute step) нажми Import cURL.

  4. В открывшемся диалоге Import cURL command вставь скопированную curl-команду в textarea. Нажми Import.

  5. n8n автоматически разберёт curl и заполнит поля HTTP Request ноды: Method (POST/GET/DELETE/…), URL, Send Headers, Send Body, и сами заголовки/тело. Проверь, что всё подставилось корректно — Method, URL и заголовки видны сразу, Body может быть ниже по скроллу.

  6. Если в curl был placeholder вроде ВАШ_КЛЮЧ или YOUR_TOKEN — найди его в импортированных полях (обычно в Header Parameters или в Body) и замени на реальное значение.

  7. Нажми Execute step — справа в панели OUTPUT появится ответ API.

Зачем это нужно знать: большинство сторонних API даёт примеры в виде curl-команд. Зная Import from cURL, ты можешь подключить почти любой сервис без чтения его SDK — просто скопировал команду из доков, импортировал, заменил ключ.

Подсказка: если в curl используется HTTP-заголовок Authorization: Bearer …, можно после импорта переключить ноду на использование сохранённого Generic Header Auth credential (см. статью «Как добавить Header Auth credential в n8n»). Тогда ключ не лежит «в открытом виде» в воркфлоу.

🔑 Как добавить Header Auth credential в n8n статья · ~3 мин ▾

Header Auth — универсальный тип credential в n8n для сервисов, которые принимают API-ключ через HTTP-заголовок Authorization (Cohere, Voyage, OpenRouter и многие другие).

  1. В левом меню n8n нажми Credentials.

  2. Нажми Add credential (кнопка в правом верхнем углу).

  3. В поиске набери Header Auth → выбери Header Auth.

  4. Заполни поля:

    • Credential name: <имя_сервиса> — задача-носитель скажет конкретное имя (например, Cohere API). Ноды будут искать credential по этому имени.
    • Name: Authorization — это имя HTTP-заголовка, его не меняй.
    • Value: Bearer <твой-api-key> — подставь ключ от сервиса. Пробел между Bearer и ключом обязателен.

  5. Нажми Save. Credential появится в списке и станет доступен во всех нодах.

Где взять API-ключ? Возможны два варианта:

  • Школьные сервисы (LiteLLM-прокси, школьный Qdrant Master, общие OpenAI/Anthropic) — ключ лежит в личном кабинете школы, раздел «Мои Credentials». Открой её — скопируй нужный ключ.
  • Внешние сервисы под твою задачу (Cohere, Voyage, своя OpenAI-подписка) — регистрируешься сам, ключ получаешь в дашборде сервиса. Это описано в шагах задачи.

Почему именно Header Auth? Это универсальный способ передать Authorization: Bearer ключ в любую HTTP-ноду. В некоторых версиях n8n есть встроенный тип под конкретный сервис (например, Cohere API) — если он есть, используй его, удобнее. Если нет — Header Auth работает одинаково хорошо.

🗝 Раздел «Мои Credentials» в личном кабинете школы статья · ~1 мин ▾

В личном кабинете школы (этой платформе) есть раздел «Мои Credentials». Там лежат школьные ключи и параметры доступа, которые ты можешь использовать в своих воркфлоу:

  • Qdrant Master — API-ключ для школьного Qdrant (для DELETE-запросов, прямых HTTP-вызовов).
  • OpenAI / Anthropic / DeepSeek — школьные ключи через LiteLLM-прокси.
  • student_id — твой идентификатор (например, c01), нужен для имени Qdrant-коллекции.
  • Postgres — реквизиты школьной БД для логов и chat memory.

Внешние сервисы (Cohere, Voyage, Mem0 и т.п.) под свои задачи студент регистрирует сам — их ключи в школьный раздел не попадают.

Если нужного ключа нет в «Мои Credentials» или ключ не работает — напиши в учебный чат школы.

Шаги внедрения

1 Пересоздай Qdrant-коллекцию делать ПЕРВЫМ

Почему первым: Qdrant хранит коллекцию с фиксированной размерностью. OpenAI делал векторы на 1536 чисел, Cohere — на 1024. Эти два формата несовместимы. Если сначала поменяешь ноды, а потом попытаешься загрузить файл — Qdrant вернёт ошибку. Поэтому сначала удаляем старую коллекцию, создаём новую, а потом меняем ноды.

🧭 Что такое Qdrant и как с ним работать в n8n статья · ~3 мин ▾

Qdrant — векторная база данных. В обычной БД ты ищешь по точному совпадению («где имя = Иван»). В Qdrant ты ищешь по смыслу — «найди документы, похожие на этот текст».

Чтобы это работало, документы и запросы предварительно прогоняются через embedding-модель (OpenAI, Cohere, bge…). Модель превращает текст в вектор — список из 1024 / 1536 / 768 чисел. Похожий смысл = близкие векторы. Qdrant умеет быстро находить «ближайшие» векторы к заданному.

Структура хранения

  • Коллекция — это «таблица» в Qdrant. У неё фиксированная размерность векторов и метрика расстояния. Поменять размерность нельзя — только пересоздать коллекцию.
  • Точка (point) — одна запись внутри коллекции: id + вектор + произвольный JSON-payload (например, { "filename": "manual.pdf", "page": 4 }).
  • Search — отдай вектор запроса, получи топ-N ближайших точек с их payload'ами.

Ноды Qdrant в n8n

В стандартной поставке n8n работает с Qdrant через три «AI-родителя»:

  • Qdrant Insert — кладёт чанки текста в коллекцию (используется в WF ingestion). Подключает sub-ноды Embeddings (откуда брать векторы) и Document Loader (откуда брать текст).
  • Qdrant KB (как Tool у AI Agent) — даёт агенту инструмент «искать по базе знаний». Тоже подключает sub-ноду Embeddings — обязательно ту же модель, что и при Insert.
  • HTTP Request — для всего остального: создать/удалить коллекцию, посмотреть статистику, очистить точки. Прямые REST-вызовы Qdrant API.

Школьная инфра

Школа держит общий Qdrant-инстанс на https://qdrant.school.vibe-business.site. У каждого студента свои коллекции — имена изолированы префиксом student_id. Master-ключ для прямых HTTP-вызовов лежит в разделе «Мои Credentials» в личном кабинете.

🏷 Правила именования Qdrant-коллекций в школе статья · ~1 мин ▾

Все коллекции в школьном Qdrant именуются по шаблону:

<student_id>-<что_хранится>

Примеры:

  • c01-kb-agent — у студента c01 хранится база знаний агента.
  • c12-faq-bot — у студента c12 данные FAQ-бота.
  • c07-products — у студента c07 каталог товаров.

Зачем: на одном Qdrant-инстансе живут коллекции всех студентов. Префикс student_id исключает пересечения. Никогда не используй чужой prefix — увидишь чужие данные или сломаешь чужую коллекцию.

Свой student_id найдёшь в «Мои Credentials» в личном кабинете. Имя коллекции, которое ты задал при создании агента, обычно дублируется в нескольких местах: в нодах Qdrant Insert (workflow загрузки), в sub-ноде Qdrant KB (workflow агента), а также в конфиге продукта (если используется DataTable-конфиг). При смене embedding-провайдера достаточно сохранить то же имя коллекции — пересоздать её с новой размерностью.

📐 Почему нельзя смешивать векторы разной размерности статья · ~3 мин ▾

Когда ты создаёшь коллекцию в Qdrant, ты один раз говоришь: «в этой коллекции векторы будут размером 1536». Это прошивается в структуру БД навсегда — как тип колонки в SQL-таблице.

Embedding-модели выдают векторы конкретного размера. Это не случайность — размер определяет «сколько смысла» помещается в вектор:

  • OpenAI text-embedding-3-small → 1536 чисел
  • Cohere embed-v4.0 → 1024 числа (или меньше через Matryoshka)
  • bge-m3 → 1024 числа

Если ты попытаешься записать 1024-мерный вектор в коллекцию, созданную под 1536 — Qdrant откажет с ошибкой Wrong input: expected dim 1536, got 1024.

Что делать:

  1. Удалить старую коллекцию (все векторы пропадут).
  2. Создать новую с правильной размерностью. Это делается автоматически при первом insert'е через ноду Qdrant Insert.
  3. Перезалить все файлы KB через WF ingestion — теперь они запишутся с правильными векторами.

Бояться потери данных? Нет. KB — это производное от твоих исходных файлов. Файлы у тебя есть, workflow загрузки умеет их грузить. Удаление коллекции = «очисти индекс», а не «удали документы».

Что конкретно сделать

В Qdrant нет отдельной ноды «Delete Collection» — используем универсальный HTTP Request. Скопируй curl-запрос ниже и импортируй его в HTTP Request ноду через Import from cURL (см. раскрывайку «Как быстро настроить HTTP Request ноду через Import from cURL» в шаге 0.3 — инструкция универсальная).

curl-запрос для удаления коллекции 
curl -X DELETE \
  https://qdrant.school.vibe-business.site/collections/<имя_твоей_коллекции> \
  -H "api-key: <Qdrant Master ключ>"
<имя_твоей_коллекции> — то самое имя, которое ты задал в базовом агенте (см. Product Config.qdrant.collection). После удаления оно останется тем же — просто следующий запуск workflow загрузки пересоздаст коллекцию автоматически с новой размерностью.
<Qdrant Master ключ> — лежит в личном кабинете школы, раздел «Мои Credentials».
Ожидаемый ответ: JSON {"result": true, "status": "ok"}.

DELETE удаляет коллекцию вместе со всеми векторами. Файлы-источники, из которых ты строил KB, не трогаются — они останутся у тебя, и workflow загрузки перезальёт их в новую коллекцию.

До удаления
Коллекция <твоё_имя> с векторами 1536d. Файлы KB уже залиты через OpenAI embedding.
После удаления
Коллекции нет. При следующем запуске workflow загрузки нода Qdrant Insert создаст её заново с размерностью 1024 (под Cohere). Старые данные не нужны.
2 Workflow загрузки в базу знаний — замени Embeddings OpenAI → Cohere

Что делаем: в workflow загрузки ноды Qdrant Insert TXT и Qdrant Insert PDF содержат sub-ноду Embeddings OpenAI. Заменяем её на Embeddings Cohere.

Embeddings OpenAI (text-embedding-3-small) → Embeddings Cohere (embed-v4.0)
🔀 Что такое sub-нода в n8n и как её заменить статья · ~2 мин ▾

В n8n AI-ноды устроены как «родитель + дети». Родитель — основная нода (AI Agent, Qdrant Insert, Qdrant Vector Store…). Дети — sub-ноды, которые подключаются к специальным кружкам на нижней стороне родителя. Каждая sub-нода — это «поставщик ресурса» для родителя.

Какие бывают sub-ноды

  • Embeddings * — поставляет векторы (OpenAI / Cohere / Voyage / bge / …).
  • Chat Model — поставляет LLM для AI Agent.
  • Document Loader — поставляет «нарезанные» куски документа.
  • Text Splitter — внутри Document Loader, разбивает на чанки.
  • Memory — поставляет историю диалога для AI Agent.
  • Tool — поставляет инструменты, которые AI Agent может вызвать.

Когда родитель запускается (например, Qdrant Insert получил файл и собирается записать его в БД), он по очереди обращается к sub-нодам: «дай мне чанки», «переведи каждый чанк в вектор». Sub-ноды отвечают, родитель собирает результат.

Как выглядит sub-нода в настройках родителя

Если открыть NDV (панель настроек) родителя двойным кликом, на нижней границе панели видны кружки sub-нод. Подключённая sub-нода показана как круглый цветной индикатор с иконкой провайдера. При наведении — tooltip с именем и статусом.

Как заменить sub-ноду

  1. Закрой NDV родителя (Esc).

  2. На canvas найди подключённую sub-ноду — она расположена под родителем. Кликни на ней, нажми Delete (или правый клик → Delete). Sub-нода удалится, у родителя освободится слот.

  3. На родителе освободившийся слот покажет «+» (например, Add document embedding). Нажми — справа откроется список доступных sub-нод нужного типа.

  4. Выбери нужную sub-ноду — она автоматически подключится к родителю на освободившемся слоте.

  5. Заполни параметры новой sub-ноды (credential, модель, режим). Сохрани workflow.

Важно: sub-ноды локальны для своего родителя — между нодами не разделяются. Если в воркфлоу несколько родителей одного типа (например, две ноды Qdrant Insert — для txt и для pdf — и у каждой своя sub-нода Embeddings) — заменить нужно в каждом родителе отдельно.

Что конкретно сделать
  1. Открой workflow, отвечающий за загрузку файлов в базу знаний.
  2. Замени sub-ноду Embeddings OpenAI на Embeddings Cohere в ноде Qdrant Insert TXT (как — в раскрывайке про sub-ноду выше).
  3. Повтори для Qdrant Insert PDF — там та же sub-нода.

⚙️ Что такое конфиг продукта в DataTable и зачем он нужен статья · ~3 мин ▾

Поведение AI-агента в n8n определяется множеством параметров: какая модель LLM, какая коллекция в Qdrant, какой системный промпт, в какую БД писать логи. Эти параметры можно «зашить» прямо в ноды воркфлоу — но тогда любая правка требует открыть редактор, найти ноду, поменять и опубликовать новую версию.

DataTable-конфиг — альтернативный способ. Параметры лежат отдельно, в специальной таблице (DataTable) внутри n8n. Воркфлоу при запуске читает строку конфига и подставляет значения в свои ноды на лету.

Как это выглядит

Создаётся отдельная таблица DataTable (например, Agent Config) с двумя колонками:

  • key — идентификатор строки конфига (например, main, support, telegram).
  • config — строка JSON со всеми параметрами агента.

Внутри config лежит обычный JSON с понятной структурой:

config (JSON-string) 
{
  "agent": {
    "system_prompt": "Ты помощник интернет-магазина. Отвечай вежливо и по делу.",
    "model": "gpt-4.1-mini",
    "temperature": 0.3
  },
  "qdrant": {
    "collection": "my-knowledge-base",
    "embedding_model": "text-embedding-3-small"
  },
  "log_table": "agent_logs"
}

Каждое поле — это параметр одной из нод агента: системный промпт уйдёт в AI Agent, модель — туда же, имя коллекции — в Qdrant KB sub-ноду, имя таблицы логов — в Postgres Insert.

Что это даёт

  • Правка без редактирования воркфлоу. Хочешь сменить системный промпт — открой DataTable, поменяй ячейку, сохрани. Воркфлоу перезагружать не нужно — он подхватит новое значение при следующем запуске.
  • Один воркфлоу — много конфигураций. Можно запустить тот же воркфлоу под несколько разных контекстов: разные продукты, разные каналы (Telegram / web / WhatsApp), разные клиенты — каждый в своей строке таблицы.
  • История изменений. Версии конфига видны в одном месте, легче откатить ошибочную правку.

Как работает чтение конфига при запуске агента

  1. Триггер запускает workflow агента (новое сообщение от пользователя).
  2. Нода Get rows читает строку из DataTable по ключу (например, key = main). Получает config в виде строки.
  3. Нода Set парсит JSON и раскладывает поля по плоским переменным: system_prompt, model, collection, log_table, …
  4. AI Agent и его sub-ноды (Qdrant KB, Postgres Chat Memory) ссылаются на эти переменные через n8n-выражения вида {{$json.system_prompt}}. При каждом запуске берётся актуальное значение из таблицы.

В сложных продуктах конфиг можно разделить на два уровня — product config (то, что общее для всего продукта: база знаний, embedding, логи) и channel config (то, что меняется по каналам: промпт, модель, лимит длины ответа). Workflow тогда делает два последовательных Get rows и сливает результаты — это шаблон, который ты увидишь в этой задаче.

Что меняем в поле config

Открой DataTable [AI-01] Product Config → найди строку product_id = ai-01 → отредактируй JSON в поле config. В секции qdrant:

  • embedding_provider "openai" → "cohere"
  • embedding_model "text-embedding-3-small" → "embed-v4.0"
  • embedding_dimensions 1536 → 1024

Остальные поля (collection, kb_description, chat_memory, escalation_tool, log_table) не трогай.

Было (Product Config[ai-01].config)
{
  "qdrant": {

    "embedding_dimensions": 1536
  },

}

Три места где обычно застревают
  1. input_type одинаковый в обоих workflow. Самая частая ошибка. В workflow загрузки search_document, в workflow агента search_query. Разные.
