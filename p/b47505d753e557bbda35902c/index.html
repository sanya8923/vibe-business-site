M-HttpV2-06 Реферальный сокращатель

Представь: ты ведёшь маркетинг и раздаёшь длинные ссылки на лендинги через несколько каналов сразу — Telegram-канал, рассылка, друзья-инфлюенсеры. По коротким ссылкам в TinyURL не видно откуда пришёл клик — все идут на один и тот же URL. Соберём чат-бот, который принимает строку длинная_ссылка|имя_реферера, добавляет к URL метку ?ref=имя и сокращает результат через TinyURL. Параллельно бот ведёт журнал в Data Table — каждое сокращение пишется отдельной строкой с реферером и временем, чтобы потом посчитать кто привёл больше людей. Главное новое — авторизация по Bearer Token: стандарт, который используют большинство современных API.

Что собираем

Шесть нод в одну линию. Chat Trigger поднимает встроенный веб-чат прямо в n8n: маркетолог приходит по ссылке и шлёт сообщение в формате https://example.com|ivan — длинная ссылка и имя реферера через вертикальную черту.

Первая Edit Fields разбирает входную строку: разрезает её по | на две части, собирает реферальный URL https://example.com?ref=ivan и складывает в отдельное поле. Без этого шага TinyURL получит «голую» ссылку без реферера, и трек откуда пришёл клиент потеряется.

Нода HTTP Request идёт во внешний API api.tinyurl.com через POST-запрос и просит сократить уже готовую реферальную ссылку. Доступ закрыт Bearer Token'ом, поэтому токен хранится отдельно в Header Auth credential и подставляется в заголовок Authorization каждого запроса автоматически.

Вторая Edit Fields собирает из четырёх кусков аккуратную запись для журнала — original_url, short_url, username, created_at. Нода Data Table пишет эту четвёрку новой строкой в журнал m-httpv2-06-links — со временем там копится история всех реферальных сокращений, по которой можно посчитать сколько пришло от каждого реферера.

Последняя Edit Fields готовит ответ чату: одна строка вида Короткая: https://tinyurl.com/... в поле output. Chat Trigger сам забирает поле output из последней ноды и отправляет его пользователю в чат.

Главное новое тут — Bearer Token в авторизации HTTP-запроса. Bearer — это стандарт, который используют большинство современных API: ключ всегда начинается со слова Bearer, потом пробел, потом сам токен.

Шаги выполнения

1Создай Data Table под журнал сокращений▾

Начнём с хранилища — куда будем складывать историю реферальных сокращений. Data Table это встроенная в n8n таблица: ей задают имя и колонки с типами, после чего любая нода на canvas может писать в неё новые строки.

🗄DataTable в n8n: что это и когда использовать вместо базы данныхстатья · ~5 мин▾

Что такое DataTable

DataTable — встроенное хранилище n8n, что-то среднее между Excel-листом и базой данных. Лежит внутри n8n, не требует отдельного сервера и подключений, редактируется прямо в UI. Каждая таблица принадлежит проекту: видна и доступна всем воркфлоу этого проекта.

В одной таблице — фиксированная схема колонок (имя + тип) и произвольное количество строк. Доступ из воркфлоу — через ноду Data table (resource = Row, операции Get rows, Insert row, Update row, Delete row).

Когда выбирать DataTable

DataTable подходит для случаев, где данных немного и нужно удобство редактирования через UI:

  • Конфиги воркфлоу — настройки агента, ключи API-описаний, лимиты. Меняешь значение в ячейке, воркфлоу при следующем запуске берёт новое.
  • Небольшие справочники — список разрешённых пользователей, маппинг каналов на промпты, словарь ответов.
  • Состояние между запусками — счётчики, последний обработанный ID, кеш токена с TTL.
Когда DataTable, когда Postgres
Главное отличие — назначение

DataTable — для редко меняющихся данных, которые удобно править руками: конфиги, справочники, флаги. Postgres — для журналов, истории, аналитики: туда воркфлоу пишет много строк, оттуда читает выборки с условиями и сортировкой.

Условный порог: если ожидается больше 1000 строк или нужна аналитика с агрегацией — это Postgres. Если до 100 строк и они меняются вручную — это DataTable.

Тонкость
Параллельные записи не идемпотентны

Если два воркфлоу одновременно дописывают строку в одну и ту же DataTable — порядок может смешаться, а часть операций — провалиться. Для счётчиков и журналов, которые пишут часто из разных триггеров, выбирай Postgres.

Частый паттерн: конфиг продукта в DataTable

Параметры AI-агента (системный промпт, имя модели, имя коллекции Qdrant, имя таблицы логов) можно зашить прямо в ноды — но тогда любая правка требует открыть редактор, найти ноду, поменять, опубликовать новую версию. Альтернатива — вынести всё в одну строку DataTable.

Схема таблицы (например, Agent Config) — две колонки:

  • key (string) — идентификатор конфига: main, support, telegram.
  • config (string) — JSON со всеми параметрами.

Внутри config — обычный JSON с понятной структурой:

{
  "agent": {
    "system_prompt": "Ты помощник интернет-магазина. Отвечай вежливо и по делу.",
    "model": "gpt-4.1-mini",
    "temperature": 0.3
  },
  "qdrant": {
    "collection": "<имя_коллекции>",
    "embedding_model": "text-embedding-3-small"
  },
  "log_table": "agent_logs"
}

Как воркфлоу читает этот конфиг

  1. Триггер запускает workflow (новое сообщение от пользователя).
  2. Нода Get rows читает строку из Agent Config по ключу (например, key = main). Получает config как строку.
  3. Нода Set (или Code) парсит JSON и раскладывает поля по плоским переменным: system_prompt, model, collection, log_table.
  4. AI Agent и его sub-ноды (Qdrant KB, Postgres Chat Memory) ссылаются на эти переменные через выражения вида {{$json.system_prompt}}. При каждом запуске берётся актуальное значение.

Что это даёт

  • Правка без редактирования воркфлоу. Сменил системный промпт — открыл DataTable, поменял ячейку, сохранил. Воркфлоу подхватит новое значение при следующем запуске.
  • Один воркфлоу — много конфигураций. Тот же workflow обслуживает несколько контекстов (разные продукты, разные каналы, разные клиенты) — каждый со своей строкой.
  • История правок в одном месте. Легче откатить ошибочное изменение.
Двухуровневый конфиг
Product config + Channel config

В сложных продуктах конфиг разделяют на два уровня: product config — общее для всего продукта (база знаний, embedding-модель, имя таблицы логов), и channel config — что меняется по каналам (промпт, модель, лимит длины ответа). Workflow тогда делает два последовательных Get rows и сливает результаты.

2Получи Bearer Token у TinyURL▾

Чтобы воркфлоу мог сокращать ссылки, ему нужен доступ к TinyURL — это сервис коротких ссылок с бесплатным API. Доступ закрыт Bearer Token'ом: каждый запрос должен принести с собой строку-идентификатор, иначе TinyURL ответит ошибкой 401 Unauthorized.

Регистрация бесплатная, лимит — 600 ссылок в месяц на бесплатном тарифе. Этот шаг ты делаешь в браузере, в личном кабинете TinyURL.

🔑Как получить Bearer Token у TinyURLстатья · ~2 мин▾

Что за сервис

tinyurl.com — сокращатель ссылок с публичным REST API. По длинному URL отдаёт короткий вида tinyurl.com/abc123, можно задать кастомный alias и срок жизни. Используется в маркетинговых рассылках, ботах и везде, где длинная ссылка не влезает по символам.

Как получить ключ

  • Открой tinyurl.com и зарегистрируйся (email + пароль).
  • Зайди в My Account (правый верхний угол).
  • В левом меню — пункт API.
  • Нажми кнопку Create API Token.
  • Укажи название токена (для себя — например n8n-bot) и срок жизни.
  • Скопируй токен сразу — после закрытия модалки он больше не покажется.

После этого подставляй токен в заголовок:

Authorization: Bearer <token>

Что умеет

  • POST /create — создать короткую ссылку из длинной.
  • POST /alias — создать ссылку с кастомным алиасом (tinyurl.com/my-promo).
  • GET /alias/{alias} — получить статистику по короткой ссылке (число переходов).
  • DELETE /alias/{alias} — удалить короткую ссылку.
Лимиты
Free tier ограничен по запросам

У бесплатного тарифа TinyURL месячный лимит на число API-запросов (точную цифру смотри в личном кабинете — она меняется со временем). При превышении сервер начнёт отдавать 429. Для боевых ботов с большим трафиком — переходи на платный тариф или используй альтернативу (например Bitly).

Типичные ошибки

  • Закрыл окно с токеном, не успев скопировать → старый токен восстановить нельзя → создавай новый через Create API Token.
  • Перепутал API с публичным сокращателем без авторизации (tinyurl.com/api-create.php) — это легаси-эндпоинт без токена и без статистики. Новый API живёт на api.tinyurl.com и требует Bearer.
  • Написал Authorization: <token> без слова Bearer → 401 → префикс Bearer обязателен.

Создай API Token. Дай ему понятное название (например, n8n-shortener). TinyURL покажет тебе длинную строку из букв и цифр — это и есть твой Bearer Token.

3Подключи Chat Trigger▾

Первая нода воркфлоу — Chat Trigger. Это специальный триггер от n8n, который поднимает встроенный веб-чат: пользователь открывает чат по уникальной ссылке и шлёт сообщения, а воркфлоу запускается на каждом новом сообщении.

В отличие от Telegram, Chat Trigger не требует регистрации внешнего бота — чат живёт прямо внутри n8n и работает «из коробки».

💬Публичный чат Chat Trigger — Make Chat Publicly Availableстатья · ~2 мин▾

Зачем включать публичный чат

По умолчанию Chat Trigger показывает чат-окно только внутри редактора n8n — через кнопку Open chat снизу canvas. Это удобно для разработки, но боту не отправишь ссылку коллеге и не откроешь его на телефоне.

Переключатель Make Chat Publicly Available в параметрах Chat Trigger превращает встроенный чат в самостоятельную хостинговую страницу: n8n начинает обслуживать публичный URL вида https://<твой-домен>/webhook/<uuid>/chat, и любой кто откроет этот адрес — попадает на готовый чат с твоим ботом. Без логинов, без n8n-аккаунта, без редактора.

Что меняется в ноде до и после переключения

До включения в параметрах Chat Trigger всего три блока: тоггл Make Chat Publicly Available (выключен), тоггл Make Available in n8n Chat Hub (выключен) и пустой раздел Options. Больше ничего настраивать нельзя — нода ловит сообщения только из встроенного Open chat.

После включения Make Chat Publicly Available нода сразу раскрывает четыре дополнительных блока:

Что появляется после включения публичного режима
  • Chat URL — Production URL твоей публичной страницы чата. Это и есть адрес, который можно открыть в любом браузере или скинуть в мессенджер. Уникальный для каждого workflow (внутри webhook/<uuid>/chat).
  • Mode — Hosted Chat (n8n сам отдаёт готовую страницу чата) или Embedded Chat (встраивание чата виджетом на свой сайт через <script>-тег). Для большинства задач хватает Hosted Chat.
  • Authentication — None (любой человек с URL заходит без логина), Basic Auth (логин + пароль) или n8n User Auth (только пользователи твоего n8n). По умолчанию None.
  • Initial Message(s) — приветствие бота на странице чата. По умолчанию там стоит Hi there! 👋 My name is Nathan. How can I assist you today? — это нужно заменить на своё, чтобы посетитель сразу понимал что за бот и о чём с ним говорить.

Параллельно появляется бежевое предупреждение: «Chat will be live at the URL above once this workflow is published». То есть сам по себе тоггл публичный чат не включает — нужно ещё нажать Publish. До публикации URL вернёт 404.

Порядок действий

  1. Открой Chat Trigger, включи Make Chat Publicly Available (тоггл станет зелёным).
  2. Замени Initial Message(s) на своё приветствие (необязательно, но дефолтный «Nathan» выглядит странно).
  3. Закрой ноду и нажми Publish в правом верхнем углу.
  4. Скопируй Chat URL из ноды и открой его в браузере — увидишь свою публичную страницу чата.

Типичные ошибки

  • Включил тоггл, открыл URL — 404. Не нажат Publish. Тоггл только готовит URL, поднимает его публикация.
  • Меняешь параметры опубликованного workflow — на странице чата ничего не меняется. После каждой правки в Chat Trigger или AI-нодах нужно нажимать Publish заново. Хостинговая страница работает на последней опубликованной версии.
  • Поделился URL — посетитель видит «Nathan». Забыл поменять Initial Message(s). Замени и нажми Publish.
  • Хочу скрыть бота от всех кроме команды — переключи Authentication в Basic Auth и задай логин/пароль; иначе любой кому утечёт URL получит доступ.

Когда не подходит

Публичный чат Hosted Chat — это готовая страница n8n, она минималистичная и без брендинга. Для серьёзного продакшна или встройки в свой сайт подходит Embedded Chat (виджет на свою страницу) или вообще отдельный канал — Telegram-бот, сайтовый чат-виджет, мессенджер. Но для быстрого «дать ссылку на бота кому-то прямо сейчас» — Hosted Chat ровно то что нужно.

Добавление ноды на canvas

Нода — это шаг воркфлоу: триггер, действие или преобразование данных. Добавить ноду можно двумя способами: в пустой canvas или после существующей ноды. В обоих случаях открывается панель выбора нод с поиском.

Пошагово

  1. Открой воркфлоу и нажми + на canvas. Если воркфлоу пустой — большая кнопка в центре «Add first step». Если уже есть ноды — маленький + на стрелке после нужной ноды или плюс на самой ноде справа.
  1. В открывшейся панели справа набери название ноды или сервиса в поле поиска. Например, http, slack, openai. Список фильтруется по мере ввода.
  1. Кликни по нужной ноде в списке — она появится на canvas и откроется её панель параметров. Заполни поля и закрой панель. Готово.
Подсказка
Быстрая вставка хоткеем

Нажми Tab на canvas — откроется та же панель выбора нод, не надо целиться мышкой в плюс. Работает и в пустом воркфлоу, и после выбранной ноды.

Подводный камень
Триггер должен быть первым

Первая нода в воркфлоу — всегда триггер (Schedule, Webhook, Manual). Если выбрать обычное действие первым, n8n покажет ошибку при запуске: воркфлоу не из чего стартовать.

Два способа открыть чат для посетителей

После включения Make Chat Publicly Available в ноде Chat Trigger появляется поле Mode с двумя вариантами. По умолчанию стоит Hosted Chat — но разница между ними существенная, и выбор зависит от того, где будет жить твой бот.

Hosted Chat — n8n сам хостит страницу

n8n отдаёт готовую HTML-страницу с чат-интерфейсом по адресу из поля Chat URL. Ты просто копируешь этот адрес и отправляешь кому надо — посетитель открывает ссылку в браузере и сразу видит чат.

Когда подходит:

  • Нужно быстро дать боту ссылку — команде, тестировщику, заказчику.
  • Нет своего сайта или некуда встраивать виджет.
  • Скорость важнее внешнего вида.

Ограничение: страница выглядит как базовый чат n8n — без твоего логотипа и брендинга. URL содержит /webhook/ и выглядит технически.

Embedded Chat — виджет на свой сайт

n8n генерирует код для встройки — <script>-тег, который вставляется в HTML любой страницы. После этого на странице появляется всплывающий виджет: кнопка в углу → раскрывающееся окно чата.

При выборе этого мода нода показывает блок с готовым кодом вида:

<link href="https://<домен>/chat/assets/chat.css" rel="stylesheet" />
<script type="module">
  import Chat from 'https://<домен>/chat/assets/chat.js'
  new Chat({ webhookUrl: 'https://<домен>/webhook/<uuid>/chat' })
</script>

Когда подходит:

  • Есть свой сайт и бот должен быть его частью.
  • Нужен виджет в углу страницы — как ассистент или поддержка.
  • Хочешь встроить чат в лендинг или внутренний портал.

Ограничение: нужен доступ к HTML-коду страницы. Без возможности вставить <script> не подойдёт.

Включил мод, скопировал URL или код — но ничего не работает. Причина в том, что любой публичный чат поднимается только после нажатия Publish в правом верхнем углу. До публикации URL вернёт 404.

Pin Data в n8n: как зафиксировать тестовый вход ноды

Pin Data — это способ «прибить» к ноде заранее заготовленные данные, которые n8n будет подставлять вместо настоящего входа на каждом тестовом запуске. Полезно, когда дёргать реальный источник (внешний воркфлоу, веб-хук, API) на каждое нажатие Execute Workflow неудобно или невозможно.

Зачем это нужно

Часто отлаживается не сам триггер, а ноды после него — логика, маппинг, запрос в базу. Чтобы не звать каждый раз источник данных (другой воркфлоу, бот, форму), удобнее один раз получить пример настоящего входа, зафиксировать его на триггере и потом крутить запуски сколько нужно.

Поэтому Pin Data — это инструмент разработчика на этапе сборки, не часть рабочего воркфлоу. В боевом режиме закреплённые данные перестают использоваться: n8n берёт реальный вход.

Как это работает

К любой ноде с выходом можно прикрепить «запинённый» JSON. После этого:

  • При запуске воркфлоу через UI (Execute Workflow, тестовый прогон) n8n не вызывает саму ноду, а сразу выдаёт следующим нодам зафиксированные данные.
  • Pin Data привязана к конкретной ноде, не к workflow целиком — можно запинить триггер и оставить остальные ноды живыми.
  • На canvas у запинённой ноды появляется визуальный значок (булавка/иконка), чтобы не забыть, что данные «искусственные».

Удобнее всего пинить триггер: на нём поле выхода — то, что в боевом режиме приходит снаружи (Telegram-апдейт, тело веб-хука, поля Execute Workflow Trigger). Тестовый JSON собирают с теми же ключами, что отдаёт реальный источник.

К примеру
Где это встречается

К примеру, в Sub-Workflow с триггером Execute Workflow Trigger пин на триггере позволяет запускать сам Sub-WF кнопкой сверху и отлаживать ноды после триггера, не запуская вызывающий workflow. Иногда — пин ставят на ноде HTTP Request к платному API, чтобы при разработке не жечь квоту запросами «ещё разок проверить парсинг».

Типичные ошибки

  • Забыли убрать пин перед боевым запуском — workflow пошёл в production, но данные ноды всегда одни и те же; реальные входы игнорируются. Перед публикацией пины снимают.
  • Структура запинённого JSON не совпадает с реальным — отладка проходит идеально, но в проде ломается на полях, которых не было в тестовом примере; делайте пин из реальной execution, а не «на глаз».
  • Запинили середину воркфлоу, забыли — переименовали поле в ноде до пина, а пин остался со старой структурой. Следующие ноды читают «фантомные» данные.
  • Pin Data перепутали с переменными среды — пин не передаётся между воркфлоу и не сохраняется при экспорте на другую инсталляцию n8n как «конфиг»; это именно отладочный фиксаж.

Когда не подходит

Pin Data не заменяет настоящие тесты — это удобство в редакторе, не автоматизация. Если нужна повторяемая проверка с разными входами, лучше делать отдельный тестовый workflow или мини-набор тест-кейсов; пин подходит, когда хочется быстро прогнать «один и тот же» сценарий N раз.

.split() — строка превращается в массив

Метод .split(разделитель) разбивает строку на части по указанному символу или подстроке и возвращает массив строк. Сам разделитель в результат не попадает.

'Иван|Петров|ivan@mail.ru'.split('|')
// → ['Иван', 'Петров', 'ivan@mail.ru']

Три части по разделителю | → три элемента в массиве. Чтобы достать конкретный, после .split() пишут [индекс]:

'Иван|Петров|ivan@mail.ru'.split('|')[0]  // → 'Иван'
'Иван|Петров|ivan@mail.ru'.split('|')[1]  // → 'Петров'
'Иван|Петров|ivan@mail.ru'.split('|')[2]  // → 'ivan@mail.ru'

В выражении n8n это выглядит так:

{{ $json.someField.split('|')[0] }}
Подсказка
Разделитель может быть любым символом или подстрокой

Часто это |, запятая ,, пробел , двоеточие : или даже подстрока вроде -> . Главное — написать его точно так, как он встречается в строке, внутри кавычек.

Типичные ошибки

  • Разделитель не найден в строке → .split() вернёт массив из одного элемента (вся строка целиком). [1] при этом вернёт undefined — данные потеряются без ошибки.
  • Разделитель — пустая строка '' → строка разобьётся на отдельные символы: 'abc'.split('') → ['a','b','c']. Так бывает нужно, но чаще это опечатка.
  • Забыл кавычки вокруг разделителя → split(|) вместо split('|') — синтаксическая ошибка, выражение не выполнится.

Внутри круглых скобок укажи разделитель |, чтобы строка разрезалась по символу вертикальной черты. Разделитель должен быть внутри кавычек.

После этого за скобками поставь ноль внутри квадратных скобок. [0] берёт нулевой (первый) элемент массива — это всё что было до |, то есть длинная ссылка.

📖Почему в программировании счёт начинается с нулястатья · ~2 мин▾

Первый элемент — это нулевой

В обычной жизни мы считаем «первый, второй, третий». В программировании первый элемент имеет номер 0, второй — 1, третий — 2. Это называется индексация с нуля (zero-based indexing), и так устроены массивы, списки, выходы нод, символы в строках — почти всё, у чего есть «номер по порядку».

Почему так

Индекс — это не «который по счёту», а «на сколько шагов отступить от начала». Первый элемент стоит в самом начале — отступать не нужно, шагов 0. Второй — на один шаг дальше, индекс 1. Третий — на два, индекс 2.

Этот подход родился из того, как данные физически лежат в памяти компьютера: есть точка начала и смещение от неё. Прижилось — теперь почти все языки программирования (JavaScript, Python, Java, C, и многие другие) и почти все инструменты считают именно так.

Switch как пример

У ноды Switch в режиме Expression ты пишешь выражение, которое возвращает номер выхода. Первый выход (самый верхний на canvas) — это 0, второй — 1, третий — 2.

// первая ветка (output 0) — срочные заявки
// вторая ветка (output 1) — обычные
// третья ветка (output 2) — спам
={{ $json.priority === 'high' ? 0 : $json.priority === 'spam' ? 2 : 1 }}

Если у Switch четыре выхода, значит допустимые индексы — 0, 1, 2, 3. Не 1..4. Вернёшь 4 — элемент потеряется, потому что выхода с таким индексом нет.

Правило одной строкой
N элементов → индексы от 0 до N−1

Десять элементов в массиве — индексы 0..9, не 1..10. Пять выходов у Switch — индексы 0..4. Если где-то увидишь индекс, равный длине, — это уже за границей и обычно приводит к ошибке или потерянным данным.

Где ещё это встретится

  • Массивы в выражениях n8n: $json.items[0] — первый элемент, $json.items[1] — второй.
  • Символы в строке: первый символ имеет позицию 0.
  • Циклы: for (let i = 0; i < items.length; i++) — i начинается с нуля и идёт до length − 1.
  • Колонки и строки в таблицах через код: часто первая строка/колонка — [0], а не [1] (в формулах Excel/Sheets — наоборот, там с единицы; в коде Python/JS — с нуля).

Типичные ошибки

  • Хочешь обратиться к первому элементу и пишешь $json.items[1] — получаешь второй и удивляешься «почему данные не те» → первый это [0].
  • В режиме Expression у Switch пишешь return 1 для первой ветки, а данные уходят во вторую → первая ветка это 0.
  • Считаешь длину массива и думаешь, что последний индекс совпадает с длиной → последний индекс это длина − 1. Массив из 3 элементов: индексы 0, 1, 2, длина 3.

Должно получиться выражение:

{{ $json.chatInput.split('|')[0] }}

Fixed и Expression — два режима любого поля

В правом верхнем углу почти каждого поля параметров ноды есть переключатель Fixed | Expression. Это два режима ввода значения: фиксированное (написал текст — этот текст и пойдёт) или формула на JavaScript, которая вычисляется заново при каждом выполнении ноды. Один и тот же тумблер есть у обычных текстовых полей, у dropdown-ов, у credential pickers и у resource locator полей (где выбираешь объект из списка — таблицу, документ, канал).

Как работает каждый режим

  • Fixed — обычный ввод. Что написал — то и попадёт в запрос/в подстановку. У resource locator (когда выбираешь объект — таблицу, документ, sheet) Fixed означает выбор из dropdown «From list» или ручной ввод id одним значением.
  • Expression — поле подсвечивается оранжевым, а внутри можно писать формулу в фигурных скобках: {{ $json.field }}, {{ $now.toISO() }} и т.п. Значение пересчитывается на каждой итерации, для каждого пришедшего элемента.
Когда переключать в Expression
Значение зависит от данных или меняется между запусками

Если значение должно зависеть от того, что пришло из предыдущей ноды (имя пользователя, id записи, текст сообщения), от текущего времени или от любой логики — переключай в Expression. Если значение всегда одно и то же (метод HTTP POST, имя таблицы, фиксированный slug) — оставляй Fixed.

Как переключить режим

Три способа сделать одно и то же:

  1. Клик по Fixed | Expression в правом верхнем углу поля. Самый явный — видно сразу, в каком режиме поле.
  2. Иконка шестерёнки слева от поля → пункт Add Expression. Удобно когда хочешь сразу открыть expression builder (окно с деревом нод слева и preview значения справа).
  3. Drag-and-drop значения из панели Input слева в поле. Это самый частый способ в учебных задачах: открываешь NDV предыдущей ноды через мини-карту слева или раскрываешь панель Input, цепляешь нужное поле мышкой и тянешь в параметр текущей ноды. n8n при отпускании сам переключает режим в Expression и подставляет правильную ссылку — {{ $json.field }} или {{ $('Имя ноды').item.json.field }}.

В обратную сторону — клик по Fixed в углу. Содержимое expression-режима очистится, поле станет обычным текстовым.

Тонкость
У resource locator в Expression нет dropdown

В resource locator полях (например, выбор Data table или Spreadsheet) при переключении в Expression список объектов из dropdown пропадает — теперь ты должен сам подставить нужный id или slug формулой. Полезно когда id берётся из предыдущей ноды; если нужно просто выбрать существующий объект мышкой — оставайся в Fixed.

Типичные ошибки

  • Забыл переключить в Expression, написал формулу как текст. Поле в Fixed-режиме воспринимает {{ $json.name }} как обычные символы и отправит эту строку буквально. Признак: значение в preview / в результате выполнения содержит {{ и }} как есть. Лечение: переключить в Expression вручную (тумблер в правом верхнем углу) или перетянуть значение из панели Input заново.
  • Стёр выбор из resource locator, переключив в Expression и обратно. Переключение режима очищает содержимое поля. Если в Fixed уже был выбран нужный объект из списка — после Expression → Fixed его придётся выбрать заново.

Что такое expression

В n8n expression — это маленький кусочек кода в фигурных скобках, который динамически подставляет значение в поле ноды. Записывается так:

={{ <выражение> }}

Знак равенства = в начале — это маркер для n8n: «трактуй содержимое этой ячейки не как обычную строку, а как выражение, которое нужно вычислить». Без = n8n запишет в поле литеральный текст {{ ... }} как есть.

Внутри {{ }} пишется JavaScript-выражение. Самое частое — чтение значения из другой ноды.

Зачем нужны expressions

Каждое поле в параметрах ноды может быть либо фиксированным значением (например, URL https://api.example.com), либо expression'ом — формулой, которая считается на лету в момент исполнения ноды. Expressions нужны, когда значение зависит от данных, пришедших из предыдущей ноды.

К примеру
Где это встречается

К примеру, ты сделал ноду «прочитать строку из таблицы конфига», а следующая нода — HTTP Request к внешнему API. В URL этой HTTP-ноды можно подставить значение из конфига: ={{ $('Get Config').item.json.api_url }} — n8n при каждом исполнении возьмёт актуальный URL из таблицы, а не зашитый в ноду.

Как читать значения из других нод

Базовая формула: ={{ $('<имя_ноды>').item.json.<поле> }}

Разбор по частям:

  • $('<имя_ноды>') — обращение к ноде по имени (то, что написано на canvas под нодой). Имя — в одинарных кавычках, точно как на canvas.
  • .item — текущий элемент, который обрабатывается. У большинства нод элементов несколько (массив), item берёт тот, что соответствует текущей итерации.
  • .json — содержимое поля json элемента (там лежат все данные, которые нода вернула).
  • .<поле> — конкретное поле внутри json. Если поле вложенное — пишется через точку: .config.embedding_model.

Пошагово — как вставить expression в поле

  1. Кликни на иконку шестерёнки рядом с полем, выбери Add Expression (или нажми Expression в верхнем правом углу поля).
  1. В появившемся окне (expression builder) — слева список нод воркфлоу, можно кликать по полям и они вставятся в формулу.
  1. Справа — preview: видно, какое значение получится прямо сейчас. Если красным — формула или путь к полю некорректные.
  2. Закрой окно, поле остаётся в режиме expression (синий фон + иконка fx).
Типичная ошибка
`=` в начале — обязателен

Если скопировал готовое выражение из учебной задачи и вставил без = в начале — n8n запишет в поле текст {{ ... }} как есть, нода уйдёт делать HTTP-запрос на URL вроде ={{ $('...').item.json.api_url }} буквально. Симптом — нода падает с ошибкой Invalid URL или возвращает 404 на странную строку.

Типичные ошибки

  • Имя ноды не совпадает. В формуле $('Get Config') имя должно точно совпадать с подписью ноды на canvas. Если переименовал ноду — старые expressions, ссылающиеся на неё, сломаются.
  • Путь к полю не существует. .item.json.embedding_dimensions упадёт с undefined, если у элемента такого поля нет. Препетий — посмотри preview в expression builder, он сразу показывает значение или ошибку.
  • Забыл = в начале. Самая частая опечатка — описано в callout выше.
  • Кавычки внутри {{ }}. Внутри expression строки оборачиваются одинарными кавычками: ={{ 'Hello ' + $json.name }}. Двойные конфликтуют с JSON.

Когда expression не нужен

Иногда поле должно быть фиксированным. Например, метод HTTP-запроса (GET / POST) или имя коллекции, которое одинаково для всего воркфлоу — нет смысла читать его динамически. Простое правило: если значение меняется от итерации к итерации или зависит от пришедших данных — нужен expression; если значение постоянное — оставляй фиксированным.

Две формы обращения — разница только в источнике

В n8n у одной и той же ноды есть две формы прочитать поле из данных воркфлоу:

{{ $json.business_idea }}
{{ $('Form Trigger').item.json.business_idea }}

Обе достают одно и то же значение, но смотрят в разные места:

  • $json — это выход ноды, которая стоит прямо перед текущей. Короткая форма для «возьми поле из предыдущей ноды».
  • $('Имя ноды') — это обращение к любой ноде воркфлоу по имени, в том числе к той, что стоит на 2-3-5 шагов раньше.

Когда писать какую

Правило простое:

  • Хочешь взять данные у предыдущей ноды (стрелка идёт прямо в текущую) — пиши $json.поле.
  • Хочешь дотянуться до ноды глубже по цепочке (между ней и текущей есть промежуточные ноды) — пиши $('Имя ноды').item.json.поле.
Пример
Цепочка из трёх нод

Цепочка: Form Trigger → OpenAI → Send Message.

В Send Message обращения к двум разным источникам выглядят так:

  • к ответу OpenAI (предыдущая нода): ={{ $json.output }}
  • к полю формы (нода на два шага назад): ={{ $('Form Trigger').item.json['Email'] }}

Если в Send Message написать ={{ $json['Email'] }} — поле не найдётся, потому что у OpenAI в выходе нет Email. Это типичная ошибка студентов.

Почему короткая форма вообще существует

$json — это просто синоним для $input.item.json, то есть «текущий обрабатываемый элемент входа». А вход текущей ноды формируется из выхода предыдущей. Поэтому $json.поле и $('Предыдущая нода').item.json.поле всегда дают одно и то же значение — пока стрелка идёт напрямую.

$('Имя ноды') нужна там, где короткая форма не справится: между источником и текущей нодой есть промежуточные шаги, которые этого поля уже не передают (например, OpenAI вернул только текст ответа, и поле формы Email к моменту Send Message живёт только в выходе Form Trigger).

Типичные ошибки

  • Написал $json.Email в ноде после OpenAI. OpenAI возвращает output, а не Email. Решение — обратиться по имени: $('Form Trigger').item.json['Email'].
  • Имя ноды в $('...') не совпадает с подписью на canvas. В expression имя — буквально то, что написано под нодой на canvas. Переименовал ноду — все старые expressions, ссылавшиеся на её прежнее имя, сломаются.
  • Забыл .item после имени ноды. Полная форма — $('Имя').item.json.поле. Без .item обращаешься не к данным, а к самой ноде как объекту — поле не найдётся.

Запустить одну ноду, не дёргая весь workflow

Execute step — это кнопка внутри панели ноды (открывается двойным кликом по ноде). По смыслу — младший брат большой кнопки Execute workflow внизу canvas: та запускает весь воркфлоу с начала, эта — только текущую ноду. Удобно, когда настраиваешь параметры и хочешь сразу видеть, что отдаёт нода, не перезапуская всю цепочку.

Как это работает

Когда ты жмёшь Execute step, n8n берёт уже существующие данные на входе ноды (output предыдущей ноды из последнего прогона) и прогоняет через эту ноду. Сама нода выполняется реально: HTTP-запрос уходит, запись в базу пишется, письмо отправляется. Это не «сухой прогон», а полноценное выполнение — только одного шага.

  • key: Execute workflow

value: Запускает весь воркфлоу от триггера. Нужен полный прогон с самого начала. Каждый раз заново дёргает все API.

  • key: Execute step

value: Запускает только эту ноду. Берёт вход из последнего output предыдущей ноды. Идеально для подбора параметров и отладки.

  • key: Когда работает

value: Только если на входе ноды есть данные с прошлого прогона. Если ни разу не запускал предыдущие — кнопка либо ничего не вернёт, либо упадёт «No input data».

Типичный сценарий: запустил весь воркфлоу один раз (Execute workflow), увидел что нода HTTP Request отдаёт не то, поправил поле URL, нажал Execute step — увидел новый результат за секунду. Предыдущие ноды повторно не выполнялись.

У триггеров кнопка называется иначе
Смысл тот же — запустить один шаг и получить тестовое событие

У некоторых триггерных нод n8n кнопка Execute step подписана по-другому, потому что у триггера нет «входа» — он сам должен сходить за событием. Например, у RSS Trigger она называется Fetch Test Event: клик подтянет последние записи из ленты и положит их на выход ноды как тестовый input для остального воркфлоу. Работает идентично Execute step обычной ноды — точно так же используется для отладки и подбора параметров без полного прогона.

Важно про побочные эффекты
Каждый Execute step — это реальный вызов

Нода с записью в базу или отправкой сообщения каждое нажатие Execute step делает реально: 5 нажатий = 5 записей / 5 писем. Для отладки таких нод используй тестовые данные или подменяй credentials.

Типичные ошибки

  • Жать Execute step на ноде, у которой ещё не было входных данных → «No input data», нода не запустится. Сначала прогони весь воркфлоу через Execute workflow или хотя бы триггер вручную, чтобы у каждой ноды появился output.
  • Использовать Execute step для проверки изменений в предыдущих нодах → текущая нода работает со старым output, изменения не подхватятся. Если поправил предыдущую ноду — перезапусти её, а потом уже текущую.
  • Тестировать ноду с побочным эффектом (запись в БД, отправка письма) на проде → каждый клик создаёт реальную запись. Использовать тестовый credential или временно подменять параметры на безопасные.

В блоке Output появится объект из трёх полей: url с исходной длинной ссылкой, username с именем реферера, original_url с собранной реферальной ссылкой.

Bearer Token авторизация

Bearer Token — самая распространённая схема авторизации в современных API. Сервер выдаёт пользователю длинный токен (строку из букв и цифр), а клиент в каждом запросе посылает заголовок Authorization: Bearer <твой_токен>. Если токен валиден — сервер обрабатывает запрос, если нет — возвращает 401 Unauthorized.

Слово Bearer буквально переводится как «предъявитель». То есть «кто предъявил этот токен — тот и пользователь». В этом и суть: токен сам по себе достаточен для доступа, никакого пароля больше не нужно.

Как это работает

Заголовок собирается строго в этом формате:

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
  • Имя заголовка — всегда Authorization (с большой буквы).
  • Префикс — слово Bearer, затем один пробел, затем сам токен.
  • Токен — строка которую выдал сервис (часто JWT, OAuth access token, API-key). Внутрь не подсматриваем — просто передаём как есть.

Чем Bearer отличается от соседних схем:

  • Basic Auth — передаёт логин и пароль вместе, закодированные в base64. Менее безопасно: если токен утёк, его можно отозвать, а пароль — придётся менять.
  • Header API Key (api-key: ..., X-API-Key: ...) — просто ключ в произвольном заголовке без префикса. Старый паттерн, ещё встречается у небольших сервисов.

В n8n этот заголовок собирается двумя путями: либо через предустановленный Generic Credential Type = Header Auth (вписываешь Name = Authorization, Value = Bearer <token>), либо через готовый credential конкретного сервиса (если он есть в списке) — там префикс Bearer уже зашит.

Важно
Bearer Token = пароль

Кто угодно с этим токеном получит доступ к твоему аккаунту в этом сервисе. Не клади токен в публичные репозитории, не отправляй в чаты, не вписывай в URL (URL пишется в логи серверов). Внутри n8n храни всегда через credential, а не как plain-текст в поле параметра.

Типичные ошибки

  • Пропустил слово Bearer. Послал просто Authorization: <token> — сервер вернёт 401. Префикс обязателен.
  • Лишний пробел или табуляция между Bearer и токеном. Должен быть ровно один пробел. Скопировал из чата вместе с невидимым символом — получишь 401.
  • Передал токен в URL вместо заголовка. Некоторые API такое поддерживают, но это плохо: URL виден в логах, истории браузера, аналитике. Всегда — заголовок.

POST с JSON-телом в HTTP Request

Большинство современных API принимают данные не в URL, а в теле запроса в формате JSON. Это значит: метод POST (или PUT/PATCH), заголовок Content-Type: application/json и сам JSON-объект в body. В n8n всё это включается через один переключатель.

Как это работает

Внутри ноды HTTP Request после Method есть переключатель Send Body. Включаешь — появляется Body Content Type с выпадающим списком. Для JSON выбирай значение JSON. После этого открываются два режима ввода:

  • Using JSON — вписываешь готовый JSON-объект целиком. Удобно когда структура нетривиальная или есть вложенные объекты. Выражения через {{ }} тут тоже работают — можно вставить значение из предыдущей ноды прямо внутрь JSON.
  • Using Fields Below — заполняешь таблицу с парами Name/Value. n8n сам соберёт JSON-объект из этих полей. Подходит для плоских объектов из 2–4 полей, без вложенности.

Заголовок Content-Type: application/json n8n проставляет автоматически — отдельно его прописывать в Send Headers не нужно.

К примеру
Один и тот же запрос двумя способами

Объект { "title": "Hello", "tags": ["news", "test"] } через Using JSON пишется как есть — одной вставкой. Через Using Fields Below придётся завести строку title=Hello и для массива tags использовать выражение ["news", "test"] в Value — что менее читаемо. Чем сложнее структура, тем выгоднее Using JSON.

Типичные ошибки

  • Невалидный JSON в Using JSON. Лишняя запятая, одинарные кавычки вместо двойных, пропущенный } — и сервер вернёт 400 Bad Request. Проверяй структуру глазами или прогоняй через любой JSON-валидатор перед запуском.
  • Дублирование заголовка Content-Type. Если вручную в Send Headers добавить ещё один Content-Type, могут поехать кодировки или сервер откажет. Не трогай — n8n сам поставит правильный.
  • POST без Send Body. Метод включил, тело — нет. На выходе уйдёт пустой POST-запрос, сервер скорее всего вернёт ошибку валидации.

Header Auth — универсальный credential для API с ключом в заголовке

Многие API-провайдеры (oauth-эндпоинты Сбера, простые сервисы с API-key, custom-аутентификация) ожидают, что ты пришлёшь авторизацию в HTTP-заголовке — например, Authorization: Basic ... или X-Api-Key: .... В n8n для этого есть универсальный тип credential — Header Auth. Он работает в любой ноде HTTP Request.

Это generic-credential: один credential подходит для десятков сервисов. Для каждого нового API не нужно создавать отдельный кастомный тип — достаточно знать, какой заголовок ждёт сервер и какое значение в него положить.

Пошагово

  1. В n8n открой Credentials (меню слева) → Add credential → в поиске введи Header Auth → выбери его.
  1. Заполни два поля:
  • Name — имя заголовка (например, Authorization или X-Api-Key). Просто слово, без двоеточия и значения.
  • Value — значение заголовка целиком, включая префикс если он нужен (например, Basic MGJj...== или Bearer eyJhbGc...).
  1. В шапке credential задай понятное имя — например, <Имя сервиса> OAuth или <Имя сервиса> API. Это имя ты будешь видеть в выпадашке HTTP Request ноды.
  1. Нажми Save. Готово — credential можно подключать к HTTP Request ноде.

Как использовать в HTTP Request

В ноде HTTP Request → раздел Authentication → Predefined Credential Type → в списке Generic Auth Type выбери Header Auth → ниже выпадашка с твоим credential.

Не забудь префикс Basic / Bearer
Самый частый источник 401

Если API использует OAuth2 — в Value пиши Basic <ключ> или Bearer <токен> целиком, со словом и пробелом. Только сам ключ без префикса — частая причина 401. Проверь заголовок curl-ом сначала, потом перенеси в credential.

Когда Header Auth не подойдёт

Если сервер ждёт значение в query-параметре (?api_key=...), а не в заголовке — Header Auth не сработает. Используй тип Query Auth (тоже generic) или подставляй ключ вручную через выражение в URL ноды.

Если в заголовке нужно динамическое значение (например, Authorization: Bearer <живой_access_token>) — credential не подходит вообще: значение в credential статическое. Пиши заголовок выражением прямо в HTTP Request: Authorization = ={{ "Bearer " + $('Get Token').item.json.access_token }}.

⚡Как передать данные из одной ноды в другую через drag-and-dropстатья · ~2 мин▾

Drag-and-drop из панели Input — самый быстрый способ связать ноды

В n8n каждая нода видит данные предыдущих нод в панели Input слева от параметров. Чтобы подставить любое поле из этого Input в параметр текущей ноды, не нужно вручную писать expression — достаточно перетащить поле мышкой. n8n сам подставит правильное выражение со ссылкой на нужную ноду.

Как это работает

  1. Открой параметры ноды двойным кликом. Слева автоматически появится панель Input — это выходные данные предыдущей ноды (или одной из предыдущих, если их несколько).
  2. Разверни в Input структуру до нужного поля — она показывается как дерево вложенных объектов: message → chat → id, output → [0] → content → [0] → text и так далее.
  3. Перетащи нужное поле мышкой прямо в поле параметра справа. В поле появится выражение вида ={{ $('<имя предыдущей ноды>').item.json.<путь.к.полю> }}.
  4. Поле в правой панели окрасится синим — это значит, что параметр теперь не литерал, а expression: на каждом запуске значение будет браться из реальных данных.
К примеру
Откуда n8n берёт имя ноды

К примеру, если предыдущая нода называется Telegram Trigger, и ты перетащил из неё поле message.text — n8n запишет ={{ $('Telegram Trigger').item.json.message.text }}. Имя ноды берётся ровно такое, как написано на canvas. Иногда — если переименуешь ноду уже после drag-and-drop, выражение сломается, потому что в формуле останется старое имя.

Откуда выбрать источник данных

Если у ноды несколько входов или нужно дотянуться не до прямого предыдущего шага, а до более раннего — сверху панели Input есть переключатель источника. К примеру, в финальной ноде workflow можно вывести Input от первого триггера, чтобы достать оттуда поле, которого нет в выходе ближайшей предыдущей ноды.

Типичные ошибки

  • Перетащил не на то поле параметра. Drag-and-drop срабатывает на любом текстовом поле — легко уронить значение не в Text, а в Chat ID, и наоборот. После перетаскивания проверь, что в нужном поле появилось ожидаемое выражение.
  • Перетащил из объекта-обёртки, а не из конкретного поля. Если потянуть за message, а не за message.text — n8n подставит весь объект, в текстовое поле уйдёт [object Object] или JSON-строка. Раскрывай дерево до примитивного значения.
  • Сломался путь после переименования ноды. В выражении $('Telegram Trigger') имя ноды фиксированное. Если потом переименовать Telegram Trigger в что-то другое — нужно либо вернуть старое имя, либо переписать выражение вручную.

Когда лучше не использовать drag-and-drop

Если значение должно быть фиксированным (например, имя коллекции, метод HTTP, статичный текст приветствия) — drag-and-drop не нужен, просто впечатай значение в поле как обычный текст. Drag-and-drop — это про динамические параметры, которые меняются от запуска к запуску в зависимости от входных данных. </content> </invoke>

Сначала проверь в Headers, потом переноси в credential

Когда получаешь 401 и не уверен, правильно ли написан заголовок в credential — не угадывай. Добавь заголовок прямо в ноду через раздел Send Headers, проверь что запрос проходит, и только потом перенеси проверенные значения в credential.

Шаг 1 — добавь заголовок напрямую в ноду

Открой ноду HTTP Request. В разделе Authentication временно выставь None — иначе credential и ручной заголовок могут конфликтовать.

Раскрой раздел Headers → включи Send Headers → нажми Add Parameter. Появится строка с двумя полями:

Параметры заголовка
ПолеЧто вписать
NameИмя заголовка — например, Authorization
ValueЗначение целиком — например, Bearer eyJhbGc...

Пробела перед словом Bearer быть не должно. После токена — тоже ничего лишнего.

Шаг 2 — выполни запрос и убедись что 200

Нажми Execute step на этой ноде. Если ответ 200 — заголовок написан правильно. Если снова 401 — ошибка в самом токене или его формате, не в credential.

Шаг 3 — скопируй значения и перенеси в credential

Пока нода открыта, скопируй:

  • точное значение поля Name (например, Authorization)
  • точное значение поля Value (например, Bearer eyJhbGc...)

Потом:

  1. Удали строку из Headers (или выключи Send Headers).
  2. В Authentication верни обратно Predefined Credential Type → Header Auth.
  3. Открой или создай credential, вставь скопированные значения — Name и Value один в один.
Частая ошибка
Лишний пробел или перенос строки в Value

При вставке токена из другого источника (Telegram, браузер, PDF) в конец строки иногда попадает невидимый пробел или перенос строки. Credential сохранится без ошибки, но каждый запрос будет возвращать 401. Проверь: в поле Value не должно быть ничего после токена.

📖Зачем нормализовать данные через Setстатья · ~2 мин▾

Set приводит разнородный вход к одному виду

Разные источники приносят данные по-разному: вебхук отдаёт одни поля, чат — другие, база — третьи. У одного userId, у второго user_id, у третьего просто id. Если каждую следующую ноду учить понимать все варианты — workflow быстро превращается в кашу.

Нода Set (в новых версиях её называют Edit Fields) решает это в одном месте: на выходе получаешь предсказуемую структуру с одинаковыми именами полей, нужными типами и без мусора. Дальше по цепочке все ноды работают с известным контрактом.

Зачем это нужно

  • Стабильный контракт для следующих нод. Меняешь источник — правишь только Set, остальная цепочка не трогается. Без нормализации замена одного триггера на другой ломает всё, что было ниже.
  • Чистка лишнего. Включаешь Include Other Input Fields = off — оставляешь только то, что задал руками. Payload не пухнет, в логах видно главное, в LLM-промпте не утекают сырые поля API.
  • Приведение типов. HTTP отдаёт "123" строкой, IF/Switch и арифметика ждут число. В Set явно выбираешь тип Number / Boolean / Date — один раз, без Number($json.x) в каждой ноде.
  • Дефолты для пропусков. Источник не прислал lang — Set подставит "ru". Дальше можно не писать ($json.lang || 'ru') в каждом выражении.
  • Защита нод ниже. Code-нода, LLM-промпт, sub-workflow получают известный набор ключей и не вылетают на undefined.
Когда это особенно важно
перед БД, LLM и sub-workflow

Перед записью в БД или Data Table структура должна совпадать с колонками. Перед LLM-промптом или AI Agent — контракт промпта зашит в текст и любое лишнее или недостающее поле ломает ответ. На входе sub-workflow Execute Workflow ждёт известный объект — это твоя «функция», у неё сигнатура.

Типичные ошибки

  • Include Other Input Fields = on оставлен по умолчанию → к твоим аккуратным полям приклеивается весь сырой вход → в логах шум, в LLM лезут лишние ключи. Выключай, если цель — нормализация.
  • Тип поля оставлен Auto → строка "true" так и осталась строкой → IF с условием is true срабатывает не так, как ждёшь. Задавай тип явно.
  • Set ставят после каждой ноды «на всякий случай» → одно и то же поле задано в трёх местах → при отладке непонятно, чья это запись. Один Set на границу (после триггера, перед БД, перед LLM) — достаточно.
  • Нормализуют там, где нужна одна-две перезаписи на лету. Если поля и так совпадают, а добавить надо одно — это просто Edit Fields, не «нормализация». Лишняя нода без пользы.

📖$now — текущий момент времени в выражениях n8nстатья · ~2 мин▾

$now — готовый объект текущего момента

$now — встроенная переменная n8n, доступная в любом Expression-поле. Это не функция и не строка, а уже готовый объект Luxon DateTime — момент времени в часовом поясе инстанса n8n. Пересчитывается на каждой итерации каждого запуска.

Если просто подставить {{ $now }} в строку — Luxon сам приведёт объект к ISO-строке вида 2026-05-28T14:35:12.345+03:00. Но настоящая польза — в методах, которые можно вызывать прямо в выражении.

Базовые методы

::key-value

  • $now.toISO() — строка ISO 8601. Самый частый формат для API и БД.
  • $now.toFormat("yyyy-MM-dd HH:mm") — произвольный формат. Токены Luxon: yyyy, MM, dd, HH, mm, ss, EEE (день недели коротко).
  • $now.toMillis() — Unix timestamp в миллисекундах (число).
  • $now.toUnixInteger() — Unix timestamp в секундах (число).
  • $now.plus({days: 7}) / $now.minus({hours: 3}) — арифметика по календарю. Юниты: years, months, weeks, days, hours, minutes, seconds.
  • $now.startOf('day') / $now.endOf('month') — округление к началу/концу периода. Юниты те же.
  • $now.toUTC() / $now.setZone('Europe/Moscow') — перевод в другой часовой пояс.
  • $now.year / $now.month / $now.day / $now.hour / $now.weekday — доступ к отдельным компонентам. weekday: 1 — понедельник, 7 — воскресенье.

::

Родственная переменная

$today — то же самое, что $now.startOf('day'): текущая дата с обнулёнными часами/минутами/секундами. Удобно, когда нужна «дата без времени» (например, для группировки записей по дню).

Где встречается чаще всего
Типичные кейсы для `$now`

В URL HTTP-запроса: ?from={{ $now.minus({days: 1}).toISO() }}. В Set / Edit Fields — записать момент обработки в Data Table. В If/Filter — сравнить дату из входных данных с «не старше 7 дней» ($now.minus({days: 7})). В HTTP-заголовке: Date: {{ $now.toFormat("EEE, dd MMM yyyy HH:mm:ss 'GMT'") }}.

Часовой пояс
`$now` возвращает момент в TZ инстанса, не в UTC

По умолчанию n8n работает в часовом поясе операционной системы инстанса. Если сравниваешь $now с датой, которую API отдал в UTC, — будет ошибка на N часов. Перед сравнением приводи обе стороны к одному поясу: $now.toUTC() либо вызывай setZone(...) на дате из входа.

Типичные ошибки

  • Пишут $now() со скобками → $now это уже готовый объект, а не функция, скобки лишние.
  • Складывают как число: $now + 86400000 → не работает; календарную арифметику делает $now.plus({days: 1}).
  • Передают $now напрямую в БД, где поле ждёт bigint timestamp → улетит ISO-строка вместо числа; явно вызови .toMillis() или .toUnixInteger().
  • Сравнивают $now с пустым полем даты из входа → выражение упадёт; перед сравнением поставь проверку is not empty.
  • Используют is equal to для «сегодня» → совпадение только до миллисекунды; для «в течение этого дня» делай диапазон через $now.startOf('day') и $now.endOf('day').

📖Маппинг колонок Data Table — Map Automatically vs Manuallyстатья · ~2 мин▾

Что такое маппинг колонок

Когда нода Data Table пишет строку в таблицу (операция Insert, Update или Upsert), ей нужно понять — какое поле из входящих данных в какую колонку положить. Этот процесс называется маппингом. В секции Columns ноды два режима, разница — кто этим занимается: n8n сам или ты руками.

Два режима

Режимы маппинга
РежимКак работаетКогда брать
Map Automaticallyn8n берёт входящий объект, смотрит на колонки таблицы и сопоставляет их по совпадению имён: поле name уходит в колонку name, phone в phone, и так далее. Несовпавшие поля игнорируются.Когда поля во входящем объекте уже названы как колонки таблицы — обычно после ноды Edit Fields, которая привела данные к нужному контракту. Это режим по умолчанию.
Map Each Column ManuallyПод каждую колонку таблицы появляется отдельное поле — туда руками (или через drag-and-drop из левой панели) кладёшь значение или выражение.Когда имена не совпадают (поле во входе называется client_name, а колонка — name), нужно вычислить значение на лету ({{ $now.toISO() }}) или часть колонок оставить пустыми.
Автомат работает по именам
Совпадение имён — единственное правило

Map Automatically сопоставляет колонки и поля исключительно по имени, без учёта регистра. Если у тебя на входе Name (с большой буквы), а в таблице колонка name (с маленькой) — поле не подцепится. Имя поля во входящем объекте должно совпадать с именем колонки символ в символ.

Что готовить на входе под автомат

Чтобы Map Automatically сработал, перед нодой Data Table обычно ставят Edit Fields (Set) в режиме Manual Mapping. Там собирают объект с полями именно тех имён, что есть в таблице — добавляют недостающие (lead_id, received_at), переименовывают расходящиеся (client_name → name). После этого Data Table подхватит всё автоматически.

Типичные ошибки

  • Включил Map Automatically, но имена полей и колонок не совпадают → строка вставилась с пустыми значениями → переименуй поля в Edit Fields перед таблицей или переключи маппинг на Manually.
  • Имя поля во входе — с пробелом или дефисом ("client name") → автомат не нашёл колонку → нормализуй имена до kebab/snake_case без пробелов.
  • В Manually оставил пустые поля → колонка запишется как null → проверь что значения проставлены везде где нужно.

🚀Stretch▾

Сейчас бот разбирает одно сообщение буквально: всё, что до |, считает ссылкой, а всё, что после — реферером.

Если пользователь забудет поставить | или пришлёт данные не в том формате, бот соберёт неправильную ссылку: например, добавит ?ref=undefined и запишет мусорные данные в журнал.

Переделай логику в многошаговый сбор параметров.

Бот должен не ждать одну строку со всеми данными сразу, а задавать вопросы по очереди:

  1. попросить исходную ссылку;
  1. попросить utm_source;
  1. попросить utm_campaign;
  1. попросить utm_medium;
  1. собрать итоговый URL.

Так пользователь вводит каждый параметр отдельным сообщением, а бот хранит промежуточные ответы между шагами.

Подумай, где удобнее держать состояние диалога: внутри сессии Chat Trigger или через отдельное хранилище.

Когда все параметры собраны, собери ссылку через expression.

Для значений на русском языке используй encodeURIComponent, чтобы они корректно попали в URL.
