M-ErrorHandling-03 Пакетный синхронизатор

Представь: твой воркфлоу каждый час должен синхронизировать накопившиеся pending-заявки с внешним CRM, но сеть нестабильная и API иногда отдаёт пятисотки. Если просто молча упасть — потеряешь данные клиента. Если ретраить вечно — рискуешь устроить лавину запросов в момент когда внешний сервис уже на коленях. Соберём два связанных воркфлоу: главный обходит pending-очередь, шлёт POST с Retry On Fail 3 раза и Wait Between Tries 2 секунды, переводит строку в done или failed. Параллельный sub-workflow раз в сутки в 3 часа ночи забирает все failed и пробует синхронизировать ещё раз — простой dead letter queue на встроенной таблице, без message broker. Тестировать будем на httpbin.org со случайным выбором кода 200/500 в URL — половина запросов реально падает, половина проходит.

Собери пакетный синхронизатор с Retry On Fail и DLQ Lite

Главный воркфлоу с Loop по pending-заявкам, HTTP POST с Retry 3 попытки + Wait Between Tries 2 сек + Continue On Error через error-выход → Mark Done / Mark Failed. Параллельный sub-workflow DLQ-повтор раз в сутки забирает failed и пробует ещё один раз — это и есть DLQ Lite на встроенной Data Table.

Что собираем

Два воркфлоу.

Главный читает pending-строки из Data Table «Очередь синхронизации», для каждой шлёт POST на внешний CRM с Retry On Fail = 3 попытки, Wait Between Tries = 2 секунды. Если ответ 200 — строка становится done. Если все три попытки провалились — On Error: Continue (using error output) уводит поток в error-ветку, строка помечается failed с записанным last_error. После цикла Edit Fields собирает итоговый отчёт, Telegram шлёт его в чат.

Sub-workflow «DLQ-повтор» запускается раз в сутки в 03:00, забирает все failed строки и пробует каждую ещё раз без ретраев. Если прошло — done. Если снова провал — Telegram шлёт уведомление «Заявка X мертва, нужна ручная разборка».

Главное новое здесь — паттерн DLQ Lite: error-ветка фиксирует факт провала в Data Table вместо того чтобы молча падать, а отдельный суточный воркфлоу подбирает то что не прошло с первого раза. Тестировать будем на httpbin.org/status/200 и httpbin.org/status/500 — половина запросов намеренно падает пятисоткой.

1Получи токен у @BotFather▾

Бот в Telegram создаётся не в n8n и вообще не в браузере. Telegram сделал для этого специального служебного бота — @BotFather. Через него любой пользователь Telegram создаёт собственного бота и получает на него токен — длинную строку, которая даёт полный доступ к этому боту. Этот токен мы потом вставим в n8n, и все ноды Telegram через него будут говорить от имени твоего бота.

Этот шаг ты делаешь в Telegram-приложении, n8n пока даже не открывай.

📖@BotFather: как создать собственного бота в Telegramстатья · ~2 мин▾

@BotFather: как создать собственного бота в Telegram

@BotFather — это официальный бот Telegram, через которого создаются все остальные боты. Других способов завести бот нет: ни сайта, ни приложения, ни консоли — только переписка с этим аккаунтом.

Зачем это нужно

Чтобы любой workflow (в n8n, Make, на собственном бэкенде) мог слушать сообщения и отвечать от имени бота, ему нужен токен Bot API. Токен выдаётся при создании бота через @BotFather. Без токена ни одна интеграция с Telegram не заведётся: токен — это пара «логин-пароль» бота при общении с Telegram Bot API.

Как это работает

Открываешь Telegram, в поиске находишь аккаунт с username BotFather (галка подлинности рядом с именем — синяя), и пишешь ему как обычному собеседнику. Команды начинаются со слэша:

  • /newbot — создать нового бота. @BotFather спросит две вещи:
  • отображаемое имя (любое, может быть с пробелами и кириллицей) — то, что увидят пользователи в шапке чата;
  • username (только латиница, цифры, подчёркивание; обязательно заканчивается на _bot или bot) — постоянный идентификатор, по которому бот ищется и не меняется.
  • /mybots — список всех твоих ботов, доступ к настройкам (имя, описание, аватарка, команды, токен).
  • /revoke — отозвать старый токен и получить новый (если старый утёк).

После /newbot приходит сообщение с токеном — длинная строка вида 123456789:ABCdefGHI_jklMNO.... Это и есть тот самый Bot API token: его копируешь и сохраняешь в безопасное место (менеджер паролей, заметки с шифрованием) — но не в публичный репозиторий и не в чат.

Типичные ошибки

  • Потерять токен — @BotFather показывает токен один раз; чтобы получить его снова, нужно открыть /mybots → <твой бот> → API Token. Если не нашёл — можно сгенерировать новый через /revoke, но тогда старый перестанет работать во всех уже настроенных интеграциях.
  • Скинуть токен скриншотом / в открытый чат — токен даёт полный контроль над ботом. Любой, кто его увидит, может слать сообщения от имени бота и читать всё, что ему приходит. Если утёк — сразу /revoke.
  • Username без _bot на конце — @BotFather отклонит регистрацию; правило жёсткое, придётся подбирать ещё.
  • Перепутали username и отображаемое имя — отображаемое имя меняется в любой момент через /setname, а username меняется только пересозданием бота. Если задумывался серьёзный бот — username стоит выбрать осознанно.

Когда не подходит

Через @BotFather нельзя создать юзербот (бот, который ходит под обычным аккаунтом) — это уже не Bot API, а MTProto, и токен оттуда не подходит. Также @BotFather не нужен, если бот уже создан кем-то другим и тебе просто передали готовый токен — тогда сразу заводишь credential в нужном сервисе и работаешь с ним.

Что такое DataTable

DataTable — встроенное хранилище n8n, что-то среднее между Excel-листом и базой данных. Лежит внутри n8n, не требует отдельного сервера и подключений, редактируется прямо в UI. Каждая таблица принадлежит проекту: видна и доступна всем воркфлоу этого проекта.

В одной таблице — фиксированная схема колонок (имя + тип) и произвольное количество строк. Доступ из воркфлоу — через ноду Data table (resource = Row, операции Get rows, Insert row, Update row, Delete row).

Когда выбирать DataTable

DataTable подходит для случаев, где данных немного и нужно удобство редактирования через UI:

  • Конфиги воркфлоу — настройки агента, ключи API-описаний, лимиты. Меняешь значение в ячейке, воркфлоу при следующем запуске берёт новое.
  • Небольшие справочники — список разрешённых пользователей, маппинг каналов на промпты, словарь ответов.
  • Состояние между запусками — счётчики, последний обработанный ID, кеш токена с TTL.
Когда DataTable, когда Postgres
Главное отличие — назначение

DataTable — для редко меняющихся данных, которые удобно править руками: конфиги, справочники, флаги. Postgres — для журналов, истории, аналитики: туда воркфлоу пишет много строк, оттуда читает выборки с условиями и сортировкой.

Условный порог: если ожидается больше 1000 строк или нужна аналитика с агрегацией — это Postgres. Если до 100 строк и они меняются вручную — это DataTable.

Если два воркфлоу одновременно дописывают строку в одну и ту же DataTable — порядок может смешаться, а часть операций — провалиться. Для счётчиков и журналов, которые пишут часто из разных триггеров, выбирай Postgres.

Частый паттерн: конфиг продукта в DataTable

Параметры AI-агента (системный промпт, имя модели, имя коллекции Qdrant, имя таблицы логов) можно зашить прямо в ноды — но тогда любая правка требует открыть редактор, найти ноду, поменять, опубликовать новую версию. Альтернатива — вынести всё в одну строку DataTable.

Схема таблицы (например, Agent Config) — две колонки:

  • key (string) — идентификатор конфига: main, support, telegram.
  • config (string) — JSON со всеми параметрами.

Внутри config — обычный JSON с понятной структурой:

{
  "agent": {
    "system_prompt": "Ты помощник интернет-магазина. Отвечай вежливо и по делу.",
    "model": "gpt-4.1-mini",
    "temperature": 0.3
  },
  "qdrant": {
    "collection": "<имя_коллекции>",
    "embedding_model": "text-embedding-3-small"
  },
  "log_table": "agent_logs"
}

Как воркфлоу читает этот конфиг

  1. Триггер запускает workflow (новое сообщение от пользователя).
  2. Нода Get rows читает строку из Agent Config по ключу (например, key = main). Получает config как строку.
  3. Нода Set (или Code) парсит JSON и раскладывает поля по плоским переменным: system_prompt, model, collection, log_table.
  4. AI Agent и его sub-ноды (Qdrant KB, Postgres Chat Memory) ссылаются на эти переменные через выражения вида {{$json.system_prompt}}. При каждом запуске берётся актуальное значение.

Что это даёт

  • Правка без редактирования воркфлоу. Сменил системный промпт — открыл DataTable, поменял ячейку, сохранил. Воркфлоу подхватит новое значение при следующем запуске.
  • Один воркфлоу — много конфигураций. Тот же workflow обслуживает несколько контекстов (разные продукты, разные каналы, разные клиенты) — каждый со своей строкой.
  • История правок в одном месте. Легче откатить ошибочное изменение.
Двухуровневый конфиг
Product config + Channel config

В сложных продуктах конфиг разделяют на два уровня: product config — общее для всего продукта (база знаний, embedding-модель, имя таблицы логов), и channel config — что меняется по каналам (промпт, модель, лимит длины ответа). Workflow тогда делает два последовательных Get rows и сливает результаты.

Как создать Data Table из CSV

В n8n можно создать таблицу не вручную по колонкам, а сразу импортом из CSV-файла. Удобно когда у тебя уже есть готовые данные — n8n определит колонки из заголовка CSV и зальёт строки одним действием.

  1. В своём проекте n8n открой вкладку Data tables. В правом верхнем углу нажми Create data table.
  1. Откроется диалог Create new data table. В поле Data table name * впиши имя новой таблицы — это обязательное поле.
  2. Под полем имени по умолчанию выбран режим From scratch (создать с нуля по колонкам). Переключи на Import CSV.
  1. Под радио-кнопками появится дроп-зона Drop file here or click to upload. Перетащи в неё свой CSV-файл (или кликни по зоне и выбери файл в проводнике).
  1. После загрузки в дроп-зоне появится имя файла. Убедись что галка My CSV file contains a header row стоит — n8n возьмёт первую строку CSV как названия колонок. Если CSV без заголовка — сними галку, тогда n8n сгенерирует имена column_1, column_2 и т.д. сам.
  2. Нажми Create.
  1. Откроется второй шаг — Set data table columns. Сверху n8n покажет сообщение вида 'filename.csv' has been uploaded successfully. We found N columns and M rows — проверь что число колонок и строк совпадает с тем, что ты ждал.
  2. Под сообщением — список колонок: имя (из заголовка CSV) и определённый тип (String, Number, Boolean, Date). Если тип определился неправильно — поменяй его в дропдауне Data Type напротив нужной колонки.
  3. Снять галку слева от колонки = пропустить эту колонку при импорте. По умолчанию все галки стоят.
  4. Нажми Create.
  1. Готово. Откроется страница таблицы со всеми импортированными строками. Видно что к пользовательским колонкам n8n автоматически добавил служебные id, createdAt, updatedAt. Из этой страницы можно вручную добавить ещё строки (Add Row) или колонки (Add Column).

n8n определил меньше колонок чем в CSV

Если на шаге Set data table columns колонок меньше, чем в твоём CSV — проверь:

  • Если CSV выгружен из Google Sheets — выгружай через Файл → Скачать → Значения, разделённые запятыми (.csv). Если открыть исходный CSV в Sheets как таблицу и выгрузить заново — Sheets может изменить формат данных.

Schedule Trigger в n8n: расписания и cron

Schedule Trigger — стартовая нода n8n, которая запускает воркфлоу по расписанию: раз в минуту, каждый час, по будням в 9:00 или по любому cron-выражению. В отличие от polling-триггера, Schedule Trigger никуда не ходит за данными — он просто срабатывает по таймеру и передаёт управление следующей ноде.

Как это работает

В ноде есть один основной параметр — Trigger Interval. От его значения зависит набор дополнительных полей:

  • Seconds / Minutes / Hours — запуск каждые N единиц. Поле Seconds Between Triggers (или Minutes/Hours).
  • Days — раз в N дней в указанное время. Поля: Days Between Triggers, Trigger at Hour, Trigger at Minute.
  • Weeks — по выбранным дням недели в указанное время. Поля: Weeks Between Triggers, Trigger on Weekdays (мульти-выбор), Trigger at Hour/Minute.
  • Months — раз в N месяцев в указанный день месяца. Поля: Months Between Triggers, Trigger at Day of Month, Trigger at Hour/Minute.
  • Custom (Cron) — произвольное расписание через cron-выражение. Поле: Expression.

Cron-выражения

Для нестандартного расписания выбирай Custom (Cron). Формат — стандартный cron из 5 полей: минута час день_месяца месяц день_недели. Звёздочка * значит «любое значение», слеш */N — «каждые N единиц», запятая — перечисление, дефис — диапазон.

Примеры cron-выражений
*/15 * * * *    — каждые 15 минут
0 * * * *       — в начале каждого часа
0 9 * * 1-5     — по будням в 9:00
0 9,18 * * *    — каждый день в 9:00 и 18:00
0 0 1 * *       — в полночь первого числа каждого месяца
30 3 * * 0      — по воскресеньям в 3:30 ночи
Часовой пояс
Schedule Trigger работает в timezone инстанса n8n

Все расписания привязаны к часовому поясу самого n8n. Если инстанс в UTC, а ты ждёшь «9:00 по Москве» — поставь cron 0 6 * * * (UTC = МСК − 3) или поменяй timezone в настройках воркфлоу (Settings → Timezone). На школьной платформе у каждого студента свой инстанс — проверь его timezone, прежде чем настраивать критичное расписание.

Несколько расписаний в одной ноде
Список интервалов под кнопкой Add Rule

Внутри одной ноды Schedule Trigger можно добавить несколько правил кнопкой Add Rule — например, «каждые 5 минут в рабочие часы» + «раз в час ночью». Каждое срабатывание любого правила запускает воркфлоу с одного и того же выхода ноды.

Типичные ошибки

  • Воркфлоу не срабатывает в ожидаемое время → инстанс в другом timezone. Проверь Settings → Timezone в воркфлоу и сравни с тем, что ожидаешь.
  • Расписание «каждый час» срабатывает каждые ~59-61 минут → так и должно быть для режимов Seconds/Minutes/Hours: они считают от момента активации. Чтобы попадать «в ровно 0 минут», используй Custom (Cron) с выражением 0 * * * *.
  • Воркфлоу не запускается, хотя расписание правильное → нода Schedule Trigger работает только когда воркфлоу активирован (тумблер Active справа сверху). В режиме Execute workflow срабатывает один раз сразу, расписание не применяется.
  • Сложное cron-выражение возвращает ошибку → n8n использует пятипольный cron без секунд (в отличие от Quartz/Spring с шестью полями). 0 0 9 * * 1-5 — это уже шесть полей, не сработает; правильно 0 9 * * 1-5.

⚙Как достать строку из DataTable n8n по фильтрустатья · ~2 мин▾

Как достать строку из DataTable n8n по фильтру

DataTable в n8n — это таблица, которую можно вести прямо внутри проекта рядом с воркфлоу. Когда из такой таблицы нужно вытащить одну конкретную строку, используется операция Get rows с фильтром по условию.

Зачем это нужно

DataTable удобна, когда нужно хранить конфигурацию или справочник, к которому воркфлоу обращается на лету: «возьми строку с таким-то ключом и подставь её поля в дальнейшие ноды». Чтобы не доставать всю таблицу и не фильтровать её потом руками, фильтр задаётся прямо в параметрах ноды.

Как это работает

  1. В параметрах ноды Data Table выбирается Resource: Row и Operation: Get.
  2. В поле Data Table указывается имя таблицы.
  3. В блоке Filter → Conditions добавляется одно или несколько условий: имя колонки + оператор + значение.
  4. Значение фильтра почти всегда вводится в режиме выражения ={{ ... }}, чтобы взять его из предыдущей ноды (например, поле из триггера). Литералом фильтр пишется только когда значение известно заранее и никогда не меняется.
К примеру
Фильтр по выражению vs литерал

К примеру, если в таблице лежит конфиг под разные каналы (telegram, web, email), а имя канала приходит из триггера — в условии пишется ={{ $('Trigger').item.json.channel }}. Если бы стоял литерал telegram, при подключении второго канала пришлось бы дублировать ноду.

Типичные ошибки

  • Литерал вместо выражения — фильтр работает только для одного значения, нет повторного использования.
  • Опечатка в имени колонки — нода возвращает пусто, а ошибка молчаливая (нет «такой колонки нет»); проверять имя колонки буква в букву.
  • Не включён Always Output Data в Settings — при пустом результате нода не отдаёт ничего вниз, и следующая нода (например, Code) падает на cannot read property of undefined. Если по бизнесу пустой результат — это валидное состояние, включают Always Output Data, чтобы получить хотя бы пустой объект.
  • Несколько условий по AND, ожидали OR — в n8n несколько условий в Conditions объединяются AND; для OR-фильтра нужен либо отдельный путь через IF, либо выкачать всё и отфильтровать в Code.

Когда не подходит

Если в одной таблице нужно склеить данные из нескольких строк (агрегация, выборка) — лучше или вытащить всё через Get many rows, или построить логику на ноде после. Get rows с фильтром даёт по сути одну строку и не подходит для сложных запросов.

📖Кнопка «Execute step»: запуск одной ноды без всего воркфлоустатья · ~2 мин▾

Запустить одну ноду, не дёргая весь workflow

Execute step — это кнопка внутри панели ноды (открывается двойным кликом по ноде). По смыслу — младший брат большой кнопки Execute workflow внизу canvas: та запускает весь воркфлоу с начала, эта — только текущую ноду. Удобно, когда настраиваешь параметры и хочешь сразу видеть, что отдаёт нода, не перезапуская всю цепочку.

Как это работает

Когда ты жмёшь Execute step, n8n берёт уже существующие данные на входе ноды (output предыдущей ноды из последнего прогона) и прогоняет через эту ноду. Сама нода выполняется реально: HTTP-запрос уходит, запись в базу пишется, письмо отправляется. Это не «сухой прогон», а полноценное выполнение — только одного шага.

  • key: Execute workflow

value: Запускает весь воркфлоу от триггера. Нужен полный прогон с самого начала. Каждый раз заново дёргает все API.

  • key: Execute step

value: Запускает только эту ноду. Берёт вход из последнего output предыдущей ноды. Идеально для подбора параметров и отладки.

  • key: Когда работает

value: Только если на входе ноды есть данные с прошлого прогона. Если ни разу не запускал предыдущие — кнопка либо ничего не вернёт, либо упадёт «No input data».

Типичный сценарий: запустил весь воркфлоу один раз (Execute workflow), увидел что нода HTTP Request отдаёт не то, поправил поле URL, нажал Execute step — увидел новый результат за секунду. Предыдущие ноды повторно не выполнялись.

У триггеров кнопка называется иначе
Смысл тот же — запустить один шаг и получить тестовое событие

У некоторых триггерных нод n8n кнопка Execute step подписана по-другому, потому что у триггера нет «входа» — он сам должен сходить за событием. Например, у RSS Trigger она называется Fetch Test Event: клик подтянет последние записи из ленты и положит их на выход ноды как тестовый input для остального воркфлоу. Работает идентично Execute step обычной ноды — точно так же используется для отладки и подбора параметров без полного прогона.

Важно про побочные эффекты
Каждый Execute step — это реальный вызов

Нода с записью в базу или отправкой сообщения каждое нажатие Execute step делает реально: 5 нажатий = 5 записей / 5 писем. Для отладки таких нод используй тестовые данные или подменяй credentials.

Типичные ошибки

  • Жать Execute step на ноде, у которой ещё не было входных данных → «No input data», нода не запустится. Сначала прогони весь воркфлоу через Execute workflow или хотя бы триггер вручную, чтобы у каждой ноды появился output.
  • Использовать Execute step для проверки изменений в предыдущих нодах → текущая нода работает со старым output, изменения не подхватятся. Если поправил предыдущую ноду — перезапусти её, а потом уже текущую.
  • Тестировать ноду с побочным эффектом (запись в БД, отправка письма) на проде → каждый клик создаёт реальную запись. Использовать тестовый credential или временно подменять параметры на безопасные.

Четвёртая нода — Loop Over Items. Она будет крутиться по заявкам: на каждой итерации отдаст один item в тело цикла, дождётся пока тело отработает (HTTP-запрос + обновление статуса), возьмёт следующий item и так далее, пока не пройдёт все 20.

→ Loop (нижний) — срабатывает на каждой итерации, отдаёт текущий item в тело. К нему на шаге 5 подключим HTTP Request.

📖Loop Over Items v3: что в ноде на самом деле есть и три типичных промахастатья · ~3 мин▾

Что в Loop Over Items v3 на самом деле есть

В ноде Loop Over Items версии 3 — ровно два параметра: Batch Size (сколько записей отдаём в одной итерации) и Options → Reset (сбросить состояние Loop). Всё остальное, что обычно ломает циклы — это поведение runtime, а не настройки ноды. Если ищешь в Loop «лимит итераций», «таймаут», «защита от бесконечного цикла», «дождаться все ветки» — их там нет. Это придётся либо собирать руками вокруг Loop, либо переносить в соседние ноды (HTTP, Wait, IF). Эта статья — про три места, где новички промахиваются.

Промах 1: Reset в Loop ≠ ограничение пагинации

Options → Reset сбрасывает внутреннее состояние Loop (счётчик обработанных items) при следующем запуске — он нужен в редких сценариях с manual-pagination, когда мы сами контролируем сколько раз дёрнуть Loop. Это не «ограничь количество страниц». Лимит страниц при пагинации n запросов живёт совсем в другом месте — в HTTP Request → Options → Pagination → Limit Pages Fetched. Если в Loop стоит Reset, а лимита нет — Loop крутится столько, сколько отдаст HTTP Request (потенциально вся вселенная).

Где искать лимит
Limit Pages Fetched лежит в HTTP Request, не в Loop

В HTTP Request → Options → Pagination → Limit Pages Fetched. В самом Loop Over Items v3 параметра-ограничителя нет.

Промах 2: вложенный Loop держит «done» между outer-итерациями

Native nested loop — это когда внутри loop body другого Loop стоит ещё один Loop. Известный antipattern, баг подтверждён на n8n 2.20.7 (issue #23670). Inner Loop отрабатывает только в первой итерации outer. Во второй итерации outer Inner Loop уже в состоянии done=true, Inner Work внутри него не вызывается, а downstream видит мусор: задвоенные processedItems от первой итерации плюс необработанные сырые items от второй.

Правильный путь — не вкладывать Loop в Loop руками. Если действительно нужна вложенная обработка («батч ингредиентов внутри каждого рецепта»), вынеси тело внутреннего цикла в отдельный workflow и вызывай его через Execute Workflow в режиме each — каждая вложенная итерация будет работать в чистом контексте.

Промах 3: .first() / .all()[N] внутри loop body

Внутри тела Loop частая ошибка — взять значение из ноды выше через $('NodeAbove').first().json.X или .all()[0].json.X. Эти выражения всегда возвращают первую запись из output ноды, независимо от того, в какой итерации Loop ты сейчас. Workflow завершится зелёным, ошибок не будет — но во всех результирующих записях значение поля будет от первого item. Это silent corruption. Чтобы попасть в «текущую» запись итерации — используй .item (он идёт по pairedItem chain) или прямо $json.X, если поле дошло до этой точки.

Подробности и сравнительная таблица выражений — в отдельной inline-статье n8n-pairedItem-and-item-resolve.

Бонус: порядок items на выходе

Items внутри Loop отдают в downstream в том порядке, в каком приходят на вход — но если в loop body стоит async-нода (Wait, медленный HTTP Request без Always Output Data), порядок финального processedItems может отличаться от исходного. Если порядок важен (например, агрегируешь хронологию), не закладывайся на «он же был отсортирован при входе» — сортируй явно после Loop через Sort или Code.

Защита от вечного цикла
Max iterations guard собирается руками

В Loop v3 нет встроенного предохранителя «не более N итераций». Если есть риск бесконечного цикла — заведи Counter (через Set) и IF на counter > N → Stop and Error внутри loop body. Это инженерная защита, n8n её не делает за тебя.

6Отправь POST на внешний CRM с Retry On Fail▾

Четвертая нода — HTTP Request. Подключаем её к нижнему (Loop) выходу Loop Over Pending, потому что вызов внешнего API — это и есть тело цикла. На каждой итерации Loop пробросит сюда одну заявку, нода отправит POST на учебный URL с её payload и вернётся в Loop с ответом.

Учебный URL — httpbin.org, публичный сервис, который умеет отдавать любой HTTP-код по запросу. Чтобы половина запросов реально падала пятисоткой, а половина проходила — выражением случайно выбираем код в URL: Math.random() > 0.5 ? '500' : '200'.

Ключевая техника задачи — две защитные настройки внутри ноды в разделе Settings:

  • Retry On Fail делает три попытки с паузой 2 секунды между ними (это сглаживает короткие сбои сети и кратковременные 5xx).
  • On Error: Continue (using error output) включает у ноды второй выход (error), куда уходит item если все три попытки провалились — без этой настройки нода после третьего фейла уронила бы весь воркфлоу.
⚙curl — инструмент для HTTP-запросов из командной строкистатья · ~2 мин▾

curl — инструмент для HTTP-запросов из командной строки

curl — это консольный инструмент, который отправляет HTTP-запросы. Он предустановлен на macOS и большинстве Linux-систем. С его помощью можно проверить, работает ли сервис, без написания кода и без открытия браузера.

Как это работает

Базовый GET-запрос выглядит так:

curl https://example.com/health

Если нужно отправить POST с JSON-телом:

curl -X POST https://example.com/search \
  -H "Content-Type: application/json" \
  -d '{"query": "пример"}'

Что означают флаги:

  • -X POST — метод запроса (по умолчанию GET)
  • -H "Content-Type: application/json" — заголовок, говорит серверу что данные в формате JSON
  • -d '...' — тело запроса

Зачем это нужно

Самый быстрый способ проверить, что сервис живёт и отвечает правильно — запустить curl в терминале. Это занимает пять секунд и не требует никакой настройки.

К примеру
Где встречается

К примеру, после деплоя FastAPI-сервиса первым делом проверяют health-эндпоинт через curl https://your-service.onrender.com/health. Если ответ {"status": "ok"} — сервис работает. Если ошибка — смотрят логи деплоя.

Типичные ошибки

  • Одинарные кавычки в Windows — в Windows командная строка не поддерживает одинарные кавычки в теле запроса. Нужно использовать двойные или PowerShell.
  • Нет Content-Type при отправке JSON — без этого заголовка сервер может не распознать формат тела и вернуть ошибку 422 или 400.
  • Адрес с http:// вместо https:// — большинство публичных сервисов требуют HTTPS. Запрос на HTTP-адрес либо не пройдёт, либо будет перенаправлен.

Для постоянной работы с HTTP API удобнее использовать GUI-инструменты (Postman, Insomnia) или клиентские библиотеки в коде. curl хорош для быстрой проверки и отладки.

Fixed и Expression — два режима любого поля

В правом верхнем углу почти каждого поля параметров ноды есть переключатель Fixed | Expression. Это два режима ввода значения: фиксированное (написал текст — этот текст и пойдёт) или формула на JavaScript, которая вычисляется заново при каждом выполнении ноды. Один и тот же тумблер есть у обычных текстовых полей, у dropdown-ов, у credential pickers и у resource locator полей (где выбираешь объект из списка — таблицу, документ, канал).

Как работает каждый режим

  • Fixed — обычный ввод. Что написал — то и попадёт в запрос/в подстановку. У resource locator (когда выбираешь объект — таблицу, документ, sheet) Fixed означает выбор из dropdown «From list» или ручной ввод id одним значением.
  • Expression — поле подсвечивается оранжевым, а внутри можно писать формулу в фигурных скобках: {{ $json.field }}, {{ $now.toISO() }} и т.п. Значение пересчитывается на каждой итерации, для каждого пришедшего элемента.
Когда переключать в Expression
Значение зависит от данных или меняется между запусками

Если значение должно зависеть от того, что пришло из предыдущей ноды (имя пользователя, id записи, текст сообщения), от текущего времени или от любой логики — переключай в Expression. Если значение всегда одно и то же (метод HTTP POST, имя таблицы, фиксированный slug) — оставляй Fixed.

Как переключить режим

Три способа сделать одно и то же:

  1. Клик по Fixed | Expression в правом верхнем углу поля. Самый явный — видно сразу, в каком режиме поле.
  2. Иконка шестерёнки слева от поля → пункт Add Expression. Удобно когда хочешь сразу открыть expression builder (окно с деревом нод слева и preview значения справа).
  3. Drag-and-drop значения из панели Input слева в поле. Это самый частый способ в учебных задачах: открываешь NDV предыдущей ноды через мини-карту слева или раскрываешь панель Input, цепляешь нужное поле мышкой и тянешь в параметр текущей ноды. n8n при отпускании сам переключает режим в Expression и подставляет правильную ссылку — {{ $json.field }} или {{ $('Имя ноды').item.json.field }}.

В обратную сторону — клик по Fixed в углу. Содержимое expression-режима очистится, поле станет обычным текстовым.

Тонкость
У resource locator в Expression нет dropdown

В resource locator полях (например, выбор Data table или Spreadsheet) при переключении в Expression список объектов из dropdown пропадает — теперь ты должен сам подставить нужный id или slug формулой. Полезно когда id берётся из предыдущей ноды; если нужно просто выбрать существующий объект мышкой — оставайся в Fixed.

Типичные ошибки

  • Забыл переключить в Expression, написал формулу как текст. Поле в Fixed-режиме воспринимает {{ $json.name }} как обычные символы и отправит эту строку буквально. Признак: значение в preview / в результате выполнения содержит {{ и }} как есть. Лечение: переключить в Expression вручную (тумблер в правом верхнем углу) или перетянуть значение из панели Input заново.
  • Стёр выбор из resource locator, переключив в Expression и обратно. Переключение режима очищает содержимое поля. Если в Fixed уже был выбран нужный объект из списка — после Expression → Fixed его придётся выбрать заново.

Что такое expression

В n8n expression — это маленький кусочек кода в фигурных скобках, который динамически подставляет значение в поле ноды. Записывается так:

={{ <выражение> }}

Знак равенства = в начале — это маркер для n8n: «трактуй содержимое этой ячейки не как обычную строку, а как выражение, которое нужно вычислить». Без = n8n запишет в поле литеральный текст {{ ... }} как есть.

Внутри {{ }} пишется JavaScript-выражение. Самое частое — чтение значения из другой ноды.

Зачем нужны expressions

Каждое поле в параметрах ноды может быть либо фиксированным значением (например, URL https://api.example.com), либо expression'ом — формулой, которая считается на лету в момент исполнения ноды. Expressions нужны, когда значение зависит от данных, пришедших из предыдущей ноды.

К примеру
Где это встречается

К примеру, ты сделал ноду «прочитать строку из таблицы конфига», а следующая нода — HTTP Request к внешнему API. В URL этой HTTP-ноды можно подставить значение из конфига: ={{ $('Get Config').item.json.api_url }} — n8n при каждом исполнении возьмёт актуальный URL из таблицы, а не зашитый в ноду.

Как читать значения из других нод

Базовая формула: ={{ $('<имя_ноды>').item.json.<поле> }}

Разбор по частям:

  • $('<имя_ноды>') — обращение к ноде по имени (то, что написано на canvas под нодой). Имя — в одинарных кавычках, точно как на canvas.
  • .item — текущий элемент, который обрабатывается. У большинства нод элементов несколько (массив), item берёт тот, что соответствует текущей итерации.
  • .json — содержимое поля json элемента (там лежат все данные, которые нода вернула).
  • .<поле> — конкретное поле внутри json. Если поле вложенное — пишется через точку: .config.embedding_model.

Пошагово — как вставить expression в поле

  1. Кликни на иконку шестерёнки рядом с полем, выбери Add Expression (или нажми Expression в верхнем правом углу поля).
  1. В появившемся окне (expression builder) — слева список нод воркфлоу, можно кликать по полям и они вставятся в формулу.
  1. Справа — preview: видно, какое значение получится прямо сейчас. Если красным — формула или путь к полю некорректные.
  2. Закрой окно, поле остаётся в режиме expression (синий фон + иконка fx).
Типичная ошибка
`=` в начале — обязателен

Если скопировал готовое выражение из учебной задачи и вставил без = в начале — n8n запишет в поле текст {{ ... }} как есть, нода уйдёт делать HTTP-запрос на URL вроде ={{ $('...').item.json.api_url }} буквально. Симптом — нода падает с ошибкой Invalid URL или возвращает 404 на странную строку.

Типичные ошибки

  • Имя ноды не совпадает. В формуле $('Get Config') имя должно точно совпадать с подписью ноды на canvas. Если переименовал ноду — старые expressions, ссылающиеся на неё, сломаются.
  • Путь к полю не существует. .item.json.embedding_dimensions упадёт с undefined, если у элемента такого поля нет. Препетий — посмотри preview в expression builder, он сразу показывает значение или ошибку.
  • Забыл = в начале. Самая частая опечатка — описано в callout выше.
  • Кавычки внутри {{ }}. Внутри expression строки оборачиваются одинарными кавычками: ={{ 'Hello ' + $json.name }}. Двойные конфликтуют с JSON.

Когда expression не нужен

Иногда поле должно быть фиксированным. Например, метод HTTP-запроса (GET / POST) или имя коллекции, которое одинаково для всего воркфлоу — нет смысла читать его динамически. Простое правило: если значение меняется от итерации к итерации или зависит от пришедших данных — нужен expression; если значение постоянное — оставляй фиксированным.

Передай payload текущей заявки через drag-and-drop

Drag-and-drop из панели Input — самый быстрый способ связать ноды

В n8n каждая нода видит данные предыдущих нод в панели Input слева от параметров. Чтобы подставить любое поле из этого Input в параметр текущей ноды, не нужно вручную писать expression — достаточно перетащить поле мышкой. n8n сам подставит правильное выражение со ссылкой на нужную ноду.

Как это работает

  1. Открой параметры ноды двойным кликом. Слева автоматически появится панель Input — это выходные данные предыдущей ноды (или одной из предыдущих, если их несколько).
  2. Разверни в Input структуру до нужного поля — она показывается как дерево вложенных объектов: message → chat → id, output → [0] → content → [0] → text и так далее.
  3. Перетащи нужное поле мышкой прямо в поле параметра справа. В поле появится выражение вида ={{ $('<имя предыдущей ноды>').item.json.<путь.к.полю> }}.
  4. Поле в правой панели окрасится синим — это значит, что параметр теперь не литерал, а expression: на каждом запуске значение будет браться из реальных данных.
К примеру
Откуда n8n берёт имя ноды

К примеру, если предыдущая нода называется Telegram Trigger, и ты перетащил из неё поле message.text — n8n запишет ={{ $('Telegram Trigger').item.json.message.text }}. Имя ноды берётся ровно такое, как написано на canvas. Иногда — если переименуешь ноду уже после drag-and-drop, выражение сломается, потому что в формуле останется старое имя.

Откуда выбрать источник данных

Если у ноды несколько входов или нужно дотянуться не до прямого предыдущего шага, а до более раннего — сверху панели Input есть переключатель источника. К примеру, в финальной ноде workflow можно вывести Input от первого триггера, чтобы достать оттуда поле, которого нет в выходе ближайшей предыдущей ноды.

Типичные ошибки

  • Перетащил не на то поле параметра. Drag-and-drop срабатывает на любом текстовом поле — легко уронить значение не в Text, а в Chat ID, и наоборот. После перетаскивания проверь, что в нужном поле появилось ожидаемое выражение.
  • Перетащил из объекта-обёртки, а не из конкретного поля. Если потянуть за message, а не за message.text — n8n подставит весь объект, в текстовое поле уйдёт [object Object] или JSON-строка. Раскрывай дерево до примитивного значения.
  • Сломался путь после переименования ноды. В выражении $('Telegram Trigger') имя ноды фиксированное. Если потом переименовать Telegram Trigger в что-то другое — нужно либо вернуть старое имя, либо переписать выражение вручную.

Когда лучше не использовать drag-and-drop

Если значение должно быть фиксированным (например, имя коллекции, метод HTTP, статичный текст приветствия) — drag-and-drop не нужен, просто впечатай значение в поле как обычный текст. Drag-and-drop — это про динамические параметры, которые меняются от запуска к запуску в зависимости от входных данных. </content> </invoke>

Нода теперь шлёт POST, но в продакшене без защитных переключателей этого мало. Если внешний CRM ответит 500 хотя бы один раз — нода упадёт, воркфлоу прервётся посреди обхода Loop, и оставшиеся pending-заявки в этом прогоне даже не доедут до синхронизации. Решение — включить два переключателя в самой ноде в разделе Settings.

Что делает typeValidation в IF-ноде

Нода IF в n8n проверяет условие и направляет item в одну из двух веток (true / false). В параметрах есть тонкая, но важная настройка — Options → Type Validation с двумя значениями:

  • strict — значение слева и справа должны быть одного типа, иначе условие проваливается с ошибкой.
  • loose — n8n пытается привести типы перед сравнением (например, строка "5" сравнивается с числом 5 как равные).

Дефолт у новых IF-нод сейчас — strict. Это часто становится сюрпризом для тех, кто привык что «5 равно "5"».

Когда strict — правильно

К примеру, ты получил из БД число 5 и сравниваешь его с фиксированным числом 5 — оба типа number, strict отлично работает. Это самый прозрачный режим: что видишь — то и сравниваешь.

Strict хорош когда:

  • Ты контролируешь типы на входе (БД с типизированными колонками, валидированные API-ответы).
  • Хочешь видеть ошибку «типы не совпали», если что-то пошло не так с upstream-нодой.

Когда нужен loose

Часто данные приходят как строки, потому что таков транспорт:

  • Webhook принимает JSON, в котором числа пришли как строки: { "amount": "500" }.
  • HTTP-параметры всегда строки: ?count=10.
  • Telegram API возвращает chat.id как число, но в expression-выражениях оно может оказаться строкой после JSON.parse через цепочку нод.

Если ты сравниваешь $json.amount === 500 в strict-режиме — условие проваливается, потому что "500" !== 500 (строка vs число). В loose-режиме n8n приведёт строку к числу и условие сработает корректно.

Антипаттерн: AND-условие с разными типами

Самая частая ловушка — IF-нода с двумя условиями через AND, где одно сравнение про числа, второе про строки. К примеру:

amount > 100  AND  status === "active"

Если в strict-режиме amount пришёл строкой "500", всё условие падает с ошибкой типа — даже без перехода ко второму сравнению. И в логе непонятно, какое именно условие сломалось.

В loose-режиме n8n тихо приведёт "500" к числу, и условие сработает.

Как выбрать режим

Чтобы посмотреть/поменять режим — открой IF-ноду (двойной клик), в параметрах раскрой раздел Options (он по умолчанию свёрнут, нужно нажать Add Option), там увидишь Type Validation с выпадающим списком strict/loose.

Внимание
Если IF падает с ошибкой типа

Если в логе выполнения у IF-ноды ошибка вида «expected number, got string» — это strict-режим. Не торопись чинить upstream-ноду: открой IF, переключи Type Validation в loose, проверь что условие срабатывает корректно. Часто это самый быстрый фикс.

📖$json или $('Имя ноды'): когда какая форма обращения к даннымстатья · ~2 мин▾

Две формы обращения — разница только в источнике

В n8n у одной и той же ноды есть две формы прочитать поле из данных воркфлоу:

{{ $json.business_idea }}
{{ $('Form Trigger').item.json.business_idea }}

Обе достают одно и то же значение, но смотрят в разные места:

  • $json — это выход ноды, которая стоит прямо перед текущей. Короткая форма для «возьми поле из предыдущей ноды».
  • $('Имя ноды') — это обращение к любой ноде воркфлоу по имени, в том числе к той, что стоит на 2-3-5 шагов раньше.

Когда писать какую

Правило простое:

  • Хочешь взять данные у предыдущей ноды (стрелка идёт прямо в текущую) — пиши $json.поле.
  • Хочешь дотянуться до ноды глубже по цепочке (между ней и текущей есть промежуточные ноды) — пиши $('Имя ноды').item.json.поле.
Пример
Цепочка из трёх нод

Цепочка: Form Trigger → OpenAI → Send Message.

В Send Message обращения к двум разным источникам выглядят так:

  • к ответу OpenAI (предыдущая нода): ={{ $json.output }}
  • к полю формы (нода на два шага назад): ={{ $('Form Trigger').item.json['Email'] }}

Если в Send Message написать ={{ $json['Email'] }} — поле не найдётся, потому что у OpenAI в выходе нет Email. Это типичная ошибка студентов.

Почему короткая форма вообще существует

$json — это просто синоним для $input.item.json, то есть «текущий обрабатываемый элемент входа». А вход текущей ноды формируется из выхода предыдущей. Поэтому $json.поле и $('Предыдущая нода').item.json.поле всегда дают одно и то же значение — пока стрелка идёт напрямую.

$('Имя ноды') нужна там, где короткая форма не справится: между источником и текущей нодой есть промежуточные шаги, которые этого поля уже не передают (например, OpenAI вернул только текст ответа, и поле формы Email к моменту Send Message живёт только в выходе Form Trigger).

Типичные ошибки

  • Написал $json.Email в ноде после OpenAI. OpenAI возвращает output, а не Email. Решение — обратиться по имени: $('Form Trigger').item.json['Email'].
  • Имя ноды в $('...') не совпадает с подписью на canvas. В expression имя — буквально то, что написано под нодой на canvas. Переименовал ноду — все старые expressions, ссылавшиеся на её прежнее имя, сломаются.
  • Забыл .item после имени ноды. Полная форма — $('Имя').item.json.поле. Без .item обращаешься не к данным, а к самой ноде как объекту — поле не найдётся.

⚙Execute Once в n8n: нода выполняется ровно один разстатья · ~2 мин▾

Execute Once в n8n: нода выполняется ровно один раз

Контекст — про то, почему ноду нужно ограничить одним прогоном — см. [[n8n-multi-item-processing]]. Здесь — как именно работает переключатель Execute Once.

Где найти

Параметр Execute Once находится во вкладке Settings параметров ноды — не в основной вкладке Parameters. Самая частая ошибка — искать его рядом с обычными полями и не находить.

Как работает

Когда параметр включён:

  • n8n подаёт ноде только первый item из входного массива.
  • Остальные items для этой ноды игнорируются.
  • Выходные данные ноды содержат один элемент.

Параметр не влияет на последующие ноды — они получают выход ноды с Execute Once как обычный поток.

Когда применять
Ноды-отправители + конфиг-таблица

Типичный кейс: нода Telegram / Email / HTTP Request стоит после ноды с несколькими items (Data Table вернул 3 строки конфига, или Filter оставил 5 заявок). Без Execute Once отправитель прогонится по каждому item и пошлёт 3-5 одинаковых сообщений. Execute Once → одно сообщение.

Типичные ошибки

  • Не та вкладка — Execute Once не в Parameters, а в Settings. Если не нашёл — открой вкладку справа от Parameters.
  • Перепутали с фильтрацией — Execute Once не выбирает «нужный» item из нескольких, а берёт первый и игнорирует остальные. Если нужен конкретный — фильтруй заранее (через Filter / Data Table Match) и потом ставь Execute Once.
  • Поставили на ноду-источник, а не на отправителя — если включить Execute Once на ноде Get Config, она вернёт только первую строку, и следующая нода не сможет найти нужный ключ через .all().find(). Execute Once ставится на ту ноду, которая должна сработать один раз — обычно это нода-отправитель в конце цепочки.

Когда не подходит

Execute Once не нужен, когда нода действительно должна обработать каждый входящий item — например, обогащение данных, когда для каждой строки делается отдельный запрос и нужен результат на каждую. Здесь N прогонов — это желаемое поведение, не баг.

Если нужно объединить N items в один без потери данных (а Execute Once «теряет» все кроме первого) — используй Code-ноду в режиме Run Once for All Items или ноду Aggregate.

Универсальный процесс добавления credential

Credential в n8n — это сохранённая «учётка» для подключения к сервису (API-ключ, OAuth-токен, логин/пароль и т.д.). Любая нода, которой нужна авторизация, выбирает credential из общего списка — поэтому добавлять его нужно один раз, а использовать можно из любого количества нод того же типа.

Пошагово

  1. В левом меню n8n открой раздел Credentials.
  1. Нажми кнопку Add credential (правый верхний угол).
  1. В поиске введи название нужного типа credential — например, OpenAI, Header Auth, Postgres, Google — и выбери его из списка.
  1. В шапке формы дай credential понятное имя — например, OpenAI школа или Postgres prod. По этому имени ты будешь выбирать его в нодах.
  1. Заполни поля формы. Какие именно — зависит от типа: для API-ключевых это обычно одно поле API Key, для Header Auth — Name и Value заголовка, для Postgres — host/port/database/user/password, для OAuth — запуск OAuth-flow по кнопке.
  1. Нажми Save. Готово — credential можно подключать к любой ноде, которая поддерживает этот тип.
Подсказка
Один credential — много нод

Один сохранённый credential можно использовать в неограниченном количестве нод одного типа. Если ключ протухнет — поменяй его в credential один раз, обновится во всех нодах сразу.

Как узнать свой Telegram chat ID

chat_id в Telegram — это уникальный числовой идентификатор личного чата, группы или канала. Боту он нужен, чтобы понимать, кому отправлять сообщение: API принимает не username (@vasya), а именно число (например, 123456789). Свой собственный chat_id совпадает с твоим user_id — это число, которое Telegram присвоил тебе при регистрации, и оно не меняется.

Способ 1 — @userinfobot

Самый быстрый.

  1. Открой Telegram и найди в поиске @userinfobot.
  2. Нажми Start (или отправь любое сообщение).
  3. Бот пришлёт ответ с полем Id — это и есть твой chat_id.

Способ 2 — через Telegram Web

Если бот недоступен или не хочется ставить лишних ботов.

  1. Открой web.telegram.org и залогинься.
  2. Зайди в «Избранное» (Saved Messages) — это твой личный чат сам с собой.
  3. Посмотри на адресную строку: после # будет число, например https://web.telegram.org/a/#123456789. Это твой chat_id.
Когда это пригодится
Когда боту нужно слать сообщение лично тебе

Любая задача, где Telegram-нода в n8n должна отправить сообщение конкретному человеку (тебе как админу, тестовому пользователю), а не отвечать на входящий update. В поле Chat ID ноды подставляется именно это число.

Типичные ошибки

  • Указать @username вместо числа → Telegram Bot API вернёт ошибку Bad Request: chat not found. API понимает только числовой ID.
  • Использовать чужой chat_id (например, скопированный из примера) → сообщения уйдут другому человеку или, если ID не существует, бот получит chat not found.

⚙Append n8n Attribution: подпись в конце автоматических сообщенийстатья · ~2 мин▾

Append n8n Attribution: подпись в конце автоматических сообщений

Параметр «Append n8n Attribution» управляет автоматической подписью, которую n8n добавляет в конец каждого отправленного сообщения. По умолчанию он включён — и к тексту любого сообщения дописывается фраза вида «This message was sent automatically with n8n».

Зачем это нужно знать

Для личных экспериментов и отладки подпись не мешает. Но когда автоматическое сообщение идёт реальным пользователям или сотрудникам — постороннее упоминание сервиса автоматизации выглядит непрофессионально. Получатель видит не уведомление от сервиса, а артефакт настройки системы.

Как это работает

Параметр находится в разделе Additional Fields ноды отправки сообщений — он скрыт по умолчанию, чтобы не засорять основной экран настроек. Чтобы его найти:

  1. Открой параметры ноды.
  2. Прокрути вниз до раздела «Additional Fields» и раскрой его.
  3. Найди параметр «Append n8n Attribution» — переключи в положение Off.

После отключения сообщения отправляются без каких-либо лишних дополнений к тексту.

К примеру
Где это встречается

К примеру, нода Telegram отправляет оператору-человеку уведомление о новом запросе от клиента. Если Attribution включён, оператор видит в конце: «This message was sent automatically with n8n» — это избыточно для внутреннего рабочего инструмента. Иногда Attribution заметен и в Email-нодах, когда письмо с автоматической подпиской уходит клиентам — это вредит доверию к бренду.

Типичные ошибки

  • Не нашли параметр — он спрятан в Additional Fields и не видён на основном экране. Нужно явно раскрыть блок.
  • Оставили включённым для пользовательских сообщений — получатели видят рекламу сервиса вместо нейтрального уведомления.
  • Перепутали с Parse Mode — Attribution и Parse Mode находятся в одном разделе Additional Fields, но это разные настройки. Parse Mode управляет форматированием (Markdown/HTML), Attribution — подписью.

Когда не подходит отключать

Если вы намеренно хотите показать, что сообщение отправлено автоматически — Attribution можно оставить включённым. Это может быть уместно в тестовых рассылках внутри команды, где прозрачность источника важна.

📖Кнопка «Execute workflow»: ручной запуск для тестированиястатья · ~2 мин▾

Кнопка ручного запуска для теста

В редакторе workflow внизу canvas есть большая розовая кнопка «Execute workflow». Это основной инструмент тестирования: один клик — и весь workflow прогоняется от триггера до конца, прямо в редакторе. На production это никак не влияет — даже если workflow активен (тумблер сверху), кнопка запускает отдельный тестовый прогон.

Как это работает

  1. Нажимаешь Execute workflow внизу canvas.
  1. n8n запускает workflow от триггера. Ноды по очереди подсвечиваются: серая → жёлтая (выполняется) → зелёная (готово) или красная (ошибка). Внизу каждой ноды появляется счётчик пришедших на вход элементов.
  1. Клик по любой ноде после прогона — видишь её input и output. Это и есть отладка: где данные пропали или пришли не в том виде.
Запуск с середины
Кнопка ▶ на самой ноде

У каждой ноды на canvas есть своя маленькая кнопка ▶ (появляется при наведении). Она запускает только эту ноду на данных, которые лежат в её input от предыдущего прогона. Удобно, когда правишь одну ноду и не хочешь гонять весь workflow заново.

Особый случай: workflow с Chat Trigger

Если в workflow стоит триггер Chat Trigger (встроенный в n8n чат), правила для тестирования другие — Execute workflow появляется не сразу.

  • Когда у Chat Trigger ещё нет запиненных данных — внизу canvas вместо Execute workflow стоит кнопка Open chat. Клик открывает встроенный чат-виджет n8n: ты пишешь сообщение, оно идёт в триггер, workflow выполняется от события «новое сообщение в чате». Это и есть основной путь тестирования Chat Trigger.
  • Когда данные Chat Trigger запинены (через Pin data в Output триггера) — кнопка Open chat сменяется на привычную Execute workflow. n8n больше не ждёт нового сообщения в чате, а гоняет workflow на запиненных данных. Удобно для отладки downstream-нод без необходимости каждый раз писать в чат.

Логика та же что у обычных триггеров — если есть закреплённый тестовый вход, n8n показывает Execute workflow для запуска на нём; если нет — показывает способ получить реальное событие (для Chat Trigger это Open chat, для Webhook — Test URL и т.д.).

Типичные ошибки

  • Нажал кнопку, ничего не происходит → в workflow нет триггера или есть только Schedule/Webhook без Manual Trigger. Добавь временный Manual Trigger или используй кнопку ▶ на конкретной ноде.
  • Нода красная, дальше ничего не выполнилось → клик по ноде → вкладка с ошибкой. Прогон останавливается на первой упавшей ноде.
  • На ноде «0 items» и она не выполнилась → предыдущая нода вернула пустой массив. Это не ошибка, это «нечего обрабатывать». Смотри output предыдущей ноды.

🚀Stretch▾

Сейчас Retry On Fail повторяет вызов через ровно Wait Between Tries миллисекунд — у тебя это 2 секунды между попытками. Если внешний CRM упал и одновременно встают сразу несколько pending-заявок твоего воркфлоу — все они ретраятся в один и тот же момент через те же 2 секунды, потом снова через 2 секунды, и снова. На API такая синхронная волна повторов выглядит как «уже лежащий сервис атакует группа клиентов» — паттерн thundering herd. Сервису становится только хуже в момент когда он начинал восстанавливаться.

Добавь к интервалу между повторами jitter — случайный разброс в пределах ±20% от базового значения. Подумай где именно его удобно реализовать в этом воркфлоу: внутри настроек Retry On Fail у ноды HTTP Request (если n8n поддерживает) или через отдельную ноду Wait перед повторным вызовом. Финальный эффект — два запроса, выпавшие в одну секунду, ретраятся не одновременно, а в разные моменты в пределах своего временного окна.

Создал таблицу m-errorhandling-03-sync-queue, залил 20 pending-строк через Import, запусти главный воркфлоу через Execute workflow.Главный проходит ~30-90 секунд (зависит от рандома 200/500). На каждом 500 нода Sync To External CRM делает 3 попытки с паузой 2 секунды, потом item уходит по error-выходу в Mark Failed. На каждом 200 — item идёт в Mark Done через If Success. После цикла в Telegram приходит короткое сообщение «Синхронизация завершена. Обработка пакета pending-заявок закончена.». В таблице примерно половина строк со status=done, половина со status=failed (соотношение колеблется ±20% из-за рандома).
Открой Execution View прогона главного воркфлоу и нажми на ноду Sync To External CRM.У строк где статус ушёл в failed — видно реальную работу Retry: пометка «Attempt 1 / Attempt 2 / Attempt 3» в Execution Details, между попытками паузы примерно по 2 секунды. После третьей попытки item уходит по error-выходу (это видно по второму выходному порту ноды).
Запусти sub-workflow DLQ-повтор через Execute workflow после прогона главного.DLQ забирает только failed-строки (не трогает done). Каждая failed-заявка получает один новый POST на httpbin со случайным кодом. Те что 200 — становятся done (attempts +1, last_error пустой). Те что снова 500 — в Telegram приходит сообщение «Заявка req-XXX мертва: даже после DLQ-повтора...».
Опубликуй оба воркфлоу (нажми Publish) и подожди — Schedule Trigger должен сам запустить главный через час и DLQ через сутки в 3am.Главный сам запускается каждый час и подбирает свежие pending. DLQ сам запускается раз в сутки в 3am и подбирает накопившиеся failed.
