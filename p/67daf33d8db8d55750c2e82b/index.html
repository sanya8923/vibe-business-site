Что собираем

Соберём один основной воркфлоу с двумя независимыми триггерами на одной канве и отдельный маленький воркфлоу-тестер — чтобы было чем отправлять POST на свой вебхук.

Основной воркфлоу — M1-05 Application Hub. На одной канве лежат две независимые цепочки нод:

  • Цепочка А (триггер Webhook): принимает POST с формы → нода If отделяет реальных клиентов от мусора → если заявка валидна, пишет строку в Data Table и отвечает OK; если нет — отвечает Rejected.
  • Цепочка Б (триггер Schedule Trigger): раз в сутки в 10:00 забирает все строки из той же Data Table → нода Filter оставляет только необработанные → нода Telegram шлёт сводку с числом заявок.

Эти две цепочки физически не соединены стрелками. Каждая запускается своим триггером и работает независимо, но обе обращаются к одной таблице. Это нормальный паттерн n8n: на одной канве может жить сколько угодно триггеров.

Тестовый воркфлоу — M1-05 Application Tester. Две ноды: Manual Trigger (запуск по кнопке) и HTTP Request (шлёт POST на Test URL твоего вебхука с тестовым JSON). Меняя тело между запусками, прогоняешь все сценарии приёма.

Что такое DataTable

DataTable — встроенное хранилище n8n, что-то среднее между Excel-листом и базой данных. Лежит внутри n8n, не требует отдельного сервера и подключений, редактируется прямо в UI. Каждая таблица принадлежит проекту: видна и доступна всем воркфлоу этого проекта.

В одной таблице — фиксированная схема колонок (имя + тип) и произвольное количество строк. Доступ из воркфлоу — через ноду Data table (resource = Row, операции Get rows, Insert row, Update row, Delete row).

Когда выбирать DataTable

DataTable подходит для случаев, где данных немного и нужно удобство редактирования через UI:

  • Конфиги воркфлоу — настройки агента, ключи API-описаний, лимиты. Меняешь значение в ячейке, воркфлоу при следующем запуске берёт новое.
  • Небольшие справочники — список разрешённых пользователей, маппинг каналов на промпты, словарь ответов.
  • Состояние между запусками — счётчики, последний обработанный ID, кеш токена с TTL.
Когда DataTable, когда Postgres
Главное отличие — назначение

DataTable — для редко меняющихся данных, которые удобно править руками: конфиги, справочники, флаги. Postgres — для журналов, истории, аналитики: туда воркфлоу пишет много строк, оттуда читает выборки с условиями и сортировкой.

Условный порог: если ожидается больше 1000 строк или нужна аналитика с агрегацией — это Postgres. Если до 100 строк и они меняются вручную — это DataTable.

Тонкость
Параллельные записи не идемпотентны

Если два воркфлоу одновременно дописывают строку в одну и ту же DataTable — порядок может смешаться, а часть операций — провалиться. Для счётчиков и журналов, которые пишут часто из разных триггеров, выбирай Postgres.

Частый паттерн: конфиг продукта в DataTable

Параметры AI-агента (системный промпт, имя модели, имя коллекции Qdrant, имя таблицы логов) можно зашить прямо в ноды — но тогда любая правка требует открыть редактор, найти ноду, поменять, опубликовать новую версию. Альтернатива — вынести всё в одну строку DataTable.

Схема таблицы (например, Agent Config) — две колонки:

  • key (string) — идентификатор конфига: main, support, telegram.
  • config (string) — JSON со всеми параметрами.

Внутри config — обычный JSON с понятной структурой:

{
  "agent": {
    "system_prompt": "Ты помощник интернет-магазина. Отвечай вежливо и по делу.",
    "model": "gpt-4.1-mini",
    "temperature": 0.3
  },
  "qdrant": {
    "collection": "<имя_коллекции>",
    "embedding_model": "text-embedding-3-small"
  },
  "log_table": "agent_logs"
}

Как воркфлоу читает этот конфиг

  1. Триггер запускает workflow (новое сообщение от пользователя).
  2. Нода Get rows читает строку из Agent Config по ключу (например, key = main). Получает config как строку.
  3. Нода Set (или Code) парсит JSON и раскладывает поля по плоским переменным: system_prompt, model, collection, log_table.
  4. AI Agent и его sub-ноды (Qdrant KB, Postgres Chat Memory) ссылаются на эти переменные через выражения вида {{$json.system_prompt}}. При каждом запуске берётся актуальное значение.

Что это даёт

  • Правка без редактирования воркфлоу. Сменил системный промпт — открыл DataTable, поменял ячейку, сохранил. Воркфлоу подхватит новое значение при следующем запуске.
  • Один воркфлоу — много конфигураций. Тот же workflow обслуживает несколько контекстов (разные продукты, разные каналы, разные клиенты) — каждый со своей строкой.
  • История правок в одном месте. Легче откатить ошибочное изменение.
Двухуровневый конфиг
Product config + Channel config

В сложных продуктах конфиг разделяют на два уровня: product config — общее для всего продукта (база знаний, embedding-модель, имя таблицы логов), и channel config — что меняется по каналам (промпт, модель, лимит длины ответа). Workflow тогда делает два последовательных Get rows и сливает результаты.

Как отредактировать строку DataTable через UI n8n

DataTable в n8n редактируется двумя способами: через ноду Data Table в воркфлоу или прямо через UI — открыть таблицу, кликнуть в ячейку, изменить значение, сохранить. Для разовых правок конфига UI быстрее.

Зачем это нужно

DataTable часто используется как простой конфиг-стор: одна строка — это конфиг для одного канала / окружения / клиента. Когда нужно поменять одно поле (например, обновить системный промпт агента или сменить модель), собирать отдельный воркфлоу ради этого избыточно.

Как это работает

  1. В n8n открыть раздел Data Tables в нужном проекте (он рядом с разделом Workflows).
  2. Кликнуть по имени нужной таблицы — откроется представление со списком строк и колонок.
  3. Найти строку, которую нужно поправить (визуальный фильтр или поиск, если их много).
  4. Кликнуть в ячейку — она перейдёт в режим редактирования. Если значение — длинная строка или JSON, n8n откроет редактор побольше.
  5. Изменить значение и нажать Save (или Enter в зависимости от типа ячейки).
К примеру
UI vs нода в воркфлоу

К примеру, разово сменить модель gpt-4.1-mini на gpt-4.1 для канала telegram — проще через UI. Если же нужно регулярно или массово обновлять строки (например, синхронизировать конфиги при каждом релизе), удобнее делать это через ноду Data Table в техническом workflow.

Типичные ошибки

  • Сломали JSON в ячейке — DataTable не валидирует структуру внутри текстового значения. Если в ячейке хранится JSON-строка, после правки её нужно перечитать читалкой («это валидный JSON?»). Лучше править в редакторе ячейки (он подсветит структуру), не в краткой превью.
  • Не сохранили — кликнули в другую ячейку до сохранения, изменение потерялось. Привычка — сразу нажимать Enter / Save.
  • Несколько строк на один ключ — UI позволяет добавить дубликаты; если строки фильтруются по ключу, ноды могут читать любую из них. Перед правкой проверить, что строка одна.

Когда не подходит

Для регулярных или массовых операций UI неудобен — там собирают технический workflow с нодой DataTable в режиме Update / Upsert. UI хорош для разовых правок одной-двух ячеек.

Webhook в n8n: как нода ждёт входящий запрос

Webhook — это URL-адрес, по которому внешний сервис или браузер может запустить воркфлоу. Нода Webhook создаёт такой адрес автоматически — тебе остаётся только настроить метод и способ ответа.

Два URL: Test и Production

Нода Webhook всегда выдаёт два разных адреса:

  • Test URL — работает только когда ты нажал Execute workflow (нода ждёт один входящий запрос, потом останавливается). Используется при сборке и отладке.
  • Production URL — работает постоянно, пока воркфлоу опубликован и активен. Используется в боевом режиме.

Режим ответа: Using 'Respond to Webhook' Node

По умолчанию Webhook отвечает на запрос сразу, как только принял его. Когда выбираешь Using 'Respond to Webhook' Node — ответ откладывается до момента, когда нода Respond to Webhook явно его отправит. Это нужно, чтобы сформировать кастомный JSON-ответ в конце цепочки.

Pin data — зачем фиксировать данные

После первого запроса к Test URL нода получает входящие данные (headers, query, body). Кнопка Pin data фиксирует эти данные: все последующие запуски шагов будут использовать именно их, без повторного открытия браузера. Это ускоряет отладку — не нужно каждый раз отправлять новый запрос.

Добавление ноды на canvas

Нода — это шаг воркфлоу: триггер, действие или преобразование данных. Добавить ноду можно двумя способами: в пустой canvas или после существующей ноды. В обоих случаях открывается панель выбора нод с поиском.

Пошагово

  1. Открой воркфлоу и нажми + на canvas. Если воркфлоу пустой — большая кнопка в центре «Add first step». Если уже есть ноды — маленький + на стрелке после нужной ноды или плюс на самой ноде справа.
  1. В открывшейся панели справа набери название ноды или сервиса в поле поиска. Например, http, slack, openai. Список фильтруется по мере ввода.
  1. Кликни по нужной ноде в списке — она появится на canvas и откроется её панель параметров. Заполни поля и закрой панель. Готово.

📖Edit Output: как поменять запинённые данные на летустатья · ~1 мин▾

Edit Output: как поменять запинённые данные на лету

Когда у ноды зафиксированы данные через Pin Data, в панели Output рядом с иконкой булавки появляется ещё одна кнопка — Edit Output (иконка карандаша). Она открывает запинённый JSON прямо в браузере на редактирование: можно поменять значения полей, не отправляя новый запрос во внешний сервис.

Зачем это нужно

Главный сценарий — тестировать одну и ту же цепочку нод на нескольких разных входах. Запинил один пример → отредактировал ему текст → запустил воркфлоу → отредактировал на другой текст → снова запустил. Особенно полезно при отладке роутеров (Switch, IF, Text Classifier) — каждая ветка требует своего набора входных данных, и звать внешний источник под каждый кейс долго.

Как редактировать

  1. У ноды с запинёнными данными в панели Output найди кнопку Edit Output (карандаш рядом с булавкой).
  2. JSON в панели станет редактируемым полем — впиши новые значения или поменяй существующие.
  3. Нажми Save (или клик за пределами поля). Новые данные останутся запинёнными до следующего редактирования или снятия пина.

Что важно

  • Структура JSON должна остаться прежней. Если случайно удалить нужное поле или сломать вложенность, следующие ноды получат не те данные и упадут с ошибкой cannot read properties of undefined. Меняй значения, а не ключи.
  • Меняй только то, что должно поменяться. Для тестирования ветки роутера обычно достаточно подменить одно поле (например, message.text) — всё остальное оставить.
  • Edit Output не вызывает реальный источник. Это локальное редактирование пина в браузере, никакого запроса в Telegram / API / другую систему n8n не делает.
  • Чтобы вернуться к настоящим данным от триггера — сними пин и поймай новый event через Execute workflow.

🔀Нода IF в n8n: ветвление по условиюстатья · ~2 мин▾

Нода IF делит поток на две ветки

IF смотрит на каждый элемент входных данных, проверяет условие и отправляет элемент в один из двух выходов: true (условие выполнено) или false (не выполнено). Дальше каждая ветка идёт своим путём — в её ноды попадают только те элементы, которые подошли по условию.

Как настроить условие

Одно условие — это три части:

::key-value{rows=[{key:"Left value",value:"что сравниваем — обычно поле из предыдущей ноды через выражение, например {{ $json.status }}"},{key:"Operator",value:"как сравниваем — equals, not equals, contains, is empty, greater than, exists и другие"},{key:"Right value",value:"с чем сравниваем — фиксированное значение или второе выражение"}] id=if-condition-anatomy}

Операторы сгруппированы по типу данных левой части: String (текст), Number (число), Boolean (да/нет), Date & Time, Array, Object. Сначала выбери тип — в выпадающем списке останутся только операторы, которые имеют смысл для этого типа.

Несколько условий сразу

Под полем условия есть кнопка Add condition — добавляет ещё одну строку. Над списком появится переключатель AND / OR:

  • AND — элемент идёт в true только если все условия выполнены.
  • OR — достаточно одного выполненного условия.

Смешивать AND и OR в одной ноде IF нельзя — для сложной логики ставь две IF подряд или используй ноду Switch.

Когда это пригодится
Разный маршрут для разных типов входящих данных

Например, обращения с пометкой «срочно» — в одну ветку с быстрым уведомлением, остальные — в обычную обработку. Без IF пришлось бы писать одну общую логику, в которой каждая нода сама бы фильтровала ненужное.

Типичные ошибки

  • Сравниваешь число с текстом и удивляешься, что equals не срабатывает → проверь, что у левой и правой части одинаковый тип. Поле из HTTP-ответа часто приходит строкой "200", а не числом 200.
  • В правой части забыл выражение и сравниваешь буквально с текстом {{ $json.field }} → выражение начинается с = в начале значения или с переключения поля в режим Expression.
  • Ветка false не подключена ни к какой ноде → элементы, не прошедшие условие, просто отбрасываются. Это нормально, если так и задумано; иначе подключи ноду или используй Filter вместо IF.
📖Filter и IF в n8n: когда что выбиратьстатья · ~2 мин▾

Похожая задача — разные ноды

Filter и IF решают одно: пропустить дальше только нужные элементы. Условия в них настраиваются одинаково — те же операторы, те же поля. Разница не в логике сравнения, а в том, что происходит с элементами, которые условию не соответствуют.

Как это работает

  • IF — двусторонняя развилка. У ноды два выхода: true и false. Элементы, прошедшие условие, уходят в верхнюю ветку, остальные — в нижнюю. Дальше каждую ветку можно обработать по-своему: записать в лог, отправить уведомление, пометить как ошибку.
  • Filter — односторонний фильтр. Один выход. Элементы, прошедшие условие, идут дальше. Остальные молча отбрасываются — после Filter их в потоке нет.
Простое правило
IF — когда «отбракованные» нужны; Filter — когда нет

Если по «не подошедшим» элементам ничего делать не надо — бери Filter, схема будет компактнее. Если их надо обработать отдельно (залогировать, ответить пользователю, переслать в другую ветку) — бери IF.

Когда что выбирать

  • IF — нужно обработать обе группы элементов: прошедшие условие и не прошедшие.
  • Filter — отбракованные элементы не нужны, их можно просто выкинуть.
  • Switch — нужно разделить элементы на три и больше веток. Ни IF, ни Filter тут не подходят: для трёх веток IF придётся ставить каскадом, а Filter вообще не умеет «другие случаи».

Типичные ошибки

  • Поставили Filter, а «не подошедшие» элементы оказались нужны для отчёта → данные пропали молча, без следов в логах. Замена на IF решает.
  • Поставили IF и подключили только ветку true, ветка false висит пустая → по результату это тот же Filter, но в воркфлоу болтается «лишний» неподключённый выход и путает читателя. Если ветка false не нужна — замени ноду на Filter.
  • Думают, что Filter «быстрее» IF. На реальных объёмах данных в задачах школы разница в производительности нулевая. Выбор между ними — про смысл схемы, не про скорость.

Что такое expression

В n8n expression — это маленький кусочек кода в фигурных скобках, который динамически подставляет значение в поле ноды. Записывается так:

={{ <выражение> }}

Знак равенства = в начале — это маркер для n8n: «трактуй содержимое этой ячейки не как обычную строку, а как выражение, которое нужно вычислить». Без = n8n запишет в поле литеральный текст {{ ... }} как есть.

Внутри {{ }} пишется JavaScript-выражение. Самое частое — чтение значения из другой ноды.

Зачем нужны expressions

Каждое поле в параметрах ноды может быть либо фиксированным значением (например, URL https://api.example.com), либо expression'ом — формулой, которая считается на лету в момент исполнения ноды. Expressions нужны, когда значение зависит от данных, пришедших из предыдущей ноды.

К примеру
Где это встречается

К примеру, ты сделал ноду «прочитать строку из таблицы конфига», а следующая нода — HTTP Request к внешнему API. В URL этой HTTP-ноды можно подставить значение из конфига: ={{ $('Get Config').item.json.api_url }} — n8n при каждом исполнении возьмёт актуальный URL из таблицы, а не зашитый в ноду.

Как читать значения из других нод

Базовая формула: ={{ $('<имя_ноды>').item.json.<поле> }}

Разбор по частям:

  • $('<имя_ноды>') — обращение к ноде по имени (то, что написано на canvas под нодой). Имя — в одинарных кавычках, точно как на canvas.
  • .item — текущий элемент, который обрабатывается. У большинства нод элементов несколько (массив), item берёт тот, что соответствует текущей итерации.
  • .json — содержимое поля json элемента (там лежат все данные, которые нода вернула).
  • .<поле> — конкретное поле внутри json. Если поле вложенное — пишется через точку: .config.embedding_model.

Пошагово — как вставить expression в поле

  1. Кликни на иконку шестерёнки рядом с полем, выбери Add Expression (или нажми Expression в верхнем правом углу поля).
  1. В появившемся окне (expression builder) — слева список нод воркфлоу, можно кликать по полям и они вставятся в формулу.
  1. Справа — preview: видно, какое значение получится прямо сейчас. Если красным — формула или путь к полю некорректные.
  2. Закрой окно, поле остаётся в режиме expression (синий фон + иконка fx).
Типичная ошибка
`=` в начале — обязателен

Если скопировал готовое выражение из учебной задачи и вставил без = в начале — n8n запишет в поле текст {{ ... }} как есть, нода уйдёт делать HTTP-запрос на URL вроде ={{ $('...').item.json.api_url }} буквально. Симптом — нода падает с ошибкой Invalid URL или возвращает 404 на странную строку.

Типичные ошибки

  • Имя ноды не совпадает. В формуле $('Get Config') имя должно точно совпадать с подписью ноды на canvas. Если переименовал ноду — старые expressions, ссылающиеся на неё, сломаются.
  • Путь к полю не существует. .item.json.embedding_dimensions упадёт с undefined, если у элемента такого поля нет. Препетий — посмотри preview в expression builder, он сразу показывает значение или ошибку.
  • Забыл = в начале. Самая частая опечатка — описано в callout выше.
  • Кавычки внутри {{ }}. Внутри expression строки оборачиваются одинарными кавычками: ={{ 'Hello ' + $json.name }}. Двойные конфликтуют с JSON.

Когда expression не нужен

Иногда поле должно быть фиксированным. Например, метод HTTP-запроса (GET / POST) или имя коллекции, которое одинаково для всего воркфлоу — нет смысла читать его динамически. Простое правило: если значение меняется от итерации к итерации или зависит от пришедших данных — нужен expression; если значение постоянное — оставляй фиксированным.

Drag-and-drop из панели Input — самый быстрый способ связать ноды

В n8n каждая нода видит данные предыдущих нод в панели Input слева от параметров. Чтобы подставить любое поле из этого Input в параметр текущей ноды, не нужно вручную писать expression — достаточно перетащить поле мышкой. n8n сам подставит правильное выражение со ссылкой на нужную ноду.

Как это работает

  1. Открой параметры ноды двойным кликом. Слева автоматически появится панель Input — это выходные данные предыдущей ноды (или одной из предыдущих, если их несколько).
  2. Разверни в Input структуру до нужного поля — она показывается как дерево вложенных объектов: message → chat → id, output → [0] → content → [0] → text и так далее.
  3. Перетащи нужное поле мышкой прямо в поле параметра справа. В поле появится выражение вида ={{ $('<имя предыдущей ноды>').item.json.<путь.к.полю> }}.
  4. Поле в правой панели окрасится синим — это значит, что параметр теперь не литерал, а expression: на каждом запуске значение будет браться из реальных данных.
К примеру
Откуда n8n берёт имя ноды

К примеру, если предыдущая нода называется Telegram Trigger, и ты перетащил из неё поле message.text — n8n запишет ={{ $('Telegram Trigger').item.json.message.text }}. Имя ноды берётся ровно такое, как написано на canvas. Иногда — если переименуешь ноду уже после drag-and-drop, выражение сломается, потому что в формуле останется старое имя.

Откуда выбрать источник данных

Если у ноды несколько входов или нужно дотянуться не до прямого предыдущего шага, а до более раннего — сверху панели Input есть переключатель источника. К примеру, в финальной ноде workflow можно вывести Input от первого триггера, чтобы достать оттуда поле, которого нет в выходе ближайшей предыдущей ноды.

Типичные ошибки

  • Перетащил не на то поле параметра. Drag-and-drop срабатывает на любом текстовом поле — легко уронить значение не в Text, а в Chat ID, и наоборот. После перетаскивания проверь, что в нужном поле появилось ожидаемое выражение.
  • Перетащил из объекта-обёртки, а не из конкретного поля. Если потянуть за message, а не за message.text — n8n подставит весь объект, в текстовое поле уйдёт [object Object] или JSON-строка. Раскрывай дерево до примитивного значения.
  • Сломался путь после переименования ноды. В выражении $('Telegram Trigger') имя ноды фиксированное. Если потом переименовать Telegram Trigger в что-то другое — нужно либо вернуть старое имя, либо переписать выражение вручную.

Когда лучше не использовать drag-and-drop

Если значение должно быть фиксированным (например, имя коллекции, метод HTTP, статичный текст приветствия) — drag-and-drop не нужен, просто впечатай значение в поле как обычный текст. Drag-and-drop — это про динамические параметры, которые меняются от запуска к запуску в зависимости от входных данных. </content> </invoke>

🔀Оператор exists в ноде Filter: чем отличается от isNotEmptyстатья · ~2 мин▾

Оператор exists в ноде Filter

Ноды Filter и IF в n8n умеют сравнивать значения почти всеми способами — равно, не равно, больше, меньше, содержит. Среди этих операторов есть пара очень похожих, но не одинаковых: exists и isNotEmpty. Разница тонкая, но важная — на ней ловятся неочевидные баги.

Зачем это нужно

В n8n обычная ситуация: предыдущая нода вернула JSON, но одно из полей может отсутствовать. Например, у одного входящего объекта поле text есть, у другого его нет вовсе. Если попытаться сравнивать значение поля как строку — null/undefined приведутся к строке "undefined" и сравнение даст неожиданные результаты. Поэтому в Filter/IF и появилось специальное семейство операторов «проверь существование» — они не сравнивают значение, а отвечают на вопрос «такое поле в этом объекте вообще есть?».

Как это работает

  • exists — true, если ключ в объекте присутствует, любое значение (даже пустая строка, ноль, false, пустой массив). False — только если ключа нет совсем (undefined). Right value не нужен — оператор просто проверяет наличие.
  • does not exist — обратное; срабатывает только когда ключа нет.
  • isNotEmpty — true, если значение есть и оно непустое. Для строки «непустое» = не "", для массива — длина > 0, для null/undefined — false. То есть isNotEmpty строже: проверяет и существование, и содержательность.
  • isEmpty — обратное к isNotEmpty.
| Значение поля       | exists | isNotEmpty |
|---------------------|--------|------------|
| "привет"            | true   | true       |
| ""                  | true   | false      |
| 0                   | true   | true       |
| null                | true   | false      |
| undefined / нет     | false  | false      |

Когда выбирать exists, а когда isNotEmpty:

  • Поле может отсутствовать, и сам факт его отсутствия — это сигнал «такой случай нам не подходит» (item нужно отфильтровать). Тут чище работает exists. К примеру, в Telegram-боте поле message.text есть у текстовых сообщений и физически отсутствует у голосовых / стикеров.
  • Поле всегда присутствует, но значение бывает пустой строкой или null, и эти случаи нужно отбросить — нужен isNotEmpty. К примеру, форма всегда отдаёт comment, но это поле бывает пустым, если пользователь его не заполнил.

Поведение ноды Filter без callout'а

Filter молча дропает item, у которого условие false: в Executions не появляется ни ошибки, ни предупреждения, ветка после Filter просто не получает этот item. Это часто путает — кажется, что workflow «оборвался», хотя на самом деле сработало именно фильтрование. Если нужно явное поведение (например, отвечать пользователю «не понял, что ты прислал»), вместо Filter ставится IF: у IF две ветки — true и false, и в false можно навесить логику обработки отброшенных случаев.

Типичные ошибки

  • Перепутали exists и isNotEmpty при работе с пустой строкой — exists пропустит item с text: "" дальше как валидный, и следующая нода получит пустую строку вместо реального текста.
  • Сравнили несуществующее поле через equals/contains — оператор сравнения попытается привести undefined к строке, и результат будет непредсказуемым. Для проверки наличия нужно именно exists.
  • Ожидают ошибку при undefined — Filter не падает на отсутствующем поле, он просто считает условие false. Если нужно явное падение workflow, используется нода Stop and Error.
  • Не учли тип данных — в свойствах Filter тип (String, Number, Boolean, Object) выбирается отдельно, и exists всё равно требует выбора типа. Если выбран не тот тип, оператор может сработать не так, как ожидаешь.

Когда не подходит

Если по логике задачи отсутствующий item нужно не выбросить, а превратить в «значение по умолчанию» — Filter не нужен; стоит брать ноду Set с выражением ?? ({{ $json.text ?? "—" }}). Filter — это «отрежь и забудь», не «исправь».

📖Операторы contains / does not contain в IF и Filterстатья · ~2 мин▾

Содержит или нет — для строк и массивов

Среди операторов сравнения в нодах IF, Filter и Switch есть пара contains / does not contain. Они отвечают на простой вопрос: «лежит ли искомое внутри проверяемого значения?». Доступны для двух типов данных — String (подстрока внутри строки) и Array (элемент внутри массива). Для других типов их в выпадающем списке нет.

Как это работает

  • Для типа String: contains возвращает true, если правое значение подстрока левого. does not contain — наоборот. Поиск идёт по символам, длина и позиция совпадения не важны.
  • Для типа Array: contains сравнивает элементы массива с правым значением через равенство. Для массива простых значений (["paid", "open", "done"]) сравнение работает буквально. Для массива объектов сравнение целиком объекта с правой частью — на практике почти не используется, обычно нужно искать по полю, а это уже не contains.

Как настроить

  1. В блоке Conditions выбери тип данных левой части — String или Array.
  2. В выпадающем списке операторов выбери contains или does not contain.
  3. В левое поле положи выражение (строку или массив), которое проверяешь.
  4. В правое поле впиши искомое — фиксированную подстроку, выражение или элемент массива.
Регистр строк
contains для String чувствителен к регистру

По умолчанию "Paid" не содержит "paid". Если регистр в данных может скакать (текст от пользователя, заголовки HTTP-ответа), нормализуй левую часть: {{ $json.text.toLowerCase() }}. Альтернатива — переключить ignoreCase в опциях условия (Add Option → Ignore Case), но видимое выражение слева останется в исходном регистре, и спустя месяц это путает.

Когда contains — не лучший выбор

  • Нужно «значение начинается с X» или «заканчивается на X» → бери операторы starts with / ends with, схема читается понятнее.
  • Нужен сложный шаблон (классы символов, опциональные части) → matches regex.
  • Проверяешь массив объектов по полю (items[].status == "open") → contains тут не сработает. Нужен предварительный map/pluck в Code/Set, или сразу нода Filter с условием на поле объекта.

Типичные ошибки

  • Сравнили строку с пробелом по краям: " привет " не содержит "привет ", если в правой части пробел поставили не там → почисти левую часть .trim().
  • Ожидают, что для массива объектов contains найдёт по значению поля → нужен другой подход (распаковать массив через Split Out и работать с items).
  • Перепутали contains и is equal to для строки. contains "paid" сработает и на "paid", и на "paid_at_2024", и на "unpaid" — последнее обычно нежелательно. Если нужна точная подстрока на границе слова — matches regex с \b.
  • Забыли, что does not contain для отсутствующего поля даст true (пустая строка ничего не содержит). Если поле может отсутствовать — поставь до сравнения exists или подстрахуйся {{ $json.field ?? '' }}.

Fixed и Expression — два режима любого поля

В правом верхнем углу почти каждого поля параметров ноды есть переключатель Fixed | Expression. Это два режима ввода значения: фиксированное (написал текст — этот текст и пойдёт) или формула на JavaScript, которая вычисляется заново при каждом выполнении ноды. Один и тот же тумблер есть у обычных текстовых полей, у dropdown-ов, у credential pickers и у resource locator полей (где выбираешь объект из списка — таблицу, документ, канал).

Как работает каждый режим

  • Fixed — обычный ввод. Что написал — то и попадёт в запрос/в подстановку. У resource locator (когда выбираешь объект — таблицу, документ, sheet) Fixed означает выбор из dropdown «From list» или ручной ввод id одним значением.
  • Expression — поле подсвечивается оранжевым, а внутри можно писать формулу в фигурных скобках: {{ $json.field }}, {{ $now.toISO() }} и т.п. Значение пересчитывается на каждой итерации, для каждого пришедшего элемента.
Когда переключать в Expression
Значение зависит от данных или меняется между запусками

Если значение должно зависеть от того, что пришло из предыдущей ноды (имя пользователя, id записи, текст сообщения), от текущего времени или от любой логики — переключай в Expression. Если значение всегда одно и то же (метод HTTP POST, имя таблицы, фиксированный slug) — оставляй Fixed.

Как переключить режим

Три способа сделать одно и то же:

  1. Клик по Fixed | Expression в правом верхнем углу поля. Самый явный — видно сразу, в каком режиме поле.
  2. Иконка шестерёнки слева от поля → пункт Add Expression. Удобно когда хочешь сразу открыть expression builder (окно с деревом нод слева и preview значения справа).
  3. Drag-and-drop значения из панели Input слева в поле. Это самый частый способ в учебных задачах: открываешь NDV предыдущей ноды через мини-карту слева или раскрываешь панель Input, цепляешь нужное поле мышкой и тянешь в параметр текущей ноды. n8n при отпускании сам переключает режим в Expression и подставляет правильную ссылку — {{ $json.field }} или {{ $('Имя ноды').item.json.field }}.

В обратную сторону — клик по Fixed в углу. Содержимое expression-режима очистится, поле станет обычным текстовым.

Тонкость
У resource locator в Expression нет dropdown

В resource locator полях (например, выбор Data table или Spreadsheet) при переключении в Expression список объектов из dropdown пропадает — теперь ты должен сам подставить нужный id или slug формулой. Полезно когда id берётся из предыдущей ноды; если нужно просто выбрать существующий объект мышкой — оставайся в Fixed.

Типичные ошибки

  • Забыл переключить в Expression, написал формулу как текст. Поле в Fixed-режиме воспринимает {{ $json.name }} как обычные символы и отправит эту строку буквально. Признак: значение в preview / в результате выполнения содержит {{ и }} как есть. Лечение: переключить в Expression вручную (тумблер в правом верхнем углу) или перетянуть значение из панели Input заново.
  • Стёр выбор из resource locator, переключив в Expression и обратно. Переключение режима очищает содержимое поля. Если в Fixed уже был выбран нужный объект из списка — после Expression → Fixed его придётся выбрать заново.

📖Кнопка «Execute step»: запуск одной ноды без всего воркфлоустатья · ~2 мин▾

Запустить одну ноду, не дёргая весь workflow

Execute step — это кнопка внутри панели ноды (открывается двойным кликом по ноде). По смыслу — младший брат большой кнопки Execute workflow внизу canvas: та запускает весь воркфлоу с начала, эта — только текущую ноду. Удобно, когда настраиваешь параметры и хочешь сразу видеть, что отдаёт нода, не перезапуская всю цепочку.

Как это работает

Когда ты жмёшь Execute step, n8n берёт уже существующие данные на входе ноды (output предыдущей ноды из последнего прогона) и прогоняет через эту ноду. Сама нода выполняется реально: HTTP-запрос уходит, запись в базу пишется, письмо отправляется. Это не «сухой прогон», а полноценное выполнение — только одного шага.

  • key: Execute workflow

value: Запускает весь воркфлоу от триггера. Нужен полный прогон с самого начала. Каждый раз заново дёргает все API.

  • key: Execute step

value: Запускает только эту ноду. Берёт вход из последнего output предыдущей ноды. Идеально для подбора параметров и отладки.

  • key: Когда работает

value: Только если на входе ноды есть данные с прошлого прогона. Если ни разу не запускал предыдущие — кнопка либо ничего не вернёт, либо упадёт «No input data».

Типичный сценарий: запустил весь воркфлоу один раз (Execute workflow), увидел что нода HTTP Request отдаёт не то, поправил поле URL, нажал Execute step — увидел новый результат за секунду. Предыдущие ноды повторно не выполнялись.

У триггеров кнопка называется иначе
Смысл тот же — запустить один шаг и получить тестовое событие

У некоторых триггерных нод n8n кнопка Execute step подписана по-другому, потому что у триггера нет «входа» — он сам должен сходить за событием. Например, у RSS Trigger она называется Fetch Test Event: клик подтянет последние записи из ленты и положит их на выход ноды как тестовый input для остального воркфлоу. Работает идентично Execute step обычной ноды — точно так же используется для отладки и подбора параметров без полного прогона.

Важно про побочные эффекты
Каждый Execute step — это реальный вызов

Нода с записью в базу или отправкой сообщения каждое нажатие Execute step делает реально: 5 нажатий = 5 записей / 5 писем. Для отладки таких нод используй тестовые данные или подменяй credentials.

Типичные ошибки

  • Жать Execute step на ноде, у которой ещё не было входных данных → «No input data», нода не запустится. Сначала прогони весь воркфлоу через Execute workflow или хотя бы триггер вручную, чтобы у каждой ноды появился output.
  • Использовать Execute step для проверки изменений в предыдущих нодах → текущая нода работает со старым output, изменения не подхватятся. Если поправил предыдущую ноду — перезапусти её, а потом уже текущую.
  • Тестировать ноду с побочным эффектом (запись в БД, отправка письма) на проде → каждый клик создаёт реальную запись. Использовать тестовый credential или временно подменять параметры на безопасные.

📖$json или $('Имя ноды'): когда какая форма обращения к даннымстатья · ~2 мин▾

Две формы обращения — разница только в источнике

В n8n у одной и той же ноды есть две формы прочитать поле из данных воркфлоу:

{{ $json.business_idea }}
{{ $('Form Trigger').item.json.business_idea }}

Обе достают одно и то же значение, но смотрят в разные места:

  • $json — это выход ноды, которая стоит прямо перед текущей. Короткая форма для «возьми поле из предыдущей ноды».
  • $('Имя ноды') — это обращение к любой ноде воркфлоу по имени, в том числе к той, что стоит на 2-3-5 шагов раньше.

Когда писать какую

Правило простое:

  • Хочешь взять данные у предыдущей ноды (стрелка идёт прямо в текущую) — пиши $json.поле.
  • Хочешь дотянуться до ноды глубже по цепочке (между ней и текущей есть промежуточные ноды) — пиши $('Имя ноды').item.json.поле.
Пример
Цепочка из трёх нод

Цепочка: Form Trigger → OpenAI → Send Message.

В Send Message обращения к двум разным источникам выглядят так:

  • к ответу OpenAI (предыдущая нода): ={{ $json.output }}
  • к полю формы (нода на два шага назад): ={{ $('Form Trigger').item.json['Email'] }}

Если в Send Message написать ={{ $json['Email'] }} — поле не найдётся, потому что у OpenAI в выходе нет Email. Это типичная ошибка студентов.

Почему короткая форма вообще существует

$json — это просто синоним для $input.item.json, то есть «текущий обрабатываемый элемент входа». А вход текущей ноды формируется из выхода предыдущей. Поэтому $json.поле и $('Предыдущая нода').item.json.поле всегда дают одно и то же значение — пока стрелка идёт напрямую.

$('Имя ноды') нужна там, где короткая форма не справится: между источником и текущей нодой есть промежуточные шаги, которые этого поля уже не передают (например, OpenAI вернул только текст ответа, и поле формы Email к моменту Send Message живёт только в выходе Form Trigger).

Типичные ошибки

  • Написал $json.Email в ноде после OpenAI. OpenAI возвращает output, а не Email. Решение — обратиться по имени: $('Form Trigger').item.json['Email'].
  • Имя ноды в $('...') не совпадает с подписью на canvas. В expression имя — буквально то, что написано под нодой на canvas. Переименовал ноду — все старые expressions, ссылавшиеся на её прежнее имя, сломаются.
  • Забыл .item после имени ноды. Полная форма — $('Имя').item.json.поле. Без .item обращаешься не к данным, а к самой ноде как объекту — поле не найдётся.

📖Нода Respond to Webhook: что вернуть внешнему сервисустатья · ~3 мин▾

Нода Respond to Webhook: что вернуть внешнему сервису

Когда Webhook-триггер получает запрос, по умолчанию n8n сам отвечает вызывающему сервису — отдаёт результат последней ноды как JSON и код 200. Это поведение подходит не всегда: иногда нужен свой формат ответа, свой HTTP-код, редирект или вообще пустой ответ. Тогда в работу включается отдельная нода Respond to Webhook — она формирует тот ответ, который увидит вызывающий.

Как это работает

Чтобы Respond to Webhook сработала, в самом Webhook-триггере поле Respond нужно переключить на Using 'Respond to Webhook' Node. После этого триггер перестаёт отвечать сам и ждёт, пока workflow дойдёт до этой ноды. Когда дойдёт — n8n возьмёт её настройки и отправит ответ обратно вызвавшему сервису.

Основные параметры ноды:

Что положить в тело ответа: JSON, текст, бинарный файл, все входящие items, первый входящий, JWT-токен, редирект (HTTP 3xx) или ничего (No Data).

Само содержимое ответа. Для JSON — объект или ссылка через выражение. Для текста — строка. Для бинарника — имя поля с файлом.

HTTP-код ответа. По умолчанию 200. Меняй, когда хочешь явно сказать «создано» (201), «нет содержимого» (204) или вернуть ошибку (4xx/5xx).

Дополнительные заголовки: Content-Type, Location для редиректа, любые свои.

Когда это пригодится
Внешний сервис ждёт конкретный формат ответа

📖Кнопка «Execute workflow»: ручной запуск для тестированиястатья · ~2 мин▾

Кнопка ручного запуска для теста

В редакторе workflow внизу canvas есть большая розовая кнопка «Execute workflow». Это основной инструмент тестирования: один клик — и весь workflow прогоняется от триггера до конца, прямо в редакторе. На production это никак не влияет — даже если workflow активен (тумблер сверху), кнопка запускает отдельный тестовый прогон.

Как это работает

  1. Нажимаешь Execute workflow внизу canvas.
  1. n8n запускает workflow от триггера. Ноды по очереди подсвечиваются: серая → жёлтая (выполняется) → зелёная (готово) или красная (ошибка). Внизу каждой ноды появляется счётчик пришедших на вход элементов.
  1. Клик по любой ноде после прогона — видишь её input и output. Это и есть отладка: где данные пропали или пришли не в том виде.
Запуск с середины
Кнопка ▶ на самой ноде

У каждой ноды на canvas есть своя маленькая кнопка ▶ (появляется при наведении). Она запускает только эту ноду на данных, которые лежат в её input от предыдущего прогона. Удобно, когда правишь одну ноду и не хочешь гонять весь workflow заново.

Особый случай: workflow с Chat Trigger

Если в workflow стоит триггер Chat Trigger (встроенный в n8n чат), правила для тестирования другие — Execute workflow появляется не сразу.

  • Когда у Chat Trigger ещё нет запиненных данных — внизу canvas вместо Execute workflow стоит кнопка Open chat. Клик открывает встроенный чат-виджет n8n: ты пишешь сообщение, оно идёт в триггер, workflow выполняется от события «новое сообщение в чате». Это и есть основной путь тестирования Chat Trigger.
  • Когда данные Chat Trigger запинены (через Pin data в Output триггера) — кнопка Open chat сменяется на привычную Execute workflow. n8n больше не ждёт нового сообщения в чате, а гоняет workflow на запиненных данных. Удобно для отладки downstream-нод без необходимости каждый раз писать в чат.

Логика та же что у обычных триггеров — если есть закреплённый тестовый вход, n8n показывает Execute workflow для запуска на нём; если нет — показывает способ получить реальное событие (для Chat Trigger это Open chat, для Webhook — Test URL и т.д.).

Типичные ошибки

  • Нажал кнопку, ничего не происходит → в workflow нет триггера или есть только Schedule/Webhook без Manual Trigger. Добавь временный Manual Trigger или используй кнопку ▶ на конкретной ноде.
  • Нода красная, дальше ничего не выполнилось → клик по ноде → вкладка с ошибкой. Прогон останавливается на первой упавшей ноде.
  • На ноде «0 items» и она не выполнилась → предыдущая нода вернула пустой массив. Это не ошибка, это «нечего обрабатывать». Смотри output предыдущей ноды.

Schedule Trigger в n8n: расписания и cron

Schedule Trigger — стартовая нода n8n, которая запускает воркфлоу по расписанию: раз в минуту, каждый час, по будням в 9:00 или по любому cron-выражению. В отличие от polling-триггера, Schedule Trigger никуда не ходит за данными — он просто срабатывает по таймеру и передаёт управление следующей ноде.

Как это работает

В ноде есть один основной параметр — Trigger Interval. От его значения зависит набор дополнительных полей:

  • Seconds / Minutes / Hours — запуск каждые N единиц. Поле Seconds Between Triggers (или Minutes/Hours).
  • Days — раз в N дней в указанное время. Поля: Days Between Triggers, Trigger at Hour, Trigger at Minute.
  • Weeks — по выбранным дням недели в указанное время. Поля: Weeks Between Triggers, Trigger on Weekdays (мульти-выбор), Trigger at Hour/Minute.
  • Months — раз в N месяцев в указанный день месяца. Поля: Months Between Triggers, Trigger at Day of Month, Trigger at Hour/Minute.
  • Custom (Cron) — произвольное расписание через cron-выражение. Поле: Expression.

Cron-выражения

Для нестандартного расписания выбирай Custom (Cron). Формат — стандартный cron из 5 полей: минута час день_месяца месяц день_недели. Звёздочка * значит «любое значение», слеш */N — «каждые N единиц», запятая — перечисление, дефис — диапазон.

Примеры cron-выражений
*/15 * * * *    — каждые 15 минут
0 * * * *       — в начале каждого часа
0 9 * * 1-5     — по будням в 9:00
0 9,18 * * *    — каждый день в 9:00 и 18:00
0 0 1 * *       — в полночь первого числа каждого месяца
30 3 * * 0      — по воскресеньям в 3:30 ночи
Часовой пояс
Schedule Trigger работает в timezone инстанса n8n

Все расписания привязаны к часовому поясу самого n8n. Если инстанс в UTC, а ты ждёшь «9:00 по Москве» — поставь cron 0 6 * * * (UTC = МСК − 3) или поменяй timezone в настройках воркфлоу (Settings → Timezone). На школьной платформе у каждого студента свой инстанс — проверь его timezone, прежде чем настраивать критичное расписание.

Несколько расписаний в одной ноде
Список интервалов под кнопкой Add Rule

Внутри одной ноды Schedule Trigger можно добавить несколько правил кнопкой Add Rule — например, «каждые 5 минут в рабочие часы» + «раз в час ночью». Каждое срабатывание любого правила запускает воркфлоу с одного и того же выхода ноды.

Типичные ошибки

  • Воркфлоу не срабатывает в ожидаемое время → инстанс в другом timezone. Проверь Settings → Timezone в воркфлоу и сравни с тем, что ожидаешь.
  • Расписание «каждый час» срабатывает каждые ~59-61 минут → так и должно быть для режимов Seconds/Minutes/Hours: они считают от момента активации. Чтобы попадать «в ровно 0 минут», используй Custom (Cron) с выражением 0 * * * *.
  • Воркфлоу не запускается, хотя расписание правильное → нода Schedule Trigger работает только когда воркфлоу активирован (тумблер Active справа сверху). В режиме Execute workflow срабатывает один раз сразу, расписание не применяется.
  • Сложное cron-выражение возвращает ошибку → n8n использует пятипольный cron без секунд (в отличие от Quartz/Spring с шестью полями). 0 0 9 * * 1-5 — это уже шесть полей, не сработает; правильно 0 9 * * 1-5.

Multi-item processing в n8n: одна нода — несколько прогонов

Когда предыдущая нода вернула N items (например, 3 строки из Data Table) — следующая нода автоматически выполнится N раз: по одному прогону на каждый item. Никакого цикла настраивать не нужно, это нативное поведение n8n.

Как это работает

В каждом прогоне $json содержит данные ровно одного item — текущей строки. Так выражение {{ $json.name }} автоматически подставляет правильное значение для каждой записи.

Пример
3 строки → 3 сообщения в Telegram

Data Table вернул 3 записи: Иван, Мария, Сергей. Следующая нода Telegram Send Message без дополнительных настроек выполнится 3 раза и отправит 3 отдельных сообщения — по одному на каждого. Это поведение по умолчанию.

Как поменять поведение

Часто такое поведение не нужно. Если нода-отправитель должна послать одно сводное сообщение — N прогонов = N лишних сообщений. На этот случай в n8n есть три инструмента.

1. Execute Once — переключатель в Settings ноды. Включаешь его — нода прогоняется ровно один раз, n8n даёт ей только первый item, остальные игнорируются. Подходит для нод-отправителей: Telegram, Email, HTTP Request — когда нужно одно действие на весь массив. Подробнее — [[n8n-execute-once]].

2. Run Once for All Items — режим у Code-ноды (выбирается в верхнем переключателе). В этом режиме внутри Code доступен весь массив через items, и нода возвращает результат за один прогон.

3. Агрегация через Loop / Merge / SplitInBatches — отдельные ноды-цикла. Нужны когда логика сложнее «взять первый» или «всё одним куском»: пакетная обработка, объединение веток, custom-агрегация.

Как достать весь массив из выражения

Внутри выражения {{ ... }} любая нода может прочитать весь входной массив целиком — через $input.all(). Это массив items, по которому можно вызвать .length, .map(), .filter() и любые JS-методы массивов.

{{ $input.all().length }}                       // сколько items пришло
{{ $input.all().map(i => i.json.name).join(', ') }}   // имена через запятую

Особенность: $input.all() читает массив до того, как Execute Once срезает его до первого item. Поэтому если на ноде Telegram стоит Execute Once и на вход пришло 3 items — $input.all().length вернёт 3, а не 1. Это удобно для сводных сообщений вида «Получено заявок: 3».

Когда полезно знать

  • Если кейс «N сообщений вместо одного» — это не баг, это поведение по умолчанию. Решение — Execute Once или Run Once for All Items.
  • Если нужно посчитать число входных items в Set / Telegram / любой ноде — $input.all().length в выражении.
  • Если нужна последовательная обработка с большим массивом и контролем — SplitInBatches.

Похожая задача — разные ноды

Filter и IF решают одно: пропустить дальше только нужные элементы. Условия в них настраиваются одинаково — те же операторы, те же поля. Разница не в логике сравнения, а в том, что происходит с элементами, которые условию не соответствуют.

Как это работает

  • IF — двусторонняя развилка. У ноды два выхода: true и false. Элементы, прошедшие условие, уходят в верхнюю ветку, остальные — в нижнюю. Дальше каждую ветку можно обработать по-своему: записать в лог, отправить уведомление, пометить как ошибку.
  • Filter — односторонний фильтр. Один выход. Элементы, прошедшие условие, идут дальше. Остальные молча отбрасываются — после Filter их в потоке нет.
Простое правило
IF — когда «отбракованные» нужны; Filter — когда нет

Если по «не подошедшим» элементам ничего делать не надо — бери Filter, схема будет компактнее. Если их надо обработать отдельно (залогировать, ответить пользователю, переслать в другую ветку) — бери IF.

Когда что выбирать

  • IF — нужно обработать обе группы элементов: прошедшие условие и не прошедшие.
  • Filter — отбракованные элементы не нужны, их можно просто выкинуть.
  • Switch — нужно разделить элементы на три и больше веток. Ни IF, ни Filter тут не подходят: для трёх веток IF придётся ставить каскадом, а Filter вообще не умеет «другие случаи».

Типичные ошибки

  • Поставили Filter, а «не подошедшие» элементы оказались нужны для отчёта → данные пропали молча, без следов в логах. Замена на IF решает.
  • Поставили IF и подключили только ветку true, ветка false висит пустая → по результату это тот же Filter, но в воркфлоу болтается «лишний» неподключённый выход и путает читателя. Если ветка false не нужна — замени ноду на Filter.
  • Думают, что Filter «быстрее» IF. На реальных объёмах данных в задачах школы разница в производительности нулевая. Выбор между ними — про смысл схемы, не про скорость.

Equal To и Not Equal To — точное совпадение и его отрицание

В нодах IF, Filter и Switch среди операторов для типов String, Number, Boolean есть парные is equal to и is not equal to. Первый сравнивает левое значение с правым на полное совпадение и возвращает true, если они одинаковые. Второй — наоборот: true, если значения различаются. Это самые частые операторы для ветвления: «если статус == paid — иди в одну ветку, иначе — в другую», или «если автор сообщения не ты — отвечай в чат».

Как настроить

  1. В блоке Conditions выбери тип данных левой части (String, Number, Boolean).
  2. В выпадающем списке операторов выбери is equal to или is not equal to.
  3. В левое поле положи выражение со значением из предыдущей ноды, например {{ $json.status }}.
  4. В правое поле впиши то, с чем сравниваем, — фиксированный текст / число / второе выражение.

Главная ловушка — типы данных

Оба оператора чувствительны к типу значения. Строка "200" и число 200 для них разные, и условие даст false (для is equal to) либо true (для is not equal to) — хотя глазами кажется, что значения одинаковые. Откуда это берётся:

  • HTTP-ответ часто приходит со статусом в виде строки ("200"), а сравниваем с числом 200.
  • Поле из формы — всегда строка, а сравниваем с числом.
  • Поле из БД — число, а сравниваем со строкой из пользовательского ввода.
Проверь тип
Если equals не срабатывает на одинаковых на вид значениях

В UI ноды IF слева от оператора есть выпадающий список типов (String / Number / ...). Сначала проверь, что выбранный тип совпадает с реальным типом данных. Если данные приходят строкой "200", а нужно сравнить как число — оберни левую часть в {{ Number($json.code) }} и переключи тип на Number.

Регистр строк

Для строк оба оператора чувствительны к регистру: "Paid" и "paid" — разные. Если приходит непредсказуемый регистр (от пользователя, из внешнего API), нормализуй левую часть: {{ $json.status.toLowerCase() }} и сравнивай с маленькой буквы.

Своя ловушка у Not Equal To — поле может отсутствовать

is not equal to ведёт себя коварно с отсутствующим или пустым полем. Если левая часть — undefined, null или пустая строка, а сравниваем мы её с конкретным значением (например, "done"), то условие даст true («не равно done»), и item спокойно пойдёт в ветку «не равно». Это часто не то, что хотели: обычно «значение пустое» = «случай отдельный, не равен ни done, ни чему-либо другому», и обрабатывать его надо отдельной веткой.

Подстраховка
Не используй is not equal to без проверки на наличие

Перед is not equal to поставь отдельным условием exists или is not empty для того же поля — или объедини их через AND в одном IF. Тогда в ветку «не равно» попадут только item'ы, у которых поле есть и непустое. Альтернатива — нормализовать левую часть выражением: {{ $json.status ?? '' }}, и сравнивать с пустой строкой, если так удобнее.

Когда equals — не лучший выбор

  • Нужно «значение похоже на X, но не обязательно точно равно» (статус paid_at_2024, paid_success) → бери оператор contains или matches regex.
  • Сравниваешь с одним из нескольких значений ("paid", "completed", "success") → ставь несколько условий с OR или используй ноду Switch с правилами на каждое значение.
  • Проверяешь, что поле просто есть или не пустое, а не равно конкретному значению → бери exists / isNotEmpty (см. filter-exists-vs-isnotempty).
  • Нужно «всё, что не в этом списке» ("paid", "completed", "success") — три условия is not equal to, объединённые через AND, читаются громоздко. Чище — Switch с маршрутом для каждого значения и Fallback Output для остальных.

Типичные ошибки

  • Сравнил строку с числом и получил неожиданный результат → проверь тип, при необходимости приведи через Number(...) или String(...).
  • Забыл, что Equal To точное совпадение, и удивляешься, что "paid " (с пробелом в конце) не равно "paid" → подчисти левую часть: {{ $json.status.trim() }}.
  • В правой части написал выражение без режима Expression (без = в начале значения или без переключения в режим выражения) → n8n воспримет {{ $json.field }} как буквальный текст и условие никогда не сработает.
  • Сравниваешь с null через is equal to → у null/undefined своя специфика, для них правильнее is empty или does not exist, а не equals.
  • Использовал is not equal to на поле, которое может отсутствовать, и item с отсутствующим полем неожиданно ушёл в ветку «не равно» — добавь проверку exists до этого условия (см. callout выше).

Универсальный процесс добавления credential

Credential в n8n — это сохранённая «учётка» для подключения к сервису (API-ключ, OAuth-токен, логин/пароль и т.д.). Любая нода, которой нужна авторизация, выбирает credential из общего списка — поэтому добавлять его нужно один раз, а использовать можно из любого количества нод того же типа.

Пошагово

  1. В левом меню n8n открой раздел Credentials.
  1. Нажми кнопку Add credential (правый верхний угол).
  1. В поиске введи название нужного типа credential — например, OpenAI, Header Auth, Postgres, Google — и выбери его из списка.
  1. В шапке формы дай credential понятное имя — например, OpenAI школа или Postgres prod. По этому имени ты будешь выбирать его в нодах.
  1. Заполни поля формы. Какие именно — зависит от типа: для API-ключевых это обычно одно поле API Key, для Header Auth — Name и Value заголовка, для Postgres — host/port/database/user/password, для OAuth — запуск OAuth-flow по кнопке.

Append n8n Attribution: подпись в конце автоматических сообщений

Параметр «Append n8n Attribution» управляет автоматической подписью, которую n8n добавляет в конец каждого отправленного сообщения. По умолчанию он включён — и к тексту любого сообщения дописывается фраза вида «This message was sent automatically with n8n».

Зачем это нужно знать

Для личных экспериментов и отладки подпись не мешает. Но когда автоматическое сообщение идёт реальным пользователям или сотрудникам — постороннее упоминание сервиса автоматизации выглядит непрофессионально. Получатель видит не уведомление от сервиса, а артефакт настройки системы.

Как это работает

Параметр находится в разделе Additional Fields ноды отправки сообщений — он скрыт по умолчанию, чтобы не засорять основной экран настроек. Чтобы его найти:

  1. Открой параметры ноды.
  2. Прокрути вниз до раздела «Additional Fields» и раскрой его.
  3. Найди параметр «Append n8n Attribution» — переключи в положение Off.

После отключения сообщения отправляются без каких-либо лишних дополнений к тексту.

К примеру
Где это встречается

К примеру, нода Telegram отправляет оператору-человеку уведомление о новом запросе от клиента. Иногда Attribution заметен и в Email-нодах, когда письмо с автоматической подпиской уходит клиентам — это вредит доверию к бренду.

Типичные ошибки

  • Не нашли параметр — он спрятан в Additional Fields и не видён на основном экране. Нужно явно раскрыть блок.
  • Оставили включённым для пользовательских сообщений — получатели видят рекламу сервиса вместо нейтрального уведомления.
  • Перепутали с Parse Mode — Attribution и Parse Mode находятся в одном разделе Additional Fields, но это разные настройки. Parse Mode управляет форматированием (Markdown/HTML), Attribution — подписью.

Когда не подходит отключать

Если вы намеренно хотите показать, что сообщение отправлено автоматически — Attribution можно оставить включённым. Это может быть уместно в тестовых рассылках внутри команды, где прозрачность источника важна.

12Добавь HTTP Request с Test URL вебхука▾

Вторая нода тестера — HTTP Request. Она отправит POST на Test URL твоего вебхука с тестовым JSON-телом.

Test URL вебхука работает только когда нода Webhook в основном воркфлоу в режиме «слушает» — то есть после нажатия Listen for test event. Боевой Production URL работает только когда воркфлоу опубликован (Active), но для разработки этого не нужно.

12.1
Добавь ноду HTTP Request

Справа от Manual Trigger нажми + → в поиске нод введи http request → выбери HTTP Request. Нода появится на canvas и автоматически подключится к Manual Trigger.

12.2

12.3

Переключись на вкладку с M1-05 Application Hub. Открой ноду Webhook двойным кликом. Сверху есть два URL — Test URL и Production URL. Скопируй Test URL (иконка копирования рядом).

Test URL выглядит примерно так: https://n8n.school.vibe-business.site/webhook-test/m1-05-application.

12.4

12.5
Включи Send Body

Появится секция настройки тела запроса.

12.6

13Финальный тест: связка Tester → Hub через Test URL▾

Оба воркфлоу собраны. Финальный тест цепочки А работает попеременно в обеих вкладках: сначала в основном M1-05 Application Hub включаем приём (Listen for test event на ноде Webhook), потом в тестовом M1-05 Application Tester нажимаем Execute workflow.

Цепочка Б основного воркфлоу через тестер не запускается — её проверяем отдельно, нажимая Execute Step прямо на ноде Schedule Trigger в основном воркфлоу (как делали в шагах 8.5 и 10.8).

13.1
Включи Listen for test event на Webhook основного воркфлоу

Переключись на вкладку с воркфлоу M1-05 Application Hub. Открой ноду Webhook двойным кликом. Нажми кнопку Listen for test event (или Listen for Test Event — в зависимости от версии UI). Нода переходит в режим ожидания и начинает слушать Test URL.

13.2
В тестовом воркфлоу нажми Execute workflow

Переключись на вкладку с M1-05 Application Tester → внизу canvas нажми кнопку Execute workflow.

Manual Trigger отработает, передаст данные в HTTP Request, та отправит POST-запрос на Test URL основного воркфлоу.

13.3
Проверь результат в тестовом воркфлоу

В тестовом M1-05 Application Tester открой ноду HTTP Request.

В Output должен появиться ответ от Webhook — обычная строка OK (потому что JSON был валиден: name=Иван, email без test). Статус 200.

Если в JSON-теле HTTP Request поменять имя на пустое ("") или подложить email с test — ответ будет Rejected.

13.4
Проверь результат в основном воркфлоу

Переключись на вкладку с воркфлоу M1-05 Application Hub.

Открой ноду Webhook двойным кликом. В Output теперь лежит JSON принятого запроса. Внутри объекта body есть поля name со значением Иван и email со значением ivan@gmail.com — это содержимое POST-запроса, которое прислала нода HTTP Request из тестера.

Ноды If, Data Table Insert, Respond to Webhook (OK) на canvas тоже загорелись зелёной обводкой. В таблице m1-05-applications через UI появилась новая строка с этими данными.

Ноды цепочки Б (Schedule Trigger, Data Table Get, Filter, Telegram) при этом не загорались — у них свой триггер, который сейчас не сработал.

13.5
Проверь цепочку Б отдельно — Execute Step на Schedule Trigger

Прямо в M1-05 Application Hub найди ноду Schedule Trigger → нажми Execute Step. Сработает только цепочка Б: Schedule Trigger → Data Table Get → Filter → Telegram. В чате с @profy_conveyor_bot придёт сводка с числом необработанных заявок. Цепочка А (с Webhook сверху) при этом не загорается.

Прогнал — значит, оба триггера работают независимо. Это подтверждает паттерн «два триггера на одной канве».

Все, воркфлоу собран. Можешь потестировать его на разных JSON-телах из тест-кейсов, а потом выполнить Stretch (задачу со звездочкой).

🚀Stretch▾

Прогони цепочку Б ещё раз и внимательно посмотри на результат.

Сейчас в продукте есть две проблемы, которые на тестовых данных могут быть незаметны.

Первая проблема: заявки не меняют статус.

Цепочка А записывает новые заявки со status=new, цепочка Б берёт все строки со status=new для отчёта, но после отправки сводки никто не меняет статус на done.

Из-за этого завтра в отчёт попадут вчерашние заявки плюс новые. Послезавтра — уже три дня сразу.

Чтобы исправить это, после ноды Telegram добавь ноду Data Table в режиме Update Row(s).

Она должна пройтись по заявкам, которые попали в отчёт, и поставить каждой строке status=done.

После этого следующий запуск цепочки Б уже не будет повторно отчитываться по тем же заявкам.

Вторая проблема: если новых заявок нет, цепочка Б молчит.

Когда Filter возвращает пустой список, n8n не запускает следующую ноду. Поэтому Telegram-сообщение не уходит вообще.

Для владельца это выглядит неоднозначно: то ли новых заявок нет, то ли воркфлоу не сработал.

Чтобы исправить это, проверку нужно поставить до Filter, пока у нас ещё есть полная выборка из Data Table.

После Data Table Get добавь If, который проверяет, есть ли в таблице строки со status=new.

Если такие строки есть — отправь обычную сводку с количеством и списком имён, а затем пометь эти строки как done.

Если таких строк нет — отправь короткое сообщение:

За сутки новых заявок нет, всё разобрано.

После доработки прогони цепочку Б два раза подряд через Execute Step на Schedule Trigger.

Первый запуск должен отправить сводку и поменять статус найденных заявок с new на done.

Второй запуск должен отправить сообщение, что новых заявок нет.

Если оба запуска отработали так, значит цепочка Б теперь работает полноценно: прочитала заявки, отправила отчёт, пометила обработанные и сообщила даже о пустом результате.
