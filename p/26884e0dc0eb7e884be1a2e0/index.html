M-HttpV2-11 Микро-CRM в Telegram (JSONBin)

Представь: ты сделал клиенту простого Telegram-бота — закрытую микро-CRM для маленькой студии маникюра. Записал клиента — «создай Анна Смирнова». Хочешь посмотреть карточку — «список #BIN_ID». Поменять имя — «обнови #BIN_ID Анна Иванова». Удалить — «удали #BIN_ID». Все четыре операции — в одном чате, без отдельных приложений и формочек, а данные реально сохраняются между сообщениями на внешнем сервере JSONBin (не моковый сервер-имитатор как в прошлой задаче — настоящий, в котором всё что записал, останется лежать). Новое здесь — все четыре HTTP-метода CRUD на реальном хранилище, Header Auth credential для API-ключа X-Master-Key, path parameter в URL через expression и нода Split Out для расщепления массива записей.

Telegram Trigger ловит команду, Switch по первому слову (создай / список / обнови / удали) направляет item в одну из четырёх веток, в каждой HTTP Request со своим методом (POST / GET / PUT / DELETE) к JSONBin через Header Auth credential, в ветке списка дополнительно Split Out расщепляет массив записей, в ветках обновления и удаления Edit Fields парсит BIN_ID и текст из команды регулярками, fallback-ветка отвечает подсказкой по командам.

Что собираем

Telegram Trigger получает сообщение → Switch определяет команду по первому слову → данные попадают в одну из четырёх веток → нужная ветка выполняет запрос к JSONBin.

В этой задаче познакомимся сразу с несколькими новыми приёмами.

Header Auth credential позволяет хранить API-ключ отдельно от воркфлоу. JSONBin ожидает ключ в заголовке X-Master-Key, а n8n будет автоматически подставлять его в каждый запрос.

Также используем динамические URL через expression. Например, BIN_ID будет подставляться прямо в адрес запроса во время выполнения.

Ещё одна новая нода — Split Out. Она умеет разбивать массив на отдельные items, чтобы следующие ноды могли обработать каждый элемент по отдельности.

1Получи токен у @BotFather▾

Бот в Telegram создаётся не в n8n и вообще не в браузере. Telegram сделал для этого специального служебного бота — @BotFather. Через него любой пользователь Telegram создаёт собственного бота и получает на него токен — длинную строку, которая даёт полный доступ к этому боту. Этот токен мы потом вставим в n8n, и все ноды Telegram через него будут говорить от имени твоего бота.

Этот шаг ты делаешь в Telegram-приложении, n8n пока даже не открывай.

@BotFather: как создать собственного бота в Telegram

@BotFather — это официальный бот Telegram, через которого создаются все остальные боты. Других способов завести бот нет: ни сайта, ни приложения, ни консоли — только переписка с этим аккаунтом.

Зачем это нужно

Чтобы любой workflow (в n8n, Make, на собственном бэкенде) мог слушать сообщения и отвечать от имени бота, ему нужен токен Bot API. Токен выдаётся при создании бота через @BotFather. Без токена ни одна интеграция с Telegram не заведётся: токен — это пара «логин-пароль» бота при общении с Telegram Bot API.

Как это работает

Открываешь Telegram, в поиске находишь аккаунт с username BotFather (галка подлинности рядом с именем — синяя), и пишешь ему как обычному собеседнику. Команды начинаются со слэша:

  • /newbot — создать нового бота. @BotFather спросит две вещи:
  • отображаемое имя (любое, может быть с пробелами и кириллицей) — то, что увидят пользователи в шапке чата;
  • username (только латиница, цифры, подчёркивание; обязательно заканчивается на _bot или bot) — постоянный идентификатор, по которому бот ищется и не меняется.
  • /mybots — список всех твоих ботов, доступ к настройкам (имя, описание, аватарка, команды, токен).
  • /revoke — отозвать старый токен и получить новый (если старый утёк).

После /newbot приходит сообщение с токеном — длинная строка вида 123456789:ABCdefGHI_jklMNO.... Это и есть тот самый Bot API token: его копируешь и сохраняешь в безопасное место (менеджер паролей, заметки с шифрованием) — но не в публичный репозиторий и не в чат.

Типичные ошибки

  • Потерять токен — @BotFather показывает токен один раз; чтобы получить его снова, нужно открыть /mybots → <твой бот> → API Token. Если не нашёл — можно сгенерировать новый через /revoke, но тогда старый перестанет работать во всех уже настроенных интеграциях.
  • Скинуть токен скриншотом / в открытый чат — токен даёт полный контроль над ботом. Любой, кто его увидит, может слать сообщения от имени бота и читать всё, что ему приходит. Если утёк — сразу /revoke.
  • Username без _bot на конце — @BotFather отклонит регистрацию; правило жёсткое, придётся подбирать ещё.
  • Перепутали username и отображаемое имя — отображаемое имя меняется в любой момент через /setname, а username меняется только пересозданием бота. Если задумывался серьёзный бот — username стоит выбрать осознанно.

Когда не подходит

Через @BotFather нельзя создать юзербот (бот, который ходит под обычным аккаунтом) — это уже не Bot API, а MTProto, и токен оттуда не подходит. Также @BotFather не нужен, если бот уже создан кем-то другим и тебе просто передали готовый токен — тогда сразу заводишь credential в нужном сервисе и работаешь с ним.

Что за сервис

jsonbin.io — облачное key-value хранилище для JSON-объектов с REST API. Создаёшь bin (запись), кладёшь туда JSON любой формы, получаешь ID и читаешь / обновляешь по этому ID. Используется как лёгкая база для прототипов, фид-данных, конфигов ботов — там, где разворачивать настоящую БД лень.

Терминология

Базовые понятия JSONBin
ТерминЧто это

Что умеет

  • POST /v3/b — создать новый bin. В ответ — metadata.id, его сохраняешь себе.
  • GET /v3/b/{id}/latest — прочитать последнюю версию bin'а.
  • PUT /v3/b/{id} — перезаписать содержимое bin'а новым JSON.
  • DELETE /v3/b/{id} — удалить bin насовсем.

Free tier ограничен по запросам и размеру

Header Auth — универсальный credential для API с ключом в заголовке

Многие API-провайдеры (oauth-эндпоинты Сбера, простые сервисы с API-key, custom-аутентификация) ожидают, что ты пришлёшь авторизацию в HTTP-заголовке — например, Authorization: Basic ... или X-Api-Key: .... В n8n для этого есть универсальный тип credential — Header Auth. Он работает в любой ноде HTTP Request.

Это generic-credential: один credential подходит для десятков сервисов. Для каждого нового API не нужно создавать отдельный кастомный тип — достаточно знать, какой заголовок ждёт сервер и какое значение в него положить.

Пошагово

  1. В n8n открой Credentials (меню слева) → Add credential → в поиске введи Header Auth → выбери его.
  1. Заполни два поля:
  • Name — имя заголовка (например, Authorization или X-Api-Key). Просто слово, без двоеточия и значения.
  • Value — значение заголовка целиком, включая префикс если он нужен (например, Basic MGJj...== или Bearer eyJhbGc...).
  1. В шапке credential задай понятное имя — например, <Имя сервиса> OAuth или <Имя сервиса> API. Это имя ты будешь видеть в выпадашке HTTP Request ноды.
  1. Нажми Save. Готово — credential можно подключать к HTTP Request ноде.

Как использовать в HTTP Request

В ноде HTTP Request → раздел Authentication → Predefined Credential Type → в списке Generic Auth Type выбери Header Auth → ниже выпадашка с твоим credential.

Не забудь префикс Basic / Bearer
Самый частый источник 401

Если API использует OAuth2 — в Value пиши Basic <ключ> или Bearer <токен> целиком, со словом и пробелом. Только сам ключ без префикса — частая причина 401. Проверь заголовок curl-ом сначала, потом перенеси в credential.

Когда Header Auth не подойдёт

Если сервер ждёт значение в query-параметре (?api_key=...), а не в заголовке — Header Auth не сработает. Используй тип Query Auth (тоже generic) или подставляй ключ вручную через выражение в URL ноды.

Если в заголовке нужно динамическое значение (например, Authorization: Bearer <живой_access_token>) — credential не подходит вообще: значение в credential статическое. Пиши заголовок выражением прямо в HTTP Request: Authorization = ={{ "Bearer " + $('Get Token').item.json.access_token }}.

Что такое HTTP-заголовок и зачем api-key передаётся в заголовке

Когда браузер или программа обращается к API-сервису, запрос состоит из двух частей: тела (данные, которые ты отправляешь) и заголовков (metadata о запросе: кто ты, в каком формате данные, как тебя авторизовать).

HTTP-заголовок — это пара «ключ: значение», которая идёт в начале каждого запроса. К примеру:

Content-Type: application/json
api-key: abc123xyz

Заголовки невидимы в URL, пользователь их не видит. Сервер читает заголовки, чтобы понять, как обработать запрос.

Зачем это нужно

Многие API требуют передавать ключ доступа именно в заголовке — это удобно и безопасно:

  • Ключ не попадает в URL (не сохраняется в истории браузера, не видна в логах сервера)
  • Тело запроса остаётся для данных — заголовок для технических параметров
  • Разные заголовки — разные уровни доступа (например, Authorization для OAuth, api-key для простых ключей)

Разные сервисы используют разные имена заголовка: Authorization, api-key, X-Api-Key, x-access-token. Название заголовка — это просто договорённость между клиентом и сервером, зафиксированная в документации API.

Как это работает в curl

В командной строке флаг -H добавляет заголовок к запросу:

curl https://api.example.com/data \
  -H "api-key: твой_ключ_здесь" \
  -H "Content-Type: application/json"

Каждый -H — один заголовок. Можно добавить сколько угодно. Флаг -H работает одинаково в curl на macOS, Linux и Windows.

К примеру
Заголовок vs параметр URL

К примеру, можно передать ключ и через URL: ?api_key=abc123. Иногда API такое поддерживают — но это менее безопасно: URL записывается в логи серверов, браузерную историю, попадает в аналитику. Заголовок надёжнее — он не оседает нигде в открытом виде.

Типичные ошибки

  • Пропустить кавычки — если в значении ключа есть пробел или спецсимволы, curl может прочитать его неправильно. Всегда бери значение заголовка в кавычки: -H "api-key: значение".
  • Неправильное имя заголовка — разные сервисы ожидают разные имена: api-key, X-Api-Key, Authorization: Bearer .... Смотри документацию конкретного API, имя должно совпадать точно.
  • Передать ключ в теле вместо заголовка — API вернёт 401 (не авторизован), потому что смотрит только в заголовок. Тело и заголовок — разные места в запросе.

Когда не подходит

Некоторые сервисы используют OAuth 2.0 или другие схемы авторизации вместо простого ключа в заголовке. Там схема сложнее: сначала нужно получить токен, потом передавать его в каждом запросе. Простой api-key в заголовке — это паттерн для сервисов с базовой авторизацией по ключу.

Сначала проверь в Headers, потом переноси в credential

Когда получаешь 401 и не уверен, правильно ли написан заголовок в credential — не угадывай. Добавь заголовок прямо в ноду через раздел Send Headers, проверь что запрос проходит, и только потом перенеси проверенные значения в credential.

Шаг 1 — добавь заголовок напрямую в ноду

Открой ноду HTTP Request. В разделе Authentication временно выставь None — иначе credential и ручной заголовок могут конфликтовать.

Раскрой раздел Headers → включи Send Headers → нажми Add Parameter. Появится строка с двумя полями:

Параметры заголовка
ПолеЧто вписать
NameИмя заголовка — например, Authorization
ValueЗначение целиком — например, Bearer eyJhbGc...

Пробела перед словом Bearer быть не должно. После токена — тоже ничего лишнего.

Шаг 2 — выполни запрос и убедись что 200

Нажми Execute step на этой ноде. Если ответ 200 — заголовок написан правильно. Если снова 401 — ошибка в самом токене или его формате, не в credential.

Шаг 3 — скопируй значения и перенеси в credential

Пока нода открыта, скопируй:

  • точное значение поля Name (например, Authorization)
  • точное значение поля Value (например, Bearer eyJhbGc...)

Потом:

  1. Удали строку из Headers (или выключи Send Headers).
  2. В Authentication верни обратно Predefined Credential Type → Header Auth.
  3. Открой или создай credential, вставь скопированные значения — Name и Value один в один.
Частая ошибка
Лишний пробел или перенос строки в Value

При вставке токена из другого источника (Telegram, браузер, PDF) в конец строки иногда попадает невидимый пробел или перенос строки. Credential сохранится без ошибки, но каждый запрос будет возвращать 401. Проверь: в поле Value не должно быть ничего после токена.

Универсальный процесс добавления credential

Credential в n8n — это сохранённая «учётка» для подключения к сервису (API-ключ, OAuth-токен, логин/пароль и т.д.). Любая нода, которой нужна авторизация, выбирает credential из общего списка — поэтому добавлять его нужно один раз, а использовать можно из любого количества нод того же типа.

Пошагово

  1. В левом меню n8n открой раздел Credentials.
  1. Нажми кнопку Add credential (правый верхний угол).
  1. В поиске введи название нужного типа credential — например, OpenAI, Header Auth, Postgres, Google — и выбери его из списка.
  1. В шапке формы дай credential понятное имя — например, OpenAI школа или Postgres prod. По этому имени ты будешь выбирать его в нодах.
  1. Заполни поля формы. Какие именно — зависит от типа: для API-ключевых это обычно одно поле API Key, для Header Auth — Name и Value заголовка, для Postgres — host/port/database/user/password, для OAuth — запуск OAuth-flow по кнопке.
  1. Нажми Save. Готово — credential можно подключать к любой ноде, которая поддерживает этот тип.
Подсказка
Один credential — много нод

Один сохранённый credential можно использовать в неограниченном количестве нод одного типа. Если ключ протухнет — поменяй его в credential один раз, обновится во всех нодах сразу.

⚙Типы update'ов в Telegram Trigger: что значит Message, Callback Query, Inline Queryстатья · ~2 мин▾

Типы update'ов в Telegram Trigger

Telegram Bot API устроен так: всё, что происходит с ботом (новое сообщение, нажатие кнопки, отредактированный текст, входящий платёж), приходит как объект Update. У каждого Update ровно один тип — и в настройках ноды Telegram Trigger нужно явно отметить галочками те типы, которые workflow готов обрабатывать.

Зачем это нужно

Если оставить все типы по умолчанию, workflow будет срабатывать на каждое движение в боте — и придётся в каждой следующей ноде разбираться, что именно прилетело. Если, наоборот, отметить только нужные — Telegram пришлёт только их, остальное молча выбросит на своей стороне. Это сужает поверхность ошибок и упрощает логику: в $json гарантированно лежит ожидаемая структура.

Как это работает

В параметрах Telegram Trigger раздел Updates — это список галочек. Самые ходовые типы:

  • Message — обычное входящее сообщение в чате (текст, фото, голосовое, стикер, документ). В $json появляется поле message с подполями from, chat, text, date и опциональными photo / voice / sticker. Это основной тип для классических «спроси-ответь» ботов.
  • Edited Message — пользователь нажал «изменить» на ранее отправленном сообщении. Приходит почти та же структура, что у Message, но в поле edited_message. Большинство ботов игнорируют этот тип, иначе пришлось бы заново гонять отредактированное сообщение через всю логику.
  • Callback Query — нажатие на inline-кнопку под сообщением бота. В $json поле callback_query с полем data (короткая строка, которую бот сам подложил в кнопку при отправке). Используется, когда у бота есть кнопки выбора — Yes/No, варианты ответа, навигация по меню.
  • Inline Query — пользователь начал набирать имя бота в любом чате (@my_bot ...). Бот должен почти мгновенно вернуть варианты, которые подставятся в качестве сообщения. Сценарий — когда бот работает как поисковая подсказка во всех чатах, а не как собеседник.
  • Channel Post / Edited Channel Post — сообщения и редактирования в канале, где бот добавлен админом. Нужны только для ботов-модераторов каналов.

Менее частые типы (Shipping Query, Pre Checkout Query, Poll Answer, My Chat Member) нужны узким сценариям — платежи, опросы, отслеживание добавления/удаления бота из чатов.

Типичные ошибки

  • Отметили всё подряд «на всякий случай» — workflow получает обновления, которых не ожидает (отредактированные сообщения, callback'и, channel post'ы), ноды дальше падают на отсутствующих полях.
  • Отметили только Message, ждут нажатий на inline-кнопки — Callback Query тип отдельный; без галочки бот молча игнорирует все нажатия.
  • Перепутали Message и Edited Message — в Message приходит $json.message.text, в Edited Message — $json.edited_message.text. Один и тот же код через $json.message.text на edited-сообщении вернёт undefined.

Когда не подходит

Если бот совсем простой и обрабатывает только текст в личке — отметь только Message и не усложняй. Тонкие сценарии (inline-режим, платежи, модерация каналов) добавляются позже, когда становится ясно, что без них не обойтись.

Запустить одну ноду, не дёргая весь workflow

Execute step — это кнопка внутри панели ноды (открывается двойным кликом по ноде). По смыслу — младший брат большой кнопки Execute workflow внизу canvas: та запускает весь воркфлоу с начала, эта — только текущую ноду. Удобно, когда настраиваешь параметры и хочешь сразу видеть, что отдаёт нода, не перезапуская всю цепочку.

Как это работает

Когда ты жмёшь Execute step, n8n берёт уже существующие данные на входе ноды (output предыдущей ноды из последнего прогона) и прогоняет через эту ноду. Сама нода выполняется реально: HTTP-запрос уходит, запись в базу пишется, письмо отправляется. Это не «сухой прогон», а полноценное выполнение — только одного шага.

  • key: Execute workflow

value: Запускает весь воркфлоу от триггера. Нужен полный прогон с самого начала. Каждый раз заново дёргает все API.

  • key: Execute step

value: Запускает только эту ноду. Берёт вход из последнего output предыдущей ноды. Идеально для подбора параметров и отладки.

  • key: Когда работает

value: Только если на входе ноды есть данные с прошлого прогона. Если ни разу не запускал предыдущие — кнопка либо ничего не вернёт, либо упадёт «No input data».

Типичный сценарий: запустил весь воркфлоу один раз (Execute workflow), увидел что нода HTTP Request отдаёт не то, поправил поле URL, нажал Execute step — увидел новый результат за секунду. Предыдущие ноды повторно не выполнялись.

У триггеров кнопка называется иначе
Смысл тот же — запустить один шаг и получить тестовое событие

У некоторых триггерных нод n8n кнопка Execute step подписана по-другому, потому что у триггера нет «входа» — он сам должен сходить за событием. Например, у RSS Trigger она называется Fetch Test Event: клик подтянет последние записи из ленты и положит их на выход ноды как тестовый input для остального воркфлоу. Работает идентично Execute step обычной ноды — точно так же используется для отладки и подбора параметров без полного прогона.

Важно про побочные эффекты
Каждый Execute step — это реальный вызов

Нода с записью в базу или отправкой сообщения каждое нажатие Execute step делает реально: 5 нажатий = 5 записей / 5 писем. Для отладки таких нод используй тестовые данные или подменяй credentials.

Типичные ошибки

  • Жать Execute step на ноде, у которой ещё не было входных данных → «No input data», нода не запустится. Сначала прогони весь воркфлоу через Execute workflow или хотя бы триггер вручную, чтобы у каждой ноды появился output.
  • Использовать Execute step для проверки изменений в предыдущих нодах → текущая нода работает со старым output, изменения не подхватятся. Если поправил предыдущую ноду — перезапусти её, а потом уже текущую.
  • Тестировать ноду с побочным эффектом (запись в БД, отправка письма) на проде → каждый клик создаёт реальную запись. Использовать тестовый credential или временно подменять параметры на безопасные.

📖Структура Telegram Update: update_id, message, chat и fromстатья · ~3 мин▾

Структура Telegram Update

Когда пользователь пишет боту, Telegram Bot API оборачивает событие в объект Update и доставляет его в workflow одной JSON-пачкой. Чтобы корректно отвечать, нужно понимать, какие поля внутри и где какие важные идентификаторы лежат — особенно chat.id, from.id и update_id.

Зачем это нужно

В Telegram-боте каждая нода после триггера так или иначе тянет данные из Update: текст сообщения для агента, id чата для ответа, id пользователя для разделения прав, дата для логов. Если перепутать поле — бот ответит не тому, не туда, потеряет память диалога или сломается. Понимание структуры — обязательная база для любой работы с Telegram Bot API.

Как это работает

Корень объекта Update устроен просто: одно поле update_id (уникальный счётчик апдейта на стороне Telegram) и ровно одно поле полезной нагрузки — message, edited_message, callback_query, inline_query и так далее, в зависимости от типа события. Какой именно тип пришёл — определяется тем, какое из этих полей заполнено.

Для самого частого случая (текстовое сообщение) полезная нагрузка лежит в message:

{
  "update_id": 871234567,
  "message": {
    "message_id": 42,
    "from": {
      "id": 12345678,
      "is_bot": false,
      "first_name": "Аня",
      "username": "anya_test",
      "language_code": "ru"
    },
    "chat": {
      "id": 12345678,
      "first_name": "Аня",
      "username": "anya_test",
      "type": "private"
    },
    "date": 1715600000,
    "text": "Привет, как дела?"
  }
}

Ключевые поля и за что они отвечают:

  • update_id — глобальный счётчик апдейтов для конкретного бота. Возрастает с каждым новым событием. Полезен для дебага и для предотвращения повторной обработки одного и того же апдейта; но как идентификатор диалога не подходит — у каждого нового сообщения он свой.
  • message.message_id — счётчик сообщений внутри одного чата. К нему обращаются, чтобы ответить «реплаем» или удалить сообщение.
  • message.from.id — id пользователя, который написал. Один и тот же у этого пользователя во всех его чатах с этим ботом и в любых групповых чатах. Нужен, когда логика бота завязана на «кто именно написал» (правá доступа, персональные настройки).
  • message.chat.id — id чата. В личке (chat.type === "private") совпадает с from.id, поэтому в личных ботах разница не видна. В групповом или канале — это id группы/канала, общий для всех её участников.
  • message.text — текст сообщения. Только у обычных текстовых сообщений; у голосовых, фото, стикеров его нет вовсе, вместо него — voice, photo, sticker.
  • message.date — момент отправки, Unix-timestamp в секундах.

Почему chat.id, а не from.id для памяти диалога

Память диалога (chat memory у AI Agent, сессии в RAG) обычно ключуется по «комнате разговора», а не по пользователю. В личке бота это всё равно одно и то же. Но в групповом чате — один chat.id на всех участников, и десять разных from.id. Если ключевать память по from.id, у каждого участника группы будет своя память и бот будет «забывать», что только что обсуждалось общим разговором. Если по chat.id — память общая на весь чат, как у живого собеседника. Поэтому стандартный выбор sessionId для бота — message.chat.id.

Типичные ошибки

  • Перепутали from.id и chat.id — в личке выглядит одинаково, в группе — расходится; бот в группе ведёт себя странно.
  • Использовали update_id как идентификатор сессии — у каждого нового сообщения свой update_id, и память диалога никогда не находит предыдущие сообщения.
  • Полезли в message.text у голосового сообщения — поле физически отсутствует, выражение возвращает undefined, дальше всё ломается. Решение — фильтровать по наличию text ещё до основной логики.
  • Решили читать сообщения в группе и не учли тип chat — у группового чата chat.type === "group" или "supergroup", и поведение бота может отличаться (например, бот в группе видит не все сообщения, а только те, что отмечены через @-mention или ответ; для полного доступа нужно отключать privacy mode у @BotFather).

Когда не подходит

Эта структура — Update.message для обычных сообщений. Для нажатий на inline-кнопки структура другая: верхний уровень — Update.callback_query, текст сообщения в нём недоступен напрямую, вместо text есть data (строка, которую бот сам положил в кнопку при отправке). Для платежей, опросов, статуса участника — свои поля. Поэтому прежде чем парсить, всегда проверь, какой именно тип события прилетел.

📖Pin Data в n8n: как зафиксировать тестовый вход нодыстатья · ~2 мин▾

Pin Data в n8n: как зафиксировать тестовый вход ноды

Pin Data — это способ «прибить» к ноде заранее заготовленные данные, которые n8n будет подставлять вместо настоящего входа на каждом тестовом запуске. Полезно, когда дёргать реальный источник (внешний воркфлоу, веб-хук, API) на каждое нажатие Execute Workflow неудобно или невозможно.

Зачем это нужно

Часто отлаживается не сам триггер, а ноды после него — логика, маппинг, запрос в базу. Чтобы не звать каждый раз источник данных (другой воркфлоу, бот, форму), удобнее один раз получить пример настоящего входа, зафиксировать его на триггере и потом крутить запуски сколько нужно.

Поэтому Pin Data — это инструмент разработчика на этапе сборки, не часть рабочего воркфлоу. В боевом режиме закреплённые данные перестают использоваться: n8n берёт реальный вход.

Как это работает

К любой ноде с выходом можно прикрепить «запинённый» JSON. После этого:

  • При запуске воркфлоу через UI (Execute Workflow, тестовый прогон) n8n не вызывает саму ноду, а сразу выдаёт следующим нодам зафиксированные данные.
  • Pin Data привязана к конкретной ноде, не к workflow целиком — можно запинить триггер и оставить остальные ноды живыми.
  • На canvas у запинённой ноды появляется визуальный значок (булавка/иконка), чтобы не забыть, что данные «искусственные».

Удобнее всего пинить триггер: на нём поле выхода — то, что в боевом режиме приходит снаружи (Telegram-апдейт, тело веб-хука, поля Execute Workflow Trigger). Тестовый JSON собирают с теми же ключами, что отдаёт реальный источник.

К примеру
Где это встречается

К примеру, в Sub-Workflow с триггером Execute Workflow Trigger пин на триггере позволяет запускать сам Sub-WF кнопкой сверху и отлаживать ноды после триггера, не запуская вызывающий workflow. Иногда — пин ставят на ноде HTTP Request к платному API, чтобы при разработке не жечь квоту запросами «ещё разок проверить парсинг».

Типичные ошибки

  • Забыли убрать пин перед боевым запуском — workflow пошёл в production, но данные ноды всегда одни и те же; реальные входы игнорируются. Перед публикацией пины снимают.
  • Структура запинённого JSON не совпадает с реальным — отладка проходит идеально, но в проде ломается на полях, которых не было в тестовом примере; делайте пин из реальной execution, а не «на глаз».
  • Запинили середину воркфлоу, забыли — переименовали поле в ноде до пина, а пин остался со старой структурой. Следующие ноды читают «фантомные» данные.
  • Pin Data перепутали с переменными среды — пин не передаётся между воркфлоу и не сохраняется при экспорте на другую инсталляцию n8n как «конфиг»; это именно отладочный фиксаж.

Когда не подходит

Pin Data не заменяет настоящие тесты — это удобство в редакторе, не автоматизация. Если нужна повторяемая проверка с разными входами, лучше делать отдельный тестовый workflow или мини-набор тест-кейсов; пин подходит, когда хочется быстро прогнать «один и тот же» сценарий N раз.

Нода Switch делит поток на несколько веток сразу

Switch — это IF для случаев, когда веток больше двух. Каждый элемент входных данных проверяется набором правил, и в зависимости от того, какое правило сработало, элемент уходит в свой выход. У ноды столько выходов, сколько правил ты создал, плюс опционально выход «ни одно правило не подошло».

Режим Rules — основной способ

В режиме Rules ты задаёшь список правил. Каждое правило — это:

::key-value{rows=[{key:"Conditions",value:"одно или несколько условий по тому же принципу что в IF — Left value, Operator, Right value. Несколько условий внутри одного правила объединяются через AND или OR."},{key:"Rename Output",value:"переключатель: если включить — появится поле Output Name, и выход этого правила получит свой ярлык вместо номера (например, urgent или vip)."}] id=switch-rule-anatomy}

Кнопка Add Routing Rule добавляет новое правило — и автоматически появляется новый выход на canvas. Удалишь правило — выход исчезнет.

По умолчанию элемент уходит в первое подошедшее правило и дальше не проверяется. Если ни одно правило не подошло — элемент отбрасывается.

Опции — fallback и all matching outputs

В разделе Options (открывается через кнопку Add Option внизу формы) есть две важные настройки:

::key-value{rows=[{key:"Fallback Output",value:"что делать с элементами, которые не подошли ни под одно правило. None — отбросить (по умолчанию), Extra Output — добавить отдельный выход для них."},{key:"Send data to all matching outputs",value:"если включить — элемент уходит во ВСЕ правила, чьи условия совпали, а не только в первое. Полезно, когда ветки независимы и не исключают друг друга."}] id=switch-options}

Режим Expression — для динамического числа веток

В режиме Expression ты сам говоришь, сколько выходов нужно (Number of Outputs), и пишешь выражение, которое возвращает индекс выхода (0, 1, 2...). Подходит, когда ветка вычисляется формулой, а не цепочкой условий.

={{ $json.priority === 'high' ? 0 : $json.priority === 'low' ? 2 : 1 }}
Когда это пригодится
Один источник — много направлений обработки

Например, входящие заявки делятся на «срочные», «обычные» и «спам» — у каждого свой набор нод дальше. Без Switch пришлось бы ставить три IF подряд и тянуть провода в обход, либо повторять одни и те же условия в каждой ветке.

Типичные ошибки

  • Ждёшь, что элемент попадёт в несколько веток одновременно, но он уходит только в первую → включи опцию Send data to all matching outputs или поменяй порядок правил.
  • Ни одно правило не сработало, и элементы просто исчезают → включи Fallback Output со значением Extra Output, чтобы такие элементы шли в отдельную ветку и их было видно.
  • Правила пересекаются по условиям, элементы уходят не туда → пересмотри порядок: правила проверяются сверху вниз, переставь более специфичные выше общих.
  • В режиме Expression выражение вернуло число больше, чем Number of Outputs - 1 → элемент потеряется. Возвращай индекс с учётом нумерации с нуля.

Fixed и Expression — два режима любого поля

В правом верхнем углу почти каждого поля параметров ноды есть переключатель Fixed | Expression. Это два режима ввода значения: фиксированное (написал текст — этот текст и пойдёт) или формула на JavaScript, которая вычисляется заново при каждом выполнении ноды. Один и тот же тумблер есть у обычных текстовых полей, у dropdown-ов, у credential pickers и у resource locator полей (где выбираешь объект из списка — таблицу, документ, канал).

Как работает каждый режим

  • Fixed — обычный ввод. Что написал — то и попадёт в запрос/в подстановку. У resource locator (когда выбираешь объект — таблицу, документ, sheet) Fixed означает выбор из dropdown «From list» или ручной ввод id одним значением.
  • Expression — поле подсвечивается оранжевым, а внутри можно писать формулу в фигурных скобках: {{ $json.field }}, {{ $now.toISO() }} и т.п. Значение пересчитывается на каждой итерации, для каждого пришедшего элемента.
Когда переключать в Expression
Значение зависит от данных или меняется между запусками

Если значение должно зависеть от того, что пришло из предыдущей ноды (имя пользователя, id записи, текст сообщения), от текущего времени или от любой логики — переключай в Expression. Если значение всегда одно и то же (метод HTTP POST, имя таблицы, фиксированный slug) — оставляй Fixed.

Как переключить режим

Три способа сделать одно и то же:

  1. Клик по Fixed | Expression в правом верхнем углу поля. Самый явный — видно сразу, в каком режиме поле.
  2. Иконка шестерёнки слева от поля → пункт Add Expression. Удобно когда хочешь сразу открыть expression builder (окно с деревом нод слева и preview значения справа).
  3. Drag-and-drop значения из панели Input слева в поле. Это самый частый способ в учебных задачах: открываешь NDV предыдущей ноды через мини-карту слева или раскрываешь панель Input, цепляешь нужное поле мышкой и тянешь в параметр текущей ноды. n8n при отпускании сам переключает режим в Expression и подставляет правильную ссылку — {{ $json.field }} или {{ $('Имя ноды').item.json.field }}.

В обратную сторону — клик по Fixed в углу. Содержимое expression-режима очистится, поле станет обычным текстовым.

Тонкость
У resource locator в Expression нет dropdown

В resource locator полях (например, выбор Data table или Spreadsheet) при переключении в Expression список объектов из dropdown пропадает — теперь ты должен сам подставить нужный id или slug формулой. Полезно когда id берётся из предыдущей ноды; если нужно просто выбрать существующий объект мышкой — оставайся в Fixed.

Типичные ошибки

  • Забыл переключить в Expression, написал формулу как текст. Поле в Fixed-режиме воспринимает {{ $json.name }} как обычные символы и отправит эту строку буквально. Признак: значение в preview / в результате выполнения содержит {{ и }} как есть. Лечение: переключить в Expression вручную (тумблер в правом верхнем углу) или перетянуть значение из панели Input заново.
  • Стёр выбор из resource locator, переключив в Expression и обратно. Переключение режима очищает содержимое поля. Если в Fixed уже был выбран нужный объект из списка — после Expression → Fixed его придётся выбрать заново.

Что такое expression

В n8n expression — это маленький кусочек кода в фигурных скобках, который динамически подставляет значение в поле ноды. Записывается так:

={{ <выражение> }}

Знак равенства = в начале — это маркер для n8n: «трактуй содержимое этой ячейки не как обычную строку, а как выражение, которое нужно вычислить». Без = n8n запишет в поле литеральный текст {{ ... }} как есть.

Внутри {{ }} пишется JavaScript-выражение. Самое частое — чтение значения из другой ноды.

Зачем нужны expressions

Каждое поле в параметрах ноды может быть либо фиксированным значением (например, URL https://api.example.com), либо expression'ом — формулой, которая считается на лету в момент исполнения ноды. Expressions нужны, когда значение зависит от данных, пришедших из предыдущей ноды.

К примеру
Где это встречается

К примеру, ты сделал ноду «прочитать строку из таблицы конфига», а следующая нода — HTTP Request к внешнему API. В URL этой HTTP-ноды можно подставить значение из конфига: ={{ $('Get Config').item.json.api_url }} — n8n при каждом исполнении возьмёт актуальный URL из таблицы, а не зашитый в ноду.

Как читать значения из других нод

Базовая формула: ={{ $('<имя_ноды>').item.json.<поле> }}

Разбор по частям:

  • $('<имя_ноды>') — обращение к ноде по имени (то, что написано на canvas под нодой). Имя — в одинарных кавычках, точно как на canvas.
  • .item — текущий элемент, который обрабатывается. У большинства нод элементов несколько (массив), item берёт тот, что соответствует текущей итерации.
  • .json — содержимое поля json элемента (там лежат все данные, которые нода вернула).
  • .<поле> — конкретное поле внутри json. Если поле вложенное — пишется через точку: .config.embedding_model.

Пошагово — как вставить expression в поле

  1. Кликни на иконку шестерёнки рядом с полем, выбери Add Expression (или нажми Expression в верхнем правом углу поля).
  1. В появившемся окне (expression builder) — слева список нод воркфлоу, можно кликать по полям и они вставятся в формулу.
  1. Справа — preview: видно, какое значение получится прямо сейчас. Если красным — формула или путь к полю некорректные.
  2. Закрой окно, поле остаётся в режиме expression (синий фон + иконка fx).
Типичная ошибка
`=` в начале — обязателен

Если скопировал готовое выражение из учебной задачи и вставил без = в начале — n8n запишет в поле текст {{ ... }} как есть, нода уйдёт делать HTTP-запрос на URL вроде ={{ $('...').item.json.api_url }} буквально. Симптом — нода падает с ошибкой Invalid URL или возвращает 404 на странную строку.

Типичные ошибки

  • Имя ноды не совпадает. В формуле $('Get Config') имя должно точно совпадать с подписью ноды на canvas. Если переименовал ноду — старые expressions, ссылающиеся на неё, сломаются.
  • Путь к полю не существует. .item.json.embedding_dimensions упадёт с undefined, если у элемента такого поля нет. Препетий — посмотри preview в expression builder, он сразу показывает значение или ошибку.
  • Забыл = в начале. Самая частая опечатка — описано в callout выше.
  • Кавычки внутри {{ }}. Внутри expression строки оборачиваются одинарными кавычками: ={{ 'Hello ' + $json.name }}. Двойные конфликтуют с JSON.

Когда expression не нужен

Иногда поле должно быть фиксированным. Например, метод HTTP-запроса (GET / POST) или имя коллекции, которое одинаково для всего воркфлоу — нет смысла читать его динамически. Простое правило: если значение меняется от итерации к итерации или зависит от пришедших данных — нужен expression; если значение постоянное — оставляй фиксированным.

Drag-and-drop из панели Input — самый быстрый способ связать ноды

В n8n каждая нода видит данные предыдущих нод в панели Input слева от параметров. Чтобы подставить любое поле из этого Input в параметр текущей ноды, не нужно вручную писать expression — достаточно перетащить поле мышкой. n8n сам подставит правильное выражение со ссылкой на нужную ноду.

Как это работает

  1. Открой параметры ноды двойным кликом. Слева автоматически появится панель Input — это выходные данные предыдущей ноды (или одной из предыдущих, если их несколько).
  2. Разверни в Input структуру до нужного поля — она показывается как дерево вложенных объектов: message → chat → id, output → [0] → content → [0] → text и так далее.
  3. Перетащи нужное поле мышкой прямо в поле параметра справа. В поле появится выражение вида ={{ $('<имя предыдущей ноды>').item.json.<путь.к.полю> }}.
  4. Поле в правой панели окрасится синим — это значит, что параметр теперь не литерал, а expression: на каждом запуске значение будет браться из реальных данных.
К примеру
Откуда n8n берёт имя ноды

К примеру, если предыдущая нода называется Telegram Trigger, и ты перетащил из неё поле message.text — n8n запишет ={{ $('Telegram Trigger').item.json.message.text }}. Имя ноды берётся ровно такое, как написано на canvas. Иногда — если переименуешь ноду уже после drag-and-drop, выражение сломается, потому что в формуле останется старое имя.

Откуда выбрать источник данных

Если у ноды несколько входов или нужно дотянуться не до прямого предыдущего шага, а до более раннего — сверху панели Input есть переключатель источника. К примеру, в финальной ноде workflow можно вывести Input от первого триггера, чтобы достать оттуда поле, которого нет в выходе ближайшей предыдущей ноды.

Типичные ошибки

  • Перетащил не на то поле параметра. Drag-and-drop срабатывает на любом текстовом поле — легко уронить значение не в Text, а в Chat ID, и наоборот. После перетаскивания проверь, что в нужном поле появилось ожидаемое выражение.
  • Перетащил из объекта-обёртки, а не из конкретного поля. Если потянуть за message, а не за message.text — n8n подставит весь объект, в текстовое поле уйдёт [object Object] или JSON-строка. Раскрывай дерево до примитивного значения.
  • Сломался путь после переименования ноды. В выражении $('Telegram Trigger') имя ноды фиксированное. Если потом переименовать Telegram Trigger в что-то другое — нужно либо вернуть старое имя, либо переписать выражение вручную.

Когда лучше не использовать drag-and-drop

Если значение должно быть фиксированным (например, имя коллекции, метод HTTP, статичный текст приветствия) — drag-and-drop не нужен, просто впечатай значение в поле как обычный текст. Зачем переименовывать выходы Switch

По умолчанию у каждого правила в Switch выход подписан номером — 0, 1, 2. Это работает, но через неделю ты сам уже не помнишь, что было в правиле 1 — заявки на возврат или новые заказы. А когда workflow открывает другой человек — ему приходится открывать каждое правило и читать условие, чтобы понять, куда уходит ветка.

Rename Output — это переключатель внутри правила Switch. Если включить, появляется поле Output Name. Туда пишешь короткий ярлык — urgent, vip, refund, spam — и этот ярлык становится подписью выхода на canvas. На стрелке от Switch к следующей ноде будет видно осмысленное слово, а не цифру.

Когда это пригодится
Switch с 3+ ветками или ветки уходят далеко по canvas

Пока ветки две и обе видно одним взглядом — можно жить и с номерами. Как только правил три и больше, или ветка тянется через пол-холста к ноде далеко справа — без ярлыков начинаешь путаться, какая стрелка куда ведёт.

Как включить

  1. Открой ноду Switch, разверни правило (Routing Rule 1, Routing Rule 2...).
  2. Найди переключатель Rename Output под полями Conditions — включи его.
  3. В появившемся поле Output Name напиши ярлык: одно-два слова, латиницей, понятно по сути. Например, vip-client, needs-review, auto-approved.
  4. Повтори для каждого правила, которое хочешь подписать.

После сохранения на canvas рядом с каждой стрелкой от Switch появится твой ярлык.

Та же логика — для самих нод

Переименовывать стоит не только выходы Switch, но и сами ноды. По умолчанию n8n называет ноду по типу — Telegram, OpenAI, Edit Fields, HTTP Request. Если в workflow одна нода каждого типа — это нормально. Но как только их две (два HTTP Request к разным API, два Edit Fields для разных этапов) — ты теряешь ориентацию.

Чтобы переименовать ноду — кликни по её заголовку дважды и впиши осмысленное имя: Fetch user from CRM, Notify admin, Build summary prompt. Хорошее имя описывает что нода делает в контексте workflow, а не какого она типа.

Правило хорошего имени
Глагол + объект

Send + to admin, Fetch + user data, Build + prompt. Имя должно читаться как короткое описание действия, не как технический ярлык.

Зачем Fallback Output

Нода Switch раздаёт items по веткам в зависимости от правил: первый item, подошедший под правило 1, идёт в выход 1, под правило 2 — в выход 2 и так далее. А что делать с item'ом, который не подошёл ни под одно правило? По умолчанию Switch его молча отбрасывает — item исчезает из потока без следов. Fallback Output включает дополнительный выход «никуда не попало», куда сваливаются все «лишние» items.

Как включить

Включение Fallback Output
ШагДействие
1Открыть форму ноды Switch
2Раскрыть секцию Options внизу формы
3Переключить тумблер Fallback Output в положение on

После этого на канвасе у Switch появится ещё один выход — обычно с подписью extra или fallback. К нему подключается нода-обработчик «всего остального».

Когда нужен Fallback

Когда
Случаи где Fallback обязателен
  • Бот, отвечающий на команды. Правила Switch ловят /start, /help, /menu. Всё остальное — fallback на «Не понял команду, нажми /help».
  • Роутинг событий. Switch раскидывает события вебхука по типам. Незнакомый тип не должен потеряться — fallback пишет в лог «неизвестный тип события».
  • Категоризация. Switch разносит заявки по типам клиента. Неопределённый тип → fallback на ручную обработку оператором.

Что важно понимать

Fallback Output — это не «иначе» в условиях правил, а отдельный выход для items, вообще не подошедших ни под одно правило. Если хотя бы одно правило сработало, item ушёл туда — в fallback он не попадёт.

Типичные ошибки

  • Забыл включить Fallback, item не подошёл ни под одно правило → пропал без следа, в логах не видно → включи Fallback Output и подключи хотя бы лог.
  • Включил Fallback, но не подключил к нему ни одной ноды → items сваливаются в пустоту, эффект тот же что без fallback → подключи как минимум ноду-логгер или уведомление.
  • Использовал Fallback как «иначе для последнего правила» → последнее правило срабатывает, item уходит в свой выход, fallback пустой → добавь явное правило-ловушку или переосмысли логику веток.

📖HTTP-методы CRUD: GET / POST / PUT / DELETEстатья · ~2 мин▾

HTTP-методы CRUD: GET / POST / PUT / DELETE

Любая операция с данными через API сводится к четырём базовым действиям: прочитать, создать, обновить, удалить. Это и есть CRUD — Create / Read / Update / Delete. У каждой из этих операций есть свой HTTP-метод, и API-разработчики договорились использовать их одинаково. Если ты понимаешь четвёрку, ты понимаешь, что делает почти любой эндпоинт по одному взгляду в документацию.

Как это работает

МетодCRUD-операцияЧто делаетТело запросаИдемпотентность
GETReadПрочитать данные с сервераНетДа
POSTCreateСоздать новый ресурсДа (данные нового объекта)Нет
PUTUpdateОбновить существующий ресурс целикомДа (новое содержимое)Да
PATCHUpdateОбновить часть полей ресурсаДа (только изменяемые поля)Да
DELETEDeleteУдалить ресурсНетДа

Идемпотентность — это значит «вызвать N раз = вызвать 1 раз». GET идемпотентен (читать одно и то же — безопасно). PUT идемпотентен (поставить значение name=John хоть 100 раз — результат один). DELETE идемпотентен (удалить уже удалённое — обычно сервер просто скажет «нет такого» без побочных эффектов). POST не идемпотентен — каждый вызов создаёт новый ресурс с новым id.

URL-паттерн чаще всего одинаковый:

  • POST /items — создать (тело: данные нового item)
  • GET /items — получить список всех
  • GET /items/<id> — получить один по id
  • PUT /items/<id> — заменить целиком по id
  • PATCH /items/<id> — обновить часть полей по id
  • DELETE /items/<id> — удалить по id
Когда это пригодится
Чтение документации API

Любая документация API начинается со списка эндпоинтов. Если ты видишь POST /users — это «создать пользователя». DELETE /users/42 — «удалить пользователя 42». PUT /users/42 — «заменить пользователя 42 целиком». Один взгляд — и понятно, что делает эндпоинт, без чтения текста.

Типичные ошибки

  • Перепутал POST и PUT. POST создаёт новый ресурс — сервер сам присваивает ему id. PUT обновляет существующий по уже известному id. Если послать POST на /items/42 — поведение зависит от API: часть серверов вернёт ошибку, часть всё равно создаст новый.
  • Послал GET с телом запроса. В стандарте GET не имеет тела. Некоторые серверы такие запросы отклоняют, другие — игнорируют тело. Параметры в GET передаются через query string (?key=value), а не через body.
  • PUT вместо PATCH при частичном обновлении. PUT ждёт целый объект — если послать только одно поле, остальные поля могут обнулиться. Для частичных правок — PATCH. Точное поведение всегда смотри в документации конкретного API.

⚡Как быстро настроить HTTP Request ноду через Import from cURLстатья · ~2 мин▾

Импорт curl в HTTP Request ноду

cURL (читается «кёрл») — это команда для отправки HTTP-запросов из терминала. Документация большинства API устроена так: «вот пример запроса в виде curl-команды — отправь его и получишь ответ».

Не обязательно запускать curl в терминале. В n8n у HTTP Request ноды есть кнопка Import from cURL — она забирает готовую curl-команду и автоматически разбирает её на части: метод, URL, заголовки, тело запроса. Это самый быстрый способ настроить вызов любого API.

Пошагово

  1. Скопируй curl-команду из задачи или документации API.
  2. В n8n создай новый workflow (или открой sandbox). Добавь ноду HTTP Request и открой её настройки (двойной клик).
  3. В правом верхнем углу панели настроек ноды (под кнопкой Execute step) нажми Import cURL.
  1. В открывшемся диалоге Import cURL command вставь скопированную curl-команду в textarea. Нажми Import.
  1. n8n автоматически разберёт curl и заполнит поля HTTP Request ноды: Method (POST/GET/DELETE/…), URL, Send Headers, Send Body, и сами заголовки/тело. Проверь, что всё подставилось корректно — Method, URL и заголовки видны сразу, Body может быть ниже по скроллу.
  1. Если в curl был placeholder вроде ВАШ_КЛЮЧ или YOUR_TOKEN — найди его в импортированных полях (обычно в Header Parameters или в Body) и замени на реальное значение.
  2. Нажми Execute step — справа в панели OUTPUT появится ответ API.

Зачем это нужно знать: большинство сторонних API даёт примеры в виде curl-команд. Зная Import from cURL, ты можешь подключить почти любой сервис без чтения его SDK — просто скопировал команду из доков, импортировал, заменил ключ.

Подсказка
Не оставляй ключи в открытом виде

📖POST с JSON-телом в HTTP Requestстатья · ~2 мин▾

POST с JSON-телом в HTTP Request

Большинство современных API принимают данные не в URL, а в теле запроса в формате JSON. Это значит: метод POST (или PUT/PATCH), заголовок Content-Type: application/json и сам JSON-объект в body. В n8n всё это включается через один переключатель.

Как это работает

Внутри ноды HTTP Request после Method есть переключатель Send Body. Включаешь — появляется Body Content Type с выпадающим списком. Для JSON выбирай значение JSON. После этого открываются два режима ввода:

  • Using JSON — вписываешь готовый JSON-объект целиком. Удобно когда структура нетривиальная или есть вложенные объекты. Выражения через {{ }} тут тоже работают — можно вставить значение из предыдущей ноды прямо внутрь JSON.
  • Using Fields Below — заполняешь таблицу с парами Name/Value. n8n сам соберёт JSON-объект из этих полей. Подходит для плоских объектов из 2–4 полей, без вложенности.

Заголовок Content-Type: application/json n8n проставляет автоматически — отдельно его прописывать в Send Headers не нужно.

К примеру
Один и тот же запрос двумя способами

Объект { "title": "Hello", "tags": ["news", "test"] } через Using JSON пишется как есть — одной вставкой. Через Using Fields Below придётся завести строку title=Hello и для массива tags использовать выражение ["news", "test"] в Value — что менее читаемо. Чем сложнее структура, тем выгоднее Using JSON.

Типичные ошибки

  • Невалидный JSON в Using JSON. Лишняя запятая, одинарные кавычки вместо двойных, пропущенный } — и сервер вернёт 400 Bad Request. Проверяй структуру глазами или прогоняй через любой JSON-валидатор перед запуском.
  • Дублирование заголовка Content-Type. Если вручную в Send Headers добавить ещё один Content-Type, могут поехать кодировки или сервер откажет. Не трогай — n8n сам поставит правильный.
  • POST без Send Body. Метод включил, тело — нет. На выходе уйдёт пустой POST-запрос, сервер скорее всего вернёт ошибку валидации.

📖`.replace()` / `.replaceAll()` — заменить подстроку в строкестатья · ~1 мин▾

.replace() и .replaceAll() — одна и та же строка, другие слова

.replace(что, на_что) находит первое вхождение подстроки и заменяет его. .replaceAll(что, на_что) заменяет все вхождения. Оба метода возвращают новую строку — исходная не меняется.

'Иван Иванов'.replace('Иван', 'Пётр')
// → 'Пётр Иванов'   (только первое вхождение)

'+7 (999) 123-45-67'.replaceAll(' ', '')
// → '+7(999)123-45-67'   (все пробелы убраны)

В выражении n8n это выглядит так:

{{ $json.phone.replaceAll(' ', '').replaceAll('-', '') }}

Методы можно цепочкой — каждый следующий работает с результатом предыдущего.

Важно
`.replace()` заменяет только первое вхождение

'a.b.c'.replace('.', '-') вернёт 'a-b.c', а не 'a-b-c'. Для замены всех точек — .replaceAll('.', '-'). Это частая ошибка когда нужно нормализовать разделители во всей строке.

Типичные ошибки

  • .replace() вместо .replaceAll() — заменяется только первое совпадение, остальные молча остаются. Если нужно заменить все — всегда используй .replaceAll().
  • Регистрозависимость — .replace('иван', 'Пётр') не найдёт 'Иван' с заглавной буквой. Если регистр может отличаться — используй .replace() с регулярным выражением и флагом i (см. статью про .match() и .replace() с регулярками).
  • Замена числа — если поле пришло числом, .replace() упадёт с is not a function. Преобразуй сначала: String($json.value).replace(...) или $json.value.toString().replace(...).

📖$json или $('Имя ноды'): когда какая форма обращения к даннымстатья · ~2 мин▾

Две формы обращения — разница только в источнике

В n8n у одной и той же ноды есть две формы прочитать поле из данных воркфлоу:

{{ $json.business_idea }}
{{ $('Form Trigger').item.json.business_idea }}

Обе достают одно и то же значение, но смотрят в разные места:

  • $json — это выход ноды, которая стоит прямо перед текущей. Короткая форма для «возьми поле из предыдущей ноды».
  • $('Имя ноды') — это обращение к любой ноде воркфлоу по имени, в том числе к той, что стоит на 2-3-5 шагов раньше.

Когда писать какую

Правило простое:

  • Хочешь взять данные у предыдущей ноды (стрелка идёт прямо в текущую) — пиши $json.поле.
  • Хочешь дотянуться до ноды глубже по цепочке (между ней и текущей есть промежуточные ноды) — пиши $('Имя ноды').item.json.поле.
Пример
Цепочка из трёх нод

Цепочка: Form Trigger → OpenAI → Send Message.

В Send Message обращения к двум разным источникам выглядят так:

  • к ответу OpenAI (предыдущая нода): ={{ $json.output }}
  • к полю формы (нода на два шага назад): ={{ $('Form Trigger').item.json['Email'] }}

Если в Send Message написать ={{ $json['Email'] }} — поле не найдётся, потому что у OpenAI в выходе нет Email. Это типичная ошибка студентов.

Почему короткая форма вообще существует

$json — это просто синоним для $input.item.json, то есть «текущий обрабатываемый элемент входа». А вход текущей ноды формируется из выхода предыдущей. Поэтому $json.поле и $('Предыдущая нода').item.json.поле всегда дают одно и то же значение — пока стрелка идёт напрямую.

$('Имя ноды') нужна там, где короткая форма не справится: между источником и текущей нодой есть промежуточные шаги, которые этого поля уже не передают (например, OpenAI вернул только текст ответа, и поле формы Email к моменту Send Message живёт только в выходе Form Trigger).

Типичные ошибки

  • Написал $json.Email в ноде после OpenAI. OpenAI возвращает output, а не Email. Решение — обратиться по имени: $('Form Trigger').item.json['Email'].
  • Имя ноды в $('...') не совпадает с подписью на canvas. В expression имя — буквально то, что написано под нодой на canvas. Переименовал ноду — все старые expressions, ссылавшиеся на её прежнее имя, сломаются.
  • Забыл .item после имени ноды. Полная форма — $('Имя').item.json.поле. Без .item обращаешься не к данным, а к самой ноде как объекту — поле не найдётся.

Append n8n Attribution: подпись в конце автоматических сообщений

Параметр «Append n8n Attribution» управляет автоматической подписью, которую n8n добавляет в конец каждого отправленного сообщения. По умолчанию он включён — и к тексту любого сообщения дописывается фраза вида «This message was sent automatically with n8n».

Зачем это нужно знать

Для личных экспериментов и отладки подпись не мешает. Но когда автоматическое сообщение идёт реальным пользователям или сотрудникам — постороннее упоминание сервиса автоматизации выглядит непрофессионально. Получатель видит не уведомление от сервиса, а артефакт настройки системы.

Как это работает

Параметр находится в разделе Additional Fields ноды отправки сообщений — он скрыт по умолчанию, чтобы не засорять основной экран настроек. Чтобы его найти:

  1. Открой параметры ноды.
  2. Прокрути вниз до раздела «Additional Fields» и раскрой его.
  3. Найди параметр «Append n8n Attribution» — переключи в положение Off.

После отключения сообщения отправляются без каких-либо лишних дополнений к тексту.

К примеру
Где это встречается

К примеру, нода Telegram отправляет оператору-человеку уведомление о новом запросе от клиента. Если Attribution включён, оператор видит в конце: «This message was sent automatically with n8n» — это избыточно для внутреннего рабочего инструмента. Иногда Attribution заметен и в Email-нодах, когда письмо с автоматической подпиской уходит клиентам — это вредит доверию к бренду.

Типичные ошибки

  • Не нашли параметр — он спрятан в Additional Fields и не видён на основном экране. Нужно явно раскрыть блок.
  • Оставили включённым для пользовательских сообщений — получатели видят рекламу сервиса вместо нейтрального уведомления.
  • Перепутали с Parse Mode — Attribution и Parse Mode находятся в одном разделе Additional Fields, но это разные настройки. Parse Mode управляет форматированием (Markdown/HTML), Attribution — подписью.

Когда не подходит отключать

Если вы намеренно хотите показать, что сообщение отправлено автоматически — Attribution можно оставить включённым. Это может быть уместно в тестовых рассылках внутри команды, где прозрачность источника важна.

📖Path-параметры в Webhookстатья · ~2 мин▾

Path-параметры в Webhook

Когда один Webhook должен принимать события от нескольких источников, удобно зашить различия прямо в URL — не через ?source=payment, а через сегмент пути: /hook/payment/success. Для этого в n8n используются path-параметры: сегменты с двоеточием, которые становятся переменными.

Как это работает

В поле Path ноды Webhook ты пишешь URL с динамическими сегментами, например /hook/:source/:event. Двоеточие перед именем сегмента — это маркер переменной. n8n запомнит, что в этом месте URL может стоять что угодно, и передаст это «что угодно» дальше.

Когда придёт запрос на /hook/payment/success, n8n распакует его так:

$json.params.source  →  "payment"
$json.params.event   →  "success"

Все path-параметры лежат в $json.params — объект с ключами, совпадающими с именами сегментов без двоеточия.

Сравни с query-параметрами
Path vs Query

Path: https://n8n.../webhook/hook/payment/success → $json.params.source = "payment"

Query: https://n8n.../webhook/hook?source=payment → $json.query.source = "payment"

Технически оба варианта работают. Path-параметры удобнее, когда источник управляет URL (например, сторонний сервис шлёт на фиксированный адрес), а query — когда параметры формируются динамически на стороне отправителя.

Типичное применение — маршрутизация: после Webhook ставишь ноду Switch с условием на $json.params.source, и каждый источник уходит по своей ветке.

Типичные ошибки

  • Забыть двоеточие. Если написать hook/source/event вместо hook/:source/:event, n8n будет ожидать ровно такой URL, не переменный. Любой другой путь получит 404.
  • Искать параметр не там. Path-параметры живут в $json.params, а не напрямую в $json. Запись $json.source ничего не найдёт — нужно $json.params.source.
  • Опечатка в имени сегмента. Если в Path написано :Source (с заглавной), ключ в $json.params тоже будет Source. n8n чувствителен к регистру — следи за единообразием.
📖Send Query Parameters в HTTP Requestстатья · ~2 мин▾

Send Query Parameters в HTTP Request

В URL-запросе параметры обычно идут после знака ? через &: ?country=US&limit=10&q=hello world. Можно вписывать их прямо в URL руками, но n8n даёт удобную альтернативу — переключатель Send Query Parameters с таблицей.

Как это работает

Когда ты включаешь Send Query Parameters, под ним появляется таблица с двумя колонками — Name и Value. Каждая строка — один параметр. n8n сам соберёт из них итоговый query string и приклеит его к URL через ?.

Главные плюсы перед ручной вклейкой в URL:

  • Автоматическое URL-кодирование. Пробелы превращаются в %20, кириллица — в %D0%9A..., спецсимволы экранируются. Тебе не нужно помнить, что & внутри значения сломает запрос.
  • Каждое Value можно сделать выражением. Через переключатель Expression рядом с полем подставляешь значение из предыдущей ноды — {{ $json.someField }}. Если бы параметр был вшит в URL, выражение пришлось бы конкатенировать руками.
  • URL в поле остаётся чистым — без длинного хвоста ?key1=...&key2=.... Это лучше читается и быстрее правится.
К примеру
Один параметр — одна строка

Если у API в документации указано ?q=hello&limit=5, в таблице будет две строки: одна с Name=q, Value=hello, вторая с Name=limit, Value=5. Никакого ? или & руками вписывать не нужно.

Типичные ошибки

  • Оставить параметры и в URL, и в таблице. Они не объединятся — параметры из URL останутся, а таблица допишет свои поверх. Результат непредсказуем. Правило: либо всё в URL, либо всё в таблице.
  • Забыть переключить Value в Expression при подстановке из предыдущей ноды. В режиме Fixed строка {{ $json.address }} уйдёт на сервер как есть, без подстановки значения.
  • Передать пустую строку. Если Value пустое, n8n отправит ?name= — сервер может вернуть ошибку валидации. Проверь, что значение реально приходит из предыдущей ноды.

📖Edit Output: как поменять запинённые данные на летустатья · ~1 мин▾

Edit Output: как поменять запинённые данные на лету

Когда у ноды зафиксированы данные через Pin Data, в панели Output рядом с иконкой булавки появляется ещё одна кнопка — Edit Output (иконка карандаша). Она открывает запинённый JSON прямо в браузере на редактирование: можно поменять значения полей, не отправляя новый запрос во внешний сервис.

Зачем это нужно

Главный сценарий — тестировать одну и ту же цепочку нод на нескольких разных входах. Запинил один пример → отредактировал ему текст → запустил воркфлоу → отредактировал на другой текст → снова запустил. Особенно полезно при отладке роутеров (Switch, IF, Text Classifier) — каждая ветка требует своего набора входных данных, и звать внешний источник под каждый кейс долго.

Как редактировать

  1. У ноды с запинёнными данными в панели Output найди кнопку Edit Output (карандаш рядом с булавкой).
  2. JSON в панели станет редактируемым полем — впиши новые значения или поменяй существующие.
  3. Нажми Save (или клик за пределами поля). Новые данные останутся запинёнными до следующего редактирования или снятия пина.

Что важно

  • Структура JSON должна остаться прежней. Если случайно удалить нужное поле или сломать вложенность, следующие ноды получат не те данные и упадут с ошибкой cannot read properties of undefined. Меняй значения, а не ключи.
  • Меняй только то, что должно поменяться. Для тестирования ветки роутера обычно достаточно подменить одно поле (например, message.text) — всё остальное оставить.
  • Edit Output не вызывает реальный источник. Это локальное редактирование пина в браузере, никакого запроса в Telegram / API / другую систему n8n не делает.
  • Чтобы вернуться к настоящим данным от триггера — сними пин и поймай новый event через Execute workflow.

Multi-item processing в n8n: одна нода — несколько прогонов

Когда предыдущая нода вернула N items (например, 3 строки из Data Table) — следующая нода автоматически выполнится N раз: по одному прогону на каждый item. Никакого цикла настраивать не нужно, это нативное поведение n8n.

Как это работает

В каждом прогоне $json содержит данные ровно одного item — текущей строки. Так выражение {{ $json.name }} автоматически подставляет правильное значение для каждой записи.

Пример
3 строки → 3 сообщения в Telegram

Data Table вернул 3 записи: Иван, Мария, Сергей. Следующая нода Telegram Send Message без дополнительных настроек выполнится 3 раза и отправит 3 отдельных сообщения — по одному на каждого. Это поведение по умолчанию.

Как поменять поведение

Часто такое поведение не нужно. Если нода-отправитель должна послать одно сводное сообщение — N прогонов = N лишних сообщений. На этот случай в n8n есть три инструмента.

1. Execute Once — переключатель в Settings ноды. Включаешь его — нода прогоняется ровно один раз, n8n даёт ей только первый item, остальные игнорируются. Подходит для нод-отправителей: Telegram, Email, HTTP Request — когда нужно одно действие на весь массив. Подробнее — [[n8n-execute-once]].

2. Run Once for All Items — режим у Code-ноды (выбирается в верхнем переключателе). В этом режиме внутри Code доступен весь массив через items, и нода возвращает результат за один прогон.

3. Агрегация через Loop / Merge / SplitInBatches — отдельные ноды-цикла. Нужны когда логика сложнее «взять первый» или «всё одним куском»: пакетная обработка, объединение веток, custom-агрегация.

Как достать весь массив из выражения

Внутри выражения {{ ... }} любая нода может прочитать весь входной массив целиком — через $input.all(). Это массив items, по которому можно вызвать .length, .map(), .filter() и любые JS-методы массивов.

{{ $input.all().length }}                       // сколько items пришло
{{ $input.all().map(i => i.json.name).join(', ') }}   // имена через запятую

Особенность: $input.all() читает массив до того, как Execute Once срезает его до первого item. Поэтому если на ноде Telegram стоит Execute Once и на вход пришло 3 items — $input.all().length вернёт 3, а не 1. Это удобно для сводных сообщений вида «Получено заявок: 3».

Когда полезно знать

  • Если кейс «N сообщений вместо одного» — это не баг, это поведение по умолчанию. Решение — Execute Once или Run Once for All Items.
  • Если нужно посчитать число входных items в Set / Telegram / любой ноде — $input.all().length в выражении.
  • Если нужна последовательная обработка с большим массивом и контролем — SplitInBatches.

.match() — достать нужную часть из строки

Метод .match(регулярка) ищет совпадение с шаблоном и возвращает массив результатов или null, если ничего не нашлось.

Чаще всего нужна не вся строка, а конкретная часть — для этого в регулярке ставят захватывающую группу (...), и её значение окажется по индексу [1]:

'Заказ №12345 принят'.match(/№(\d+)/)?.[1]
// → '12345'

\d+ — один или несколько цифровых символов, () — захватить их, ?.[1] — безопасно взять первую группу.

В выражении n8n это выглядит так:

{{ $json.message.match(/№(\d+)/)?.[1] }}
Важно
Если совпадение не найдено — `.match()` вернёт null

Обращение [1] к null упадёт с ошибкой. Защита — optional chaining ?.: str.match(/.../)?.[1]. Тогда при отсутствии совпадения получишь undefined вместо падения.

Флаги

  • i — поиск без учёта регистра: /заказ/i найдёт и заказ, и Заказ, и ЗАКАЗ.
  • g — найти все совпадения: вернётся массив всех найденных строк (но без групп).
'один два три один'.match(/один/g)
// → ['один', 'один']

Типичные ошибки

  • Нет ?. перед [1] — если .match() вернул null, обращение к индексу упадёт. Всегда используй ?.[1], ?.[2] и т.д.
  • Флаг g и захватывающие группы — с флагом g .match() возвращает массив полных совпадений, а не групп: [1] уже не то, что ожидаешь. Для групп при флаге g используй .matchAll().
  • Не экранировал точку — /api.v1/ найдёт apiXv1 (точка = любой символ). Буквальная точка пишется как \..

📖Set: Include Other Input Fields — обогатить или финализироватьстатья · ~2 мин▾

Что делает переключатель

У ноды Edit Fields (Set) в секции Options есть переключатель Include Other Input Fields (в SDK — includeOtherFields). Он решает, что нода отдаёт на выход помимо полей, которые ты сам в ней установил.

ПоложениеЧто на выходе
onвсе исходные поля входного item плюс установленные тобой
offтолько те поля, которые нода явно установила

on — обогащение

Берёшь on, когда нода дополняет запись: исходные поля нужно сохранить, а к ним добавить новое. Например, к статье с полями title, link, content приклеить theme. Без on от статьи осталось бы одно поле theme, а всё остальное пропало бы.

off — финализация

Берёшь off, когда нода формирует строго определённый набор полей на выход — финальную структуру, которая уйдёт дальше (в БД, в сообщение, в ответ API). Все служебные поля от прошлых нод (creator, guid, categories, исходный content в HTML и т.п.) отсекаются, остаётся чистая структура.

Типичная ловушка: мусор протекает через всю систему

Две Set-ноды подряд, и в обеих Include Other Input Fields: on. Workflow завершается без ошибки, но в финальных записях болтаются служебные поля от самых ранних нод — данные «грязные», и этот мусор течёт дальше по системе.

Гигиена item shape
Обогащаешь — on, финализируешь — off

Первая Set в цепочке обычно обогащает (on — добавляет поля к существующим). Последняя Set перед отправкой данных наружу — финализирует (off — отдаёт ровно нужный набор). Если последняя Set оставлена в on, проверь, не тащит ли она лишние служебные поля.

Как решить, что ставить

Спроси: «эта нода добавляет поле к записи или собирает финальный вид записи?»

  • Добавляет (а остальное должно сохраниться) → on.
  • Собирает финальный набор наружу → off.

Нода настроена. Чтобы протестировать её, нужно сначала подменить запиненные данные триггера на команду обнови.

🚀Stretch▾

Сейчас в каждом bin'е хранится только имя контакта. На практике у клиента всегда есть ещё телефон, иногда — заметки (про предпочтения, аллергии, скидку постоянного клиента). Без этих полей бот не работает как настоящая CRM.

Расширь схему bin: пусть команда создай принимает имя и телефон одной строкой (например, создай Анна Смирнова +79001234567), а в bin сохраняются оба поля отдельно.

список показывает обе строки.

обнови тоже принимает оба поля.

Намёк: разбор команды на части — это снова регулярки .match() / .replace(), а структура body в HTTP Request меняется на два поля вместо одного.
