Что собираем

Собираем два воркфлоу.

1Получи токен у @BotFather▾

Бот в Telegram создаётся не в n8n и вообще не в браузере. Telegram сделал для этого специального служебного бота — @BotFather. Через него любой пользователь Telegram создаёт собственного бота и получает на него токен — длинную строку, которая даёт полный доступ к этому боту. Этот токен мы потом вставим в n8n, и все ноды Telegram через него будут говорить от имени твоего бота.

Этот шаг ты делаешь в Telegram-приложении, n8n пока даже не открывай.

📖@BotFather: как создать собственного бота в Telegramстатья · ~2 мин▾

@BotFather: как создать собственного бота в Telegram

@BotFather — это официальный бот Telegram, через которого создаются все остальные боты. Других способов завести бот нет: ни сайта, ни приложения, ни консоли — только переписка с этим аккаунтом.

Зачем это нужно

Чтобы любой workflow (в n8n, Make, на собственном бэкенде) мог слушать сообщения и отвечать от имени бота, ему нужен токен Bot API. Токен выдаётся при создании бота через @BotFather. Без токена ни одна интеграция с Telegram не заведётся: токен — это пара «логин-пароль» бота при общении с Telegram Bot API.

Как это работает

Открываешь Telegram, в поиске находишь аккаунт с username BotFather (галка подлинности рядом с именем — синяя), и пишешь ему как обычному собеседнику. Команды начинаются со слэша:

  • /newbot — создать нового бота. @BotFather спросит две вещи:
  • отображаемое имя (любое, может быть с пробелами и кириллицей) — то, что увидят пользователи в шапке чата;
  • username (только латиница, цифры, подчёркивание; обязательно заканчивается на _bot или bot) — постоянный идентификатор, по которому бот ищется и не меняется.
  • /mybots — список всех твоих ботов, доступ к настройкам (имя, описание, аватарка, команды, токен).
  • /revoke — отозвать старый токен и получить новый (если старый утёк).

После /newbot приходит сообщение с токеном — длинная строка вида 123456789:ABCdefGHI_jklMNO.... Это и есть тот самый Bot API token: его копируешь и сохраняешь в безопасное место (менеджер паролей, заметки с шифрованием) — но не в публичный репозиторий и не в чат.

Типичные ошибки

  • Потерять токен — @BotFather показывает токен один раз; чтобы получить его снова, нужно открыть /mybots → <твой бот> → API Token. Если не нашёл — можно сгенерировать новый через /revoke, но тогда старый перестанет работать во всех уже настроенных интеграциях.
  • Скинуть токен скриншотом / в открытый чат — токен даёт полный контроль над ботом. Любой, кто его увидит, может слать сообщения от имени бота и читать всё, что ему приходит. Если утёк — сразу /revoke.
  • Username без _bot на конце — @BotFather отклонит регистрацию; правило жёсткое, придётся подбирать ещё.
  • Перепутали username и отображаемое имя — отображаемое имя меняется в любой момент через /setname, а username меняется только пересозданием бота. Если задумывался серьёзный бот — username стоит выбрать осознанно.

Когда не подходит

Через @BotFather нельзя создать юзербот (бот, который ходит под обычным аккаунтом) — это уже не Bot API, а MTProto, и токен оттуда не подходит. Также @BotFather не нужен, если бот уже создан кем-то другим и тебе просто передали готовый токен — тогда сразу заводишь credential в нужном сервисе и работаешь с ним.

2Создай основной воркфлоу и настрой Webhook▾

В разделе Workflows создай новый воркфлоу с именем M1-08 Payments Router и переходи на canvas. До шага 7 включительно мы работаем только в нём — тестер соберём отдельным воркфлоу в шагах 8-10.

Первая нода — Webhook. Она генерирует уникальный URL и ждёт входящий HTTP-запрос на этот адрес. Когда запрос приходит — воркфлоу запускается.

Нам нужен метод POST. В POST-запрос отправитель кладёт данные в тело запроса — например, JSON с полями платежа (amount, currency, from_email). В адресную строку браузера POST не вписать: его шлют программно — через curl, Postman, нашу систему интернет-магазина или вторую n8n-ноду HTTP Request.

Параметр Respond оставим в режиме Immediately: пусть Webhook отвечает отправителю сразу, не дожидаясь пока Telegram отправит сообщение. Тому, кто пришлёт уведомление о платеже, не нужно знать что мы потом с ним сделали.

Webhook в n8n: как нода ждёт входящий запрос

Webhook — это URL-адрес, по которому внешний сервис или браузер может запустить воркфлоу. Нода Webhook создаёт такой адрес автоматически — тебе остаётся только настроить метод и способ ответа.

Два URL: Test и Production

Нода Webhook всегда выдаёт два разных адреса:

  • Test URL — работает только когда ты нажал Execute workflow (нода ждёт один входящий запрос, потом останавливается). Используется при сборке и отладке.
  • Production URL — работает постоянно, пока воркфлоу опубликован и активен. Используется в боевом режиме.

Режим ответа: Using 'Respond to Webhook' Node

По умолчанию Webhook отвечает на запрос сразу, как только принял его. Когда выбираешь Using 'Respond to Webhook' Node — ответ откладывается до момента, когда нода Respond to Webhook явно его отправит. Это нужно, чтобы сформировать кастомный JSON-ответ в конце цепочки.

Pin data — зачем фиксировать данные

После первого запроса к Test URL нода получает входящие данные (headers, query, body). Кнопка Pin data фиксирует эти данные: все последующие запуски шагов будут использовать именно их, без повторного открытия браузера. Это ускоряет отладку — не нужно каждый раз отправлять новый запрос.

Edit Output: как поменять запинённые данные на лету

Когда у ноды зафиксированы данные через Pin Data, в панели Output рядом с иконкой булавки появляется ещё одна кнопка — Edit Output (иконка карандаша). Она открывает запинённый JSON прямо в браузере на редактирование: можно поменять значения полей, не отправляя новый запрос во внешний сервис.

Зачем это нужно

Главный сценарий — тестировать одну и ту же цепочку нод на нескольких разных входах. Запинил один пример → отредактировал ему текст → запустил воркфлоу → отредактировал на другой текст → снова запустил. Особенно полезно при отладке роутеров (Switch, IF, Text Classifier) — каждая ветка требует своего набора входных данных, и звать внешний источник под каждый кейс долго.

Как редактировать

  1. У ноды с запинёнными данными в панели Output найди кнопку Edit Output (карандаш рядом с булавкой).
  2. JSON в панели станет редактируемым полем — впиши новые значения или поменяй существующие.
  3. Нажми Save (или клик за пределами поля). Новые данные останутся запинёнными до следующего редактирования или снятия пина.

Что важно

  • Структура JSON должна остаться прежней. Если случайно удалить нужное поле или сломать вложенность, следующие ноды получат не те данные и упадут с ошибкой cannot read properties of undefined. Меняй значения, а не ключи.
  • Меняй только то, что должно поменяться. Для тестирования ветки роутера обычно достаточно подменить одно поле (например, message.text) — всё остальное оставить.
  • Edit Output не вызывает реальный источник. Это локальное редактирование пина в браузере, никакого запроса в Telegram / API / другую систему n8n не делает.
  • Чтобы вернуться к настоящим данным от триггера — сними пин и поймай новый event через Execute workflow.

3Подключи Switch в режиме Expression▾

Вторая нода — Switch. У неё есть два режима:

  • Rules — задаёшь список правил «если поле X равно/содержит/больше значения Y, то отправь в этот выход». Каждое правило — отдельный выход на canvas, можно подписать имя выхода через Rename Output.
  • Expression — задаёшь одно JavaScript-выражение, которое возвращает номер выхода (число 0, 1, 2 или 3). Switch направляет item в выход с этим номером. Заранее задаёшь сколько всего выходов нужно — Switch создаст их на canvas, пронумерованные 0/1/2/3 без подписей.

Режим Expression удобен, когда нужно добавить сразу много правил и не хочется ковыряться с добавлением каждого правила отдельно.

Нода Switch делит поток на несколько веток сразу

Switch — это IF для случаев, когда веток больше двух. Каждый элемент входных данных проверяется набором правил, и в зависимости от того, какое правило сработало, элемент уходит в свой выход. У ноды столько выходов, сколько правил ты создал, плюс опционально выход «ни одно правило не подошло».

Режим Rules — основной способ

В режиме Rules ты задаёшь список правил. Каждое правило — это:

Кнопка Add Routing Rule добавляет новое правило — и автоматически появляется новый выход на canvas. Удалишь правило — выход исчезнет.

По умолчанию элемент уходит в первое подошедшее правило и дальше не проверяется. Если ни одно правило не подошло — элемент отбрасывается.

Режим Expression — для динамического числа веток

В режиме Expression ты сам говоришь, сколько выходов нужно (Number of Outputs), и пишешь выражение, которое возвращает индекс выхода (0, 1, 2...). Подходит, когда ветка вычисляется формулой, а не цепочкой условий.

Когда это пригодится
Один источник — много направлений обработки

Например, входящие заявки делятся на «срочные», «обычные» и «спам» — у каждого свой набор нод дальше. Без Switch пришлось бы ставить три IF подряд и тянуть провода в обход, либо повторять одни и те же условия в каждой ветке.

Типичные ошибки

  • Ждёшь, что элемент попадёт в несколько веток одновременно, но он уходит только в первую → включи опцию Send data to all matching outputs или поменяй порядок правил.
  • Ни одно правило не сработало, и элементы просто исчезают → включи Fallback Output со значением Extra Output, чтобы такие элементы шли в отдельную ветку и их было видно.
  • Правила пересекаются по условиям, элементы уходят не туда → пересмотри порядок: правила проверяются сверху вниз, переставь более специфичные выше общих.
  • В режиме Expression выражение вернуло число больше, чем Number of Outputs - 1 → элемент потеряется. Возвращай индекс с учётом нумерации с нуля.
3.3
Задай Number of Outputs = 4

В поле Number of Outputs впиши 4. На canvas справа от Switch появятся четыре выходные точки, пронумерованные 0, 1, 2, 3 — пока без подключённых нод.

3.4
Впиши тернарное выражение в поле Output

Впиши в поле Output Index:

{{ $json.body.currency === "RUB" ? 0 : $json.body.currency === "USD" ? 1 : $json.body.currency === "EUR" ? 2 : 3 }}

Это JavaScript-выражение через тернарный оператор условие ? значение_если_да : значение_если_нет:

  • если валюта RUB — вернуть 0
  • если USD — 1,
  • если EUR — 2,
  • в остальных случаях - 3 (GBP по умолчанию). Switch получит число от 0 до 3 и направит item в соответствующий выход.

Fixed и Expression — два режима любого поля

В правом верхнем углу почти каждого поля параметров ноды есть переключатель Fixed | Expression. Это два режима ввода значения: фиксированное (написал текст — этот текст и пойдёт) или формула на JavaScript, которая вычисляется заново при каждом выполнении ноды. Один и тот же тумблер есть у обычных текстовых полей, у dropdown-ов, у credential pickers и у resource locator полей (где выбираешь объект из списка — таблицу, документ, канал).

Как работает каждый режим

  • Fixed — обычный ввод. Что написал — то и попадёт в запрос/в подстановку. У resource locator (когда выбираешь объект — таблицу, документ, sheet) Fixed означает выбор из dropdown «From list» или ручной ввод id одним значением.
  • Expression — поле подсвечивается оранжевым, а внутри можно писать формулу в фигурных скобках: {{ $json.field }}, {{ $now.toISO() }} и т.п. Значение пересчитывается на каждой итерации, для каждого пришедшего элемента.
Когда переключать в Expression
Значение зависит от данных или меняется между запусками

Если значение должно зависеть от того, что пришло из предыдущей ноды (имя пользователя, id записи, текст сообщения), от текущего времени или от любой логики — переключай в Expression. Если значение всегда одно и то же (метод HTTP POST, имя таблицы, фиксированный slug) — оставляй Fixed.

Как переключить режим

Три способа сделать одно и то же:

  1. Клик по Fixed | Expression в правом верхнем углу поля. Самый явный — видно сразу, в каком режиме поле.
  2. Иконка шестерёнки слева от поля → пункт Add Expression. Удобно когда хочешь сразу открыть expression builder (окно с деревом нод слева и preview значения справа).
  3. Drag-and-drop значения из панели Input слева в поле. Это самый частый способ в учебных задачах: открываешь NDV предыдущей ноды через мини-карту слева или раскрываешь панель Input, цепляешь нужное поле мышкой и тянешь в параметр текущей ноды. n8n при отпускании сам переключает режим в Expression и подставляет правильную ссылку — {{ $json.field }} или {{ $('Имя ноды').item.json.field }}.

В обратную сторону — клик по Fixed в углу. Содержимое expression-режима очистится, поле станет обычным текстовым.

Тонкость
У resource locator в Expression нет dropdown

В resource locator полях (например, выбор Data table или Spreadsheet) при переключении в Expression список объектов из dropdown пропадает — теперь ты должен сам подставить нужный id или slug формулой. Полезно когда id берётся из предыдущей ноды; если нужно просто выбрать существующий объект мышкой — оставайся в Fixed.

Типичные ошибки

  • Забыл переключить в Expression, написал формулу как текст. Поле в Fixed-режиме воспринимает {{ $json.name }} как обычные символы и отправит эту строку буквально. Признак: значение в preview / в результате выполнения содержит {{ и }} как есть. Лечение: переключить в Expression вручную (тумблер в правом верхнем углу) или перетянуть значение из панели Input заново.
  • Стёр выбор из resource locator, переключив в Expression и обратно. Переключение режима очищает содержимое поля. Если в Fixed уже был выбран нужный объект из списка — после Expression → Fixed его придётся выбрать заново.

Что такое expression

В n8n expression — это маленький кусочек кода в фигурных скобках, который динамически подставляет значение в поле ноды. Записывается так:

={{ <выражение> }}

Знак равенства = в начале — это маркер для n8n: «трактуй содержимое этой ячейки не как обычную строку, а как выражение, которое нужно вычислить». Без = n8n запишет в поле литеральный текст {{ ... }} как есть.

Внутри {{ }} пишется JavaScript-выражение. Самое частое — чтение значения из другой ноды.

Зачем нужны expressions

Каждое поле в параметрах ноды может быть либо фиксированным значением (например, URL https://api.example.com), либо expression'ом — формулой, которая считается на лету в момент исполнения ноды. Expressions нужны, когда значение зависит от данных, пришедших из предыдущей ноды.

К примеру
Где это встречается

К примеру, ты сделал ноду «прочитать строку из таблицы конфига», а следующая нода — HTTP Request к внешнему API. В URL этой HTTP-ноды можно подставить значение из конфига: ={{ $('Get Config').item.json.api_url }} — n8n при каждом исполнении возьмёт актуальный URL из таблицы, а не зашитый в ноду.

Как читать значения из других нод

Базовая формула: ={{ $('<имя_ноды>').item.json.<поле> }}

Разбор по частям:

  • $('<имя_ноды>') — обращение к ноде по имени (то, что написано на canvas под нодой). Имя — в одинарных кавычках, точно как на canvas.
  • .item — текущий элемент, который обрабатывается. У большинства нод элементов несколько (массив), item берёт тот, что соответствует текущей итерации.
  • .json — содержимое поля json элемента (там лежат все данные, которые нода вернула).
  • .<поле> — конкретное поле внутри json. Если поле вложенное — пишется через точку: .config.embedding_model.

Пошагово — как вставить expression в поле

  1. Кликни на иконку шестерёнки рядом с полем, выбери Add Expression (или нажми Expression в верхнем правом углу поля).
  1. В появившемся окне (expression builder) — слева список нод воркфлоу, можно кликать по полям и они вставятся в формулу.
  1. Справа — preview: видно, какое значение получится прямо сейчас. Если красным — формула или путь к полю некорректные.
  2. Закрой окно, поле остаётся в режиме expression (синий фон + иконка fx).
Типичная ошибка
`=` в начале — обязателен

Если скопировал готовое выражение из учебной задачи и вставил без = в начале — n8n запишет в поле текст {{ ... }} как есть, нода уйдёт делать HTTP-запрос на URL вроде ={{ $('...').item.json.api_url }} буквально. Симптом — нода падает с ошибкой Invalid URL или возвращает 404 на странную строку.

Типичные ошибки

  • Имя ноды не совпадает. В формуле $('Get Config') имя должно точно совпадать с подписью ноды на canvas. Если переименовал ноду — старые expressions, ссылающиеся на неё, сломаются.
  • Путь к полю не существует. .item.json.embedding_dimensions упадёт с undefined, если у элемента такого поля нет. Препетий — посмотри preview в expression builder, он сразу показывает значение или ошибку.
  • Забыл = в начале. Самая частая опечатка — описано в callout выше.
  • Кавычки внутри {{ }}. Внутри expression строки оборачиваются одинарными кавычками: ={{ 'Hello ' + $json.name }}. Двойные конфликтуют с JSON.

Когда expression не нужен

Иногда поле должно быть фиксированным. Например, метод HTTP-запроса (GET / POST) или имя коллекции, которое одинаково для всего воркфлоу — нет смысла читать его динамически. Простое правило: если значение меняется от итерации к итерации или зависит от пришедших данных — нужен expression; если значение постоянное — оставляй фиксированным.

Switch в режиме Expression — формула решает, куда идёт элемент

У ноды Switch два режима: Rules (визуальный список условий, у каждого свой выход) и Expression (одно выражение, которое возвращает номер выхода). Expression нужен, когда условий слишком много для удобного списка, когда индекс ветки вычисляется арифметически (например, round-robin через % N), или когда логика заранее неизвестна — приходит из данных.

Как настроить

::key-value{rows=[{key:"Mode",value:"переключи в Expression."},{key:"Number of Outputs",value:"сколько выходов будет у ноды на canvas. Принимает фиксированное число (1, 2, 3...) или выражение, если выходов должно быть переменное количество — но смотри предупреждение ниже про canvas."},{key:"Output Index",value:"выражение, возвращающее целое число от 0 до Number of Outputs - 1. Это и есть номер выхода, в который уйдёт элемент. Нумерация с нуля."}] id=switch-expression-params}

Простой пример — три ветки по приоритету:

={{ $json.priority === 'high' ? 0 : $json.priority === 'low' ? 2 : 1 }}

Чуть сложнее — round-robin распределение по N воркерам:

={{ $json.id % 3 }}

Ключевые особенности и тонкости

One-liner only
Только одно выражение в одну строку

В поле Output Index нельзя написать многострочный JavaScript. Только одно выражение — тернарные операторы, методы, арифметика. Если логика не помещается в одну строку, либо складывай тернарники друг в друга, либо считай индекс в отдельной Code-ноде перед Switch и тяни значение оттуда.

Нет Fallback Output
Элементы с невалидным индексом просто исчезают

В режиме Rules есть опция Fallback Output — куда отправить элементы, не подошедшие ни под одно правило. В Expression такой опции нет. Если выражение вернуло дробное число, NaN, undefined или число больше Number of Outputs - 1 — элемент потеряется без предупреждения. Прежде чем возвращать индекс, защити его: Math.floor(...), Math.min(N - 1, ...), fallback через || или ??.

Динамический Number of Outputs ломает canvas
Не подставляй выражение в количество выходов, если их подключаешь руками

Если в Number of Outputs подставить выражение (например, ={{ $json.numberOfOut }}), n8n не сможет отрисовать выходы на canvas заранее — connection-точки не появятся, и подключить следующие ноды визуально не получится (известная проблема в n8n). Динамический Number of Outputs имеет смысл только если дальше Switch вообще не нужно подключать вручную — редкий кейс. Для обычной работы оставляй фиксированное число.

Когда Expression выгоднее Rules

  • Веток больше 5-6, и условия — это монотонные сравнения одного поля (status === 'a' / 'b' / 'c'...). Список правил в Rules будет визуально перегружен — тернарник в одну строку компактнее.
  • Индекс вычисляется арифметикой: round-robin (id % N), бакеты по диапазону (Math.floor(score / 10)), хэш по строке.
  • Логика приходит из конфига — например, в Code или Set посчитал target_branch: 0/1/2 и Switch просто берёт это поле.
Когда лучше остаться в Rules
Условия читаются как бизнес-правила

Если ветки — это «срочные / обычные / спам», и каждая имеет понятное имя — Rules выигрывает: можно включить Rename Output и подписать выходы (urgent / normal / spam), это сразу читается на canvas. В Expression выходы остаются безымянными — просто 0, 1, 2.

Типичные ошибки

  • Выражение вернуло 1.5 или строку "1" → элемент потерялся. Оборачивай в Number(...) или Math.floor(...), чтобы гарантировать integer.
  • Возвращаешь индекс с нумерацией с единицы (1, 2, 3 вместо 0, 1, 2) → последний выход пустой, элементы первой ветки теряются. Помни: с нуля.
  • Поле $json.priority отсутствует у части элементов → тернарник по undefined уходит в ветку «else». Добавляй явную проверку: $json.priority == null ? 0 : ....
  • В Number of Outputs подставлено выражение, выходы не появились на canvas → замени на фиксированное число, рассчитанное вручную по максимальному значению индекса.
  • Логика разрослась, тернарники вложены на 4-5 уровней — читать невозможно → вынеси расчёт индекса в Code-ноду или Set, а в Switch оставь {{ $json.target_branch }}.

Тернарный оператор — это if-else, сжатый в одну строку

Тернарный оператор — это короткая запись if-else, которую можно вставить прямо внутрь выражения. В n8n это особенно удобно: в полях параметров (тех, что в режиме Expression) разрешён только один JavaScript-выражение в одну строку. Полный if (...) { ... } else { ... } туда не поместится, а тернарник — да.

Базовый синтаксис

условие ? значение_если_true : значение_если_false

Читается слева направо как одна фраза: «если условие, то значение_если_true, иначе значение_если_false». JavaScript сначала проверит условие, а потом вернёт одно из двух значений — в зависимости от результата.

Простой пример:

{{ $json.amount > 1000 ? 'large' : 'small' }}

Здесь n8n возьмёт поле amount из входного item, сравнит с 1000 и вернёт строку 'large' или 'small'. Эту строку можно положить в любое поле ноды — Subject письма, тело сообщения, путь файла.

Вложенные тернарники

Если веток больше двух, тернарники можно складывать друг в друга — справа в значение_если_false ставится новый тернарник:

{{ $json.priority === 'high' ? 0 : $json.priority === 'low' ? 2 : 1 }}

Логика: «если high — вернуть 0; иначе если low — вернуть 2; иначе 1». Последняя ветка (1) — это default: туда уходят все значения, которые не подошли ни под одну предыдущую проверку.

Default-ветка молча съедает всё незнакомое
Любое неожиданное значение попадёт в последнюю ветку

Если приходит значение, которое ты не предусмотрел (priority равно urgent или null), оно попадёт в последнюю ветку — 1. Это не ошибка JavaScript, но это тихий баг: значение обрабатывается по-другому, чем должно. Перед последней веткой добавляй явную проверку, либо логируй неожиданные значения отдельно.

Когда тернарник выгоден
Один оператор вместо ветки IF или ноды Switch

Тернарник короче и нагляднее, когда веток 2-3 и каждая возвращает простое значение (строка, число, поле). Если веток больше 4 или внутри них сложная логика — лучше разнести через ноду Switch или вычислить значение в Code-ноде, иначе вложенные тернарники станут нечитаемы.

Типичные ошибки

  • Перепутал порядок: condition ? false_value : true_value. Условие первое, дальше идёт значение для true, потом для false — именно в таком порядке.
  • Забыл скобки во вложенных тернарниках с другими операторами. Если внутри тернарника есть && / || / арифметика — оборачивай части в скобки, иначе JavaScript разберёт выражение не так, как ты ожидаешь: (a && b) ? x : y понятнее, чем a && b ? x : y.
  • Положил в тернарник «побочный эффект» (вызов ноды, изменение переменной). Тернарник должен только возвращать значение, не делать что-то ещё. Логику вынеси в Code-ноду или в отдельную ноду.
  • Сравниваешь строки оператором == вместо ===. В JavaScript === — строгое сравнение (тип и значение), == — с приведением типов ("1" == 1 будет true). В тернарниках лучше всегда === — меньше неожиданного поведения.
📖Почему в программировании счёт начинается с нулястатья · ~2 мин▾

Первый элемент — это нулевой

В обычной жизни мы считаем «первый, второй, третий». В программировании первый элемент имеет номер 0, второй — 1, третий — 2. Это называется индексация с нуля (zero-based indexing), и так устроены массивы, списки, выходы нод, символы в строках — почти всё, у чего есть «номер по порядку».

Почему так

Индекс — это не «который по счёту», а «на сколько шагов отступить от начала». Первый элемент стоит в самом начале — отступать не нужно, шагов 0. Второй — на один шаг дальше, индекс 1. Третий — на два, индекс 2.

Этот подход родился из того, как данные физически лежат в памяти компьютера: есть точка начала и смещение от неё. Прижилось — теперь почти все языки программирования (JavaScript, Python, Java, C, и многие другие) и почти все инструменты считают именно так.

Switch как пример

У ноды Switch в режиме Expression ты пишешь выражение, которое возвращает номер выхода. Первый выход (самый верхний на canvas) — это 0, второй — 1, третий — 2.

// первая ветка (output 0) — срочные заявки
// вторая ветка (output 1) — обычные
// третья ветка (output 2) — спам
={{ $json.priority === 'high' ? 0 : $json.priority === 'spam' ? 2 : 1 }}

Если у Switch четыре выхода, значит допустимые индексы — 0, 1, 2, 3. Не 1..4. Вернёшь 4 — элемент потеряется, потому что выхода с таким индексом нет.

Правило одной строкой
N элементов → индексы от 0 до N−1

Десять элементов в массиве — индексы 0..9, не 1..10. Пять выходов у Switch — индексы 0..4. Если где-то увидишь индекс, равный длине, — это уже за границей и обычно приводит к ошибке или потерянным данным.

Где ещё это встретится

  • Массивы в выражениях n8n: $json.items[0] — первый элемент, $json.items[1] — второй.
  • Символы в строке: первый символ имеет позицию 0.
  • Циклы: for (let i = 0; i < items.length; i++) — i начинается с нуля и идёт до length − 1.
  • Колонки и строки в таблицах через код: часто первая строка/колонка — [0], а не [1] (в формулах Excel/Sheets — наоборот, там с единицы; в коде Python/JS — с нуля).

Типичные ошибки

  • Хочешь обратиться к первому элементу и пишешь $json.items[1] — получаешь второй и удивляешься «почему данные не те» → первый это [0].
  • В режиме Expression у Switch пишешь return 1 для первой ветки, а данные уходят во вторую → первая ветка это 0.
  • Считаешь длину массива и думаешь, что последний индекс совпадает с длиной → последний индекс это длина − 1. Массив из 3 элементов: индексы 0, 1, 2, длина 3.

Запустить одну ноду, не дёргая весь workflow

Execute step — это кнопка внутри панели ноды (открывается двойным кликом по ноде). По смыслу — младший брат большой кнопки Execute workflow внизу canvas: та запускает весь воркфлоу с начала, эта — только текущую ноду. Удобно, когда настраиваешь параметры и хочешь сразу видеть, что отдаёт нода, не перезапуская всю цепочку.

Как это работает

Когда ты жмёшь Execute step, n8n берёт уже существующие данные на входе ноды (output предыдущей ноды из последнего прогона) и прогоняет через эту ноду. Сама нода выполняется реально: HTTP-запрос уходит, запись в базу пишется, письмо отправляется. Это не «сухой прогон», а полноценное выполнение — только одного шага.

  • key: Execute workflow

value: Запускает весь воркфлоу от триггера. Нужен полный прогон с самого начала. Каждый раз заново дёргает все API.

  • key: Execute step

value: Запускает только эту ноду. Берёт вход из последнего output предыдущей ноды. Идеально для подбора параметров и отладки.

  • key: Когда работает

value: Только если на входе ноды есть данные с прошлого прогона. Если ни разу не запускал предыдущие — кнопка либо ничего не вернёт, либо упадёт «No input data».

Типичный сценарий: запустил весь воркфлоу один раз (Execute workflow), увидел что нода HTTP Request отдаёт не то, поправил поле URL, нажал Execute step — увидел новый результат за секунду. Предыдущие ноды повторно не выполнялись.

У триггеров кнопка называется иначе
Смысл тот же — запустить один шаг и получить тестовое событие

У некоторых триггерных нод n8n кнопка Execute step подписана по-другому, потому что у триггера нет «входа» — он сам должен сходить за событием. Например, у RSS Trigger она называется Fetch Test Event: клик подтянет последние записи из ленты и положит их на выход ноды как тестовый input для остального воркфлоу. Работает идентично Execute step обычной ноды — точно так же используется для отладки и подбора параметров без полного прогона.

Важно про побочные эффекты
Каждый Execute step — это реальный вызов

Нода с записью в базу или отправкой сообщения каждое нажатие Execute step делает реально: 5 нажатий = 5 записей / 5 писем. Для отладки таких нод используй тестовые данные или подменяй credentials.

Типичные ошибки

  • Жать Execute step на ноде, у которой ещё не было входных данных → «No input data», нода не запустится. Сначала прогони весь воркфлоу через Execute workflow или хотя бы триггер вручную, чтобы у каждой ноды появился output.
  • Использовать Execute step для проверки изменений в предыдущих нодах → текущая нода работает со старым output, изменения не подхватятся. Если поправил предыдущую ноду — перезапусти её, а потом уже текущую.
  • Тестировать ноду с побочным эффектом (запись в БД, отправка письма) на проде → каждый клик создаёт реальную запись. Использовать тестовый credential или временно подменять параметры на безопасные.

Универсальный процесс добавления credential

Credential в n8n — это сохранённая «учётка» для подключения к сервису (API-ключ, OAuth-токен, логин/пароль и т.д.). Любая нода, которой нужна авторизация, выбирает credential из общего списка — поэтому добавлять его нужно один раз, а использовать можно из любого количества нод того же типа.

Пошагово

  1. В левом меню n8n открой раздел Credentials.
  1. Нажми кнопку Add credential (правый верхний угол).
  1. В поиске введи название нужного типа credential — например, OpenAI, Header Auth, Postgres, Google — и выбери его из списка.
  1. В шапке формы дай credential понятное имя — например, OpenAI школа или Postgres prod. По этому имени ты будешь выбирать его в нодах.
  1. Заполни поля формы. Какие именно — зависит от типа: для API-ключевых это обычно одно поле API Key, для Header Auth — Name и Value заголовка, для Postgres — host/port/database/user/password, для OAuth — запуск OAuth-flow по кнопке.
  1. Нажми Save. Готово — credential можно подключать к любой ноде, которая поддерживает этот тип.
Подсказка
Один credential — много нод

Один сохранённый credential можно использовать в неограниченном количестве нод одного типа. Если ключ протухнет — поменяй его в credential один раз, обновится во всех нодах сразу.

📖Как узнать свой Telegram chat IDстатья · ~1 мин▾

Как узнать свой Telegram chat ID

chat_id в Telegram — это уникальный числовой идентификатор личного чата, группы или канала. Боту он нужен, чтобы понимать, кому отправлять сообщение: API принимает не username (@vasya), а именно число (например, 123456789). Свой собственный chat_id совпадает с твоим user_id — это число, которое Telegram присвоил тебе при регистрации, и оно не меняется.

Способ 1 — @userinfobot

Самый быстрый.

  1. Открой Telegram и найди в поиске @userinfobot.
  2. Нажми Start (или отправь любое сообщение).
  3. Бот пришлёт ответ с полем Id — это и есть твой chat_id.

Способ 2 — через Telegram Web

Если бот недоступен или не хочется ставить лишних ботов.

  1. Открой web.telegram.org и залогинься.
  2. Зайди в «Избранное» (Saved Messages) — это твой личный чат сам с собой.
  3. Посмотри на адресную строку: после # будет число, например https://web.telegram.org/a/#123456789. Это твой chat_id.
Когда это пригодится
Когда боту нужно слать сообщение лично тебе

Любая задача, где Telegram-нода в n8n должна отправить сообщение конкретному человеку (тебе как админу, тестовому пользователю), а не отвечать на входящий update. В поле Chat ID ноды подставляется именно это число.

Типичные ошибки

  • Указать @username вместо числа → Telegram Bot API вернёт ошибку Bad Request: chat not found. API понимает только числовой ID.
  • Использовать чужой chat_id (например, скопированный из примера) → сообщения уйдут другому человеку или, если ID не существует, бот получит chat not found.
  • Перепутать chat_id группы и личный chat_id → у групп ID начинается с минуса (например, -1001234567890), у личных чатов он положительный.

Drag-and-drop из панели Input — самый быстрый способ связать ноды

В n8n каждая нода видит данные предыдущих нод в панели Input слева от параметров. Чтобы подставить любое поле из этого Input в параметр текущей ноды, не нужно вручную писать expression — достаточно перетащить поле мышкой. n8n сам подставит правильное выражение со ссылкой на нужную ноду.

Как это работает

  1. Открой параметры ноды двойным кликом. Слева автоматически появится панель Input — это выходные данные предыдущей ноды (или одной из предыдущих, если их несколько).
  2. Разверни в Input структуру до нужного поля — она показывается как дерево вложенных объектов: message → chat → id, output → [0] → content → [0] → text и так далее.
  3. Перетащи нужное поле мышкой прямо в поле параметра справа. В поле появится выражение вида ={{ $('<имя предыдущей ноды>').item.json.<путь.к.полю> }}.
  4. Поле в правой панели окрасится синим — это значит, что параметр теперь не литерал, а expression: на каждом запуске значение будет браться из реальных данных.
К примеру
Откуда n8n берёт имя ноды

К примеру, если предыдущая нода называется Telegram Trigger, и ты перетащил из неё поле message.text — n8n запишет ={{ $('Telegram Trigger').item.json.message.text }}. Имя ноды берётся ровно такое, как написано на canvas. Иногда — если переименуешь ноду уже после drag-and-drop, выражение сломается, потому что в формуле останется старое имя.

Откуда выбрать источник данных

Если у ноды несколько входов или нужно дотянуться не до прямого предыдущего шага, а до более раннего — сверху панели Input есть переключатель источника. К примеру, в финальной ноде workflow можно вывести Input от первого триггера, чтобы достать оттуда поле, которого нет в выходе ближайшей предыдущей ноды.

Типичные ошибки

  • Перетащил не на то поле параметра. Drag-and-drop срабатывает на любом текстовом поле — легко уронить значение не в Text, а в Chat ID, и наоборот. После перетаскивания проверь, что в нужном поле появилось ожидаемое выражение.
  • Перетащил из объекта-обёртки, а не из конкретного поля. Если потянуть за message, а не за message.text — n8n подставит весь объект, в текстовое поле уйдёт [object Object] или JSON-строка. Раскрывай дерево до примитивного значения.
  • Сломался путь после переименования ноды. В выражении $('Telegram Trigger') имя ноды фиксированное. Если потом переименовать Telegram Trigger в что-то другое — нужно либо вернуть старое имя, либо переписать выражение вручную.

Когда лучше не использовать drag-and-drop

Если значение должно быть фиксированным (например, имя коллекции, метод HTTP, статичный текст приветствия) — drag-and-drop не нужен, просто впечатай значение в поле как обычный текст. Drag-and-drop — это про динамические параметры, которые меняются от запуска к запуску в зависимости от входных данных. </content> </invoke>

⚙Append n8n Attribution: подпись в конце автоматических сообщенийстатья · ~2 мин▾

Append n8n Attribution: подпись в конце автоматических сообщений

Параметр «Append n8n Attribution» управляет автоматической подписью, которую n8n добавляет в конец каждого отправленного сообщения. По умолчанию он включён — и к тексту любого сообщения дописывается фраза вида «This message was sent automatically with n8n».

Зачем это нужно знать

Для личных экспериментов и отладки подпись не мешает. Но когда автоматическое сообщение идёт реальным пользователям или сотрудникам — постороннее упоминание сервиса автоматизации выглядит непрофессионально. Получатель видит не уведомление от сервиса, а артефакт настройки системы.

Как это работает

Параметр находится в разделе Additional Fields ноды отправки сообщений — он скрыт по умолчанию, чтобы не засорять основной экран настроек. Чтобы его найти:

  1. Открой параметры ноды.
  2. Прокрути вниз до раздела «Additional Fields» и раскрой его.
  3. Найди параметр «Append n8n Attribution» — переключи в положение Off.

После отключения сообщения отправляются без каких-либо лишних дополнений к тексту.

К примеру
Где это встречается

К примеру, нода Telegram отправляет оператору-человеку уведомление о новом запросе от клиента. Если Attribution включён, оператор видит в конце: «This message was sent automatically with n8n» — это избыточно для внутреннего рабочего инструмента. Иногда Attribution заметен и в Email-нодах, когда письмо с автоматической подпиской уходит клиентам — это вредит доверию к бренду.

Типичные ошибки

  • Не нашли параметр — он спрятан в Additional Fields и не видён на основном экране. Нужно явно раскрыть блок.
  • Оставили включённым для пользовательских сообщений — получатели видят рекламу сервиса вместо нейтрального уведомления.
  • Перепутали с Parse Mode — Attribution и Parse Mode находятся в одном разделе Additional Fields, но это разные настройки. Parse Mode управляет форматированием (Markdown/HTML), Attribution — подписью.

Когда не подходит отключать

Если вы намеренно хотите показать, что сообщение отправлено автоматически — Attribution можно оставить включённым. Это может быть уместно в тестовых рассылках внутри команды, где прозрачность источника важна.

Когда Telegram присылает сообщение в Telegram Trigger, в JSON-объекте message рядом с полем text появляется массив entities. Если в тексте ничего интересного нет (просто «Привет»), поля entities в объекте не будет вообще. Если есть — внутри массив объектов.

Бот видит: «есть entity типа bot_command с позиции 0 длиной 13», и понимает что в сообщении именно команда, а не просто текст. По offset+length бот может вырезать команду из строки и работать дальше с аргументами ("42" в примере).

offset считается в UTF-16, не в символах
Эмодзи и сложные символы могут занимать 2 единицы вместо 1

В большинстве случаев offset совпадает с обычным индексом символа в строке. Но если в тексте есть эмодзи или сложные unicode-символы — они занимают 2 UTF-16 code units, и offset/length будут считать их как 2. JavaScript-методы String.prototype.substring, slice работают именно в UTF-16 — им можно подставлять offset/length напрямую. А вот в массиве через Array.from(text) один эмодзи будет одним элементом, и тут offset уже не сойдётся. Если работаешь с обычным латинско-кириллическим текстом — забыть про это можно, обычно всё ровно.

Когда entities полезны
Не парсить текст регулярками — Telegram уже всё разметил

Если нужно реагировать только на команды боту — проверь entities[i].type === 'bot_command', это надёжнее регулярки ^/, потому что Telegram сам определил, что это именно команда, а не строка которая случайно начинается со слэша. То же самое для упоминаний (mention, text_mention), хэштегов, ссылок — Telegram уже отделил их от обычного текста.

Где искать entities в разных типах сообщений

  • Обычное текстовое сообщение — поле entities рядом с text.
  • Сообщение с медиа (фото, видео, документ) и подписью — поле caption_entities рядом с caption. Поля entities в таком сообщении нет — оно появляется только когда есть text.
  • Цитата ответа на сообщение — поле entities внутри объекта quote.
  • Reply — у reply_to_message свои entities, отдельно от внешнего сообщения.

Типичные ошибки

  • Ищешь команду в text регуляркой и срабатываешь на любой текст со слэшем (например, «адрес: /home/user»). Лучше — entities с type === 'bot_command': Telegram сам различает команду и обычный слэш.
  • Работаешь со ссылками через regex по тексту, хотя Telegram уже разметил их в entities с типами url (обычная ссылка) и text_link (скрытая ссылка — у этого entity есть поле url).
  • Ждёшь entities в подписи к фото — а оно там называется caption_entities. У сообщений без text (только медиа) поля entities нет вообще.
  • Считаешь offset как индекс символа в обычной строке для текста с эмодзи — получаешь смещение на 1-2 позиции. Либо режь через substring/slice (они в UTF-16, всё сойдётся), либо учитывай разницу вручную.

Как протестировать Webhook в n8n: три способа

После того как ты добавил ноду Webhook и получил Test URL, нужно как-то отправить на этот адрес запрос — иначе нода не сработает и данные не появятся. Есть три удобных способа сделать это.

Способ 1. Прямо из браузера или через n8n

Самый простой случай — Webhook с методом GET и без обязательных параметров. Скопируй Test URL и вставь его в адресную строку браузера. Браузер отправит GET-запрос, нода поймает его и покажет полученные данные.

Этот способ подходит для первой проверки «работает или нет». Минус: браузер умеет только GET, и ты не можешь задать заголовки или JSON-тело. Для POST или PUT нужны другие инструменты.

Способ 2. curl — из командной строки

curl — консольный инструмент, который умеет отправлять любой HTTP-запрос. Предустановлен на macOS и Linux, на Windows есть в новых версиях. Запрос пишется одной строкой в терминале.

GET-запрос:

curl https://your-n8n.com/webhook-test/abc123

POST с JSON-телом:

curl -X POST https://your-n8n.com/webhook-test/abc123 \
  -H "Content-Type: application/json" \
  -d '{"name": "Иван", "age": 30}'

Что означают флаги:

  • -X POST — метод запроса (по умолчанию GET)
  • -H "Content-Type: application/json" — заголовок, чтобы сервер понял формат тела
  • -d '...' — тело запроса в виде JSON

curl хорош для разовой проверки: пять секунд — и видно, что Webhook сработал. Подробнее об инструменте — ::inline-article{slug=curl-http-client}.

Способ 3. Postman — графический клиент

Postman — это бесплатная программа с окном и кнопками, которая делает то же самое, что curl, но через удобный интерфейс. Ты вводишь URL в поле, выбираешь метод из выпадающего списка, добавляешь заголовки в отдельную вкладку, пишешь JSON-тело в редакторе с подсветкой синтаксиса. Запросы сохраняются в коллекции — можно вернуться к ним через неделю и переотправить одним кликом.

Когда Postman удобнее curl:

  • Запрос длинный и сложный (много заголовков, большое JSON-тело) — в редакторе с подсветкой легче не ошибиться.
  • Нужно сохранить запрос и переиспользовать его много раз.
  • Нужно посмотреть ответ сервера в красивом виде с подсветкой JSON.
  • Работаешь с авторизацией (OAuth, Bearer-токены) — у Postman есть встроенные мастера для каждого типа.

Скачать можно на postman.com. Аналоги — Insomnia, Bruno, Hoppscotch: те же возможности, разный дизайн.

Совет
С чего начать

Для простых случаев — curl или браузер. Если работаешь с API часто и каждый день — поставь Postman, со временем у тебя соберётся коллекция готовых запросов на все случаи жизни.

Все, воркфлоу собран и проверен реальными POST-запросами через сеть. Можешь поэкспериментировать с другими валютами в JSON-теле тестера, а потом выполнить Stretch (задачу со звездочкой).

🚀Stretch▾

Сейчас выражение в Switch — длинный ternary с четырьмя case'ами по числу поддерживаемых валют. При добавлении пятой валюты (например, CNY — китайский юань) придётся дописывать ещё одну ветку в формулу и менять параметр Number of Outputs. На реальной работе бухгалтерия раз в полгода добавляет новые валюты — формула быстро превратится в нечитаемую лестницу ? : ? : ? : ?.

Перепиши выражение Switch так, чтобы порядок валют задавался списком (например, ["RUB", "USD", "EUR", "GBP"]), а индекс выхода вычислялся через метод массива, который возвращает позицию элемента в списке. Тогда для добавления новой валюты будет достаточно дописать один элемент в список, не трогая ternary-конструкцию. Подумай, какой встроенный метод у массивов в JavaScript подходит для «найти индекс элемента по значению».

Проверить новое выражение удобно через тот же M1-08 Payment Sender: поменяй в JSON-теле currency на пятое значение (например, CNY) и прогони POST. Без правки выражения item уйдёт в default-выход — посмотри, в какой именно.
