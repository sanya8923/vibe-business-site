Что такое sub-нода

В n8n AI-ноды устроены как «родитель + дети». Родитель — основная нода (AI Agent, Qdrant Insert, Qdrant Vector Store…). Дети — sub-ноды, которые подключаются к специальным кружкам на нижней стороне родителя. Каждая sub-нода — это «поставщик ресурса» для родителя.

Какие бывают sub-ноды

  • Embeddings — поставляет векторы (OpenAI / Cohere / Voyage / Google Gemini / bge / …).
  • Chat Model — поставляет LLM для AI Agent.
  • Document Loader — поставляет «нарезанные» куски документа.
  • Text Splitter — внутри Document Loader, разбивает на чанки.
  • Memory — поставляет историю диалога для AI Agent.
  • Tool — поставляет инструменты, которые AI Agent может вызвать.

Когда родитель запускается (например, Qdrant Insert получил файл и собирается записать его в БД), он по очереди обращается к sub-нодам: «дай мне чанки», «переведи каждый чанк в вектор». Sub-ноды отвечают, родитель собирает результат.

Как выглядит sub-нода в параметрах родителя

Если открыть параметры родителя двойным кликом — справа выезжает панель параметров, а на нижней границе панели видны кружки sub-нод. Подключённая sub-нода показана как круглый цветной индикатор с иконкой провайдера. При наведении — всплывающая подсказка с именем и статусом.

Как заменить sub-ноду

  1. Закрой панель параметров родителя (Esc).
  2. На canvas найди подключённую sub-ноду — она расположена под родителем. Кликни на ней, нажми Delete (или правый клик → Delete). Sub-нода удалится, у родителя освободится слот.
  3. На родителе освободившийся слот покажет «+» (например, Add document embedding). Нажми — справа откроется список доступных sub-нод нужного типа.
  1. Выбери нужную sub-ноду — она автоматически подключится к родителю на освободившемся слоте.
  2. Заполни параметры новой sub-ноды (credential, модель, режим). Сохрани workflow.
Важно
Sub-ноды локальны для своего родителя

Sub-ноды не разделяются между нодами. Если в воркфлоу несколько родителей одного типа (например, две ноды Qdrant Insert — для txt и для pdf — и у каждой своя sub-нода Embeddings) — заменить нужно в каждом родителе отдельно.

Главная тонкость
Все параметры через выражения из конфига

В этой задаче почти каждое поле каждой ноды — это не литерал, а выражение ={{ ... }} со ссылкой на конфиг из DataTable. Имя модели, температура, system prompt, описание инструмента, имя коллекции Qdrant — всё. Единственный хардкод — Limit=5 в Qdrant KB (типовое значение для базы). Зачем так: чтобы менять поведение агента можно было через одну строку в DataTable, без правки самого workflow. Завтра ты захочешь сменить модель — открыл DataTable, поправил ячейку, сохранил, готово.

Зачем выносить логику в отдельный workflow

Когда в одном workflow собирается больше 20-30 нод, его становится тяжело читать и тестировать. К примеру, у тебя есть бот: он принимает сообщение, классифицирует, ходит в базу знаний, вызывает LLM, форматирует ответ, отправляет обратно. Если всё это в одном workflow — любая правка одной части требует разбирательства со всем полотном.

Решение — разбить логику на несколько workflow и вызывать одни из других. В n8n для этого есть три ноды: Execute Workflow Trigger, Execute Workflow и Tool Workflow.

Execute Workflow Trigger — точка входа в sub-workflow

Это специальный триггер: вместо того чтобы сработать на вебхук или расписание, он срабатывает когда другой workflow вызвал текущий. Поставь его в начало sub-workflow вместо обычного Webhook/Schedule trigger.

В параметрах Execute Workflow Trigger описываются входные данные: какие поля и какого типа sub-workflow ожидает на входе. К примеру, для sub-workflow «классификация сообщения» вход — text (строка) и user_id (число). Эти описания подсказывают вызывающему workflow, что нужно передать.

Execute Workflow — вызов sub-workflow из основного

Кладёшь эту ноду в основной workflow там, где нужно «передать управление» sub-workflow. В параметрах:

  • Workflow ID или название — какой sub-workflow вызвать.
  • Input data — что передать на вход (берётся из предыдущей ноды основного workflow).
  • Wait for sub-workflow completion — обычно On (вызывающий ждёт результат).

После выполнения sub-workflow возвращает данные обратно — нода Execute Workflow в основном workflow получает их как $json следующего шага.

Tool Workflow — sub-workflow как инструмент агента

Это особая разновидность Execute Workflow для AI Agent. Tool Workflow подключается к ноде AI Agent как sub-нода (через слот tool). Агент сам решает, когда его вызвать — на основании system message и текущего контекста разговора.

Внутри Tool Workflow всё то же самое что в обычном sub-workflow: Execute Workflow Trigger в начале, бизнес-логика в середине, ответ возвращается агенту в конце. Разница только в способе запуска: не явный вызов из основного workflow, а решение модели «мне нужен этот инструмент».

Когда что использовать

К примеру, в продуктовом агенте часто получается такая раскладка:

Гибкость

  • Sub-workflow можно вызывать из десятков мест — один раз написал, переиспользуешь везде.
  • Sub-workflow можно тестировать отдельно: запускаешь его руками с тестовым вводом, не трогая основной workflow.
  • Логи у каждого workflow свои — проще локализовать ошибку.

Pin Data в n8n: как зафиксировать тестовый вход ноды

Pin Data — это способ «прибить» к ноде заранее заготовленные данные, которые n8n будет подставлять вместо настоящего входа на каждом тестовом запуске. Полезно, когда дёргать реальный источник (внешний воркфлоу, веб-хук, API) на каждое нажатие Execute Workflow неудобно или невозможно.

Зачем это нужно

Часто отлаживается не сам триггер, а ноды после него — логика, маппинг, запрос в базу. Чтобы не звать каждый раз источник данных (другой воркфлоу, бот, форму), удобнее один раз получить пример настоящего входа, зафиксировать его на триггере и потом крутить запуски сколько нужно.

Поэтому Pin Data — это инструмент разработчика на этапе сборки, не часть рабочего воркфлоу. В боевом режиме закреплённые данные перестают использоваться: n8n берёт реальный вход.

Как это работает

К любой ноде с выходом можно прикрепить «запинённый» JSON. После этого:

  • При запуске воркфлоу через UI (Execute Workflow, тестовый прогон) n8n не вызывает саму ноду, а сразу выдаёт следующим нодам зафиксированные данные.
  • Pin Data привязана к конкретной ноде, не к workflow целиком — можно запинить триггер и оставить остальные ноды живыми.
  • На canvas у запинённой ноды появляется визуальный значок (булавка/иконка), чтобы не забыть, что данные «искусственные».

Удобнее всего пинить триггер: на нём поле выхода — то, что в боевом режиме приходит снаружи (Telegram-апдейт, тело веб-хука, поля Execute Workflow Trigger). Тестовый JSON собирают с теми же ключами, что отдаёт реальный источник.

К примеру
Где это встречается

К примеру, в Sub-Workflow с триггером Execute Workflow Trigger пин на триггере позволяет запускать сам Sub-WF кнопкой сверху и отлаживать ноды после триггера, не запуская вызывающий workflow. Типичные ошибки

  • Забыли убрать пин перед боевым запуском — workflow пошёл в production, но данные ноды всегда одни и те же; реальные входы игнорируются. Перед публикацией пины снимают.
  • Структура запинённого JSON не совпадает с реальным — отладка проходит идеально, но в проде ломается на полях, которых не было в тестовом примере; делайте пин из реальной execution, а не «на глаз».
  • Запинили середину воркфлоу, забыли — переименовали поле в ноде до пина, а пин остался со старой структурой. Следующие ноды читают «фантомные» данные.
  • Pin Data перепутали с переменными среды — пин не передаётся между воркфлоу и не сохраняется при экспорте на другую инсталляцию n8n как «конфиг»; это именно отладочный фиксаж.

Когда не подходит

Pin Data не заменяет настоящие тесты — это удобство в редакторе, не автоматизация. Если нужна повторяемая проверка с разными входами, лучше делать отдельный тестовый workflow или мини-набор тест-кейсов; пин подходит, когда хочется быстро прогнать «один и тот же» сценарий N раз.

Как достать строку из DataTable n8n по фильтру

DataTable в n8n — это таблица, которую можно вести прямо внутри проекта рядом с воркфлоу. Когда из такой таблицы нужно вытащить одну конкретную строку, используется операция Get rows с фильтром по условию.

Зачем это нужно

DataTable удобна, когда нужно хранить конфигурацию или справочник, к которому воркфлоу обращается на лету: «возьми строку с таким-то ключом и подставь её поля в дальнейшие ноды». Чтобы не доставать всю таблицу и не фильтровать её потом руками, фильтр задаётся прямо в параметрах ноды.

Как это работает

  1. В параметрах ноды Data Table выбирается Resource: Row и Operation: Get.
  2. В поле Data Table указывается имя таблицы.
  3. В блоке Filter → Conditions добавляется одно или несколько условий: имя колонки + оператор + значение.
  4. Значение фильтра почти всегда вводится в режиме выражения ={{ ... }}, чтобы взять его из предыдущей ноды (например, поле из триггера). Литералом фильтр пишется только когда значение известно заранее и никогда не меняется.
К примеру
Фильтр по выражению vs литерал

К примеру, если в таблице лежит конфиг под разные каналы (telegram, web, email), а имя канала приходит из триггера — в условии пишется ={{ $('Trigger').item.json.channel }}. Если бы стоял литерал telegram, при подключении второго канала пришлось бы дублировать ноду.

Типичные ошибки

  • Литерал вместо выражения — фильтр работает только для одного значения, нет повторного использования.
  • Опечатка в имени колонки — нода возвращает пусто, а ошибка молчаливая (нет «такой колонки нет»); проверять имя колонки буква в букву.
  • Не включён Always Output Data в Settings — при пустом результате нода не отдаёт ничего вниз, и следующая нода (например, Code) падает на cannot read property of undefined. Если по бизнесу пустой результат — это валидное состояние, включают Always Output Data, чтобы получить хотя бы пустой объект.
  • Несколько условий по AND, ожидали OR — в n8n несколько условий в Conditions объединяются AND; для OR-фильтра нужен либо отдельный путь через IF, либо выкачать всё и отфильтровать в Code.

Когда не подходит

Если в одной таблице нужно склеить данные из нескольких строк (агрегация, выборка) — лучше или вытащить всё через Get many rows, или построить логику на ноде после. Get rows с фильтром даёт по сути одну строку и не подходит для сложных запросов.

Always Output Data — на вкладке Settings, не Parameters

Если этого тоггла не включить, при отсутствующем канале (когда фильтр ничего не нашёл) нода тихо остановится и следующая нода управление не получит. Это нам нужно — мы хотим, чтобы дальше всё-таки попыталось работать и упало внятно. Тоггл Always Output Data живёт не в Parameters, а на соседней вкладке Settings — там общие технические настройки ноды.

📖JSON.parse() в Code-ноде n8n: десериализация строки в объектстатья · ~2 мин▾

DataTable n8n хранит значения как обычный текст (тип string). Если нужно положить в ячейку структурированные данные — их записывают в формате JSON и читают обратно через JSON.parse().

Зачем это нужно

JSON — это текстовый формат для передачи структурированных данных. Когда такой текст хранится в DataTable-ячейке или приходит из внешнего API — он всё ещё текст, а не объект. Чтобы обратиться к его полям (например, config.model), нужно сначала преобразовать строку в объект. Именно это делает JSON.parse().

Как это работает

JSON.parse(строка) — принимает строку, возвращает объект JavaScript. После этого можно читать любые поля через точку.

Обратная операция — JSON.stringify(объект) — превращает объект обратно в строку. Для записи в DataTable нужен stringify, для чтения — parse.

К примеру
Где это встречается

К примеру, в DataTable хранится строка {"model":"gpt-4.1","temperature":0.3}. В Code-ноде восстанови объект:

const config = JSON.parse($json.config);
// config.model === "gpt-4.1"
// config.temperature === 0.3
return [{ json: config }];

Иногда конфиг содержит вложенные объекты — JSON.parse() разворачивает их полностью, доступ через цепочку точек: config.qdrant.collection.

Типичные ошибки

  • SyntaxError: Unexpected token — строка содержит невалидный JSON. Проверь: кавычки не экранированы лишний раз, нет запятой после последнего поля объекта, все скобки закрыты. Скопируй значение на jsonlint.com — зелёный знак значит формат корректный.
  • JSON.parse вместо JSON.stringify — частая путаница. parse строка→объект (для чтения из DataTable), stringify объект→строка (для записи в DataTable).
  • undefined вместо значения — если поле называется model, а обращаешься как config.Model (с заглавной буквы) — вернёт undefined. JSON-ключи чувствительны к регистру.

Когда не подходит

Если данные не нужно читать по полям — парсинг избыточен. К примеру, если просто передаёшь строку дальше как есть — JSON.parse() не нужен.

⚙Code-нода в n8n: Run Once for All Items vs Run Once for Each Itemстатья · ~2 мин▾

Code-нода в n8n: Run Once for All Items vs Run Once for Each Item

Code-нода умеет работать в двух режимах — обрабатывать всю входящую пачку данных одним проходом или вызываться отдельно на каждый элемент. Выбор режима меняет, как код видит вход и как формирует выход; ошибиться легко, потому что внешне нода выглядит одинаково.

Зачем это нужно

В n8n между нодами ходят не одиночные объекты, а массивы — список items. Когда нужно собрать одну сводку из всех элементов сразу (агрегация, склейка, выборка лучшего) — удобнее увидеть всю пачку за один вызов. Когда логика поэлементная и не зависит от соседей — удобнее писать код так, будто на входе ровно один item, и не думать про индексы и циклы.

Эти два сценария и стоят за двумя режимами. Один и тот же код в неподходящем режиме либо обработает только первый элемент, либо упадёт на обращении к данным «не той формы».

Как это работает

  • Run Once for All Items. Code-нода вызывается один раз. На входе — массив всех items, и код сам решает, что с ним делать. Доступ к данным через $input.all() (массив объектов с полями json, binary) или $input.first() / $input.last(). Возвращать нужно массив items — даже если он из одного объекта (return [{ json: ... }]).
  • Run Once for Each Item. Code-нода вызывается N раз — по числу входных items, для каждого отдельно. Внутри код видит ровно один item: $input.item (один объект с json и binary) или короткие $json / $binary. Возвращать нужно один объект — return { json: ... } без массива.

Доступ к данным других нод ($('Имя ноды').item.json, $('Имя ноды').all()) работает в обоих режимах одинаково.

К примеру
Где это встречается

К примеру, нужно склеить настройки из строки конфига и поля из триггера в один объект — это «один-к-одному», и режим Run Once for All Items со взятием первого item проще и читабельнее. Иногда — нужно к каждому из 50 пришедших чанков прибавить кусок метаданных по тому же шаблону; тут Run Once for Each Item снимает с автора и for-цикл, и заботу про возврат массива.

Типичные ошибки

  • Выбран For All Items, а код возвращает один объект без массива — n8n покажет ошибку «expected array», потому что в этом режиме контракт возврата — список items.
  • Выбран For Each Item, а внутри $input.all() или индексация по массиву — код увидит только один элемент в all() (текущий), логика «собрать все» развалится.
  • return забыт совсем — нода ничего не передаёт дальше; следующая нода работает с пустым входом или ошибкой.
  • В For Each Item пишется return [{ json: ... }] массивом — формально работает (n8n развернёт), но это путаница: режим обещает один объект, проще держать однородно.

Когда не подходит

Если задача — простой маппинг одного-двух полей без логики, обычно проще обойтись нодой Edit Fields (Set) без кода вообще. Code-нода нужна там, где есть условия, парсинг, JSON-десериализация или нестандартная сборка объекта.

Код для Normalize Data
const row = $input.first().json;
const trigger = $('When Executed by Another Workflow').item.json;

const config = JSON.parse(row.config || '{}');

return [{
  json: {
    ...config,
    channel: row.channel,
    sessionId: String(trigger.sessionId),
    chatInput: trigger.chatInput,
    start_time: trigger.start_time || Date.now()
  }
}];
Тонкость
Почему `String(sessionId)`

Telegram отдаёт user_id числом (123456789), а нода Postgres Chat Memory сравнивает ключ сессии как строку. Если не привести к строке — память для одного и того же пользователя может разойтись по двум разным «сессиям» (число и строка с тем же содержимым — это разные ключи в БД).

Совет
`|| Date.now()` на случай, если триггер не передал

Если в будущем какой-то канал забудет передать start_time — в логе будет аккуратный фоллбэк, а не ошибка. Маленькая, но полезная страховка.

System message — это «должностная инструкция» агента

В ноде AI Agent в n8n есть параметр System Message (camelCase: systemMessage). Это текст, который модель видит как самое первое сообщение в каждом разговоре — раньше любого пользовательского ввода. Через него ты задаёшь агенту его роль, тон ответа, доступные инструменты и поведение в нестандартных ситуациях. Если system message не написан — агент ведёт себя как «общий ChatGPT» без понимания контекста твоей задачи.

System message в n8n выглядит как обычный многострочный текст. Никакого специального синтаксиса нет — модель читает его как живой язык.

Из чего состоит хороший system message

К примеру, типичный system message бота-помощника собирается из четырёх блоков (порядок не строгий, но обычно идёт сверху вниз):

  1. Роль и контекст. Кто этот агент и кому он помогает. «Ты — помощник по базе знаний компании. Отвечаешь сотрудникам на вопросы по внутренним документам.» Этот блок задаёт «персону» — как модель должна себя позиционировать.
  2. Инструменты и когда их использовать. Если у агента подключены tools (sub-ноды Tool Workflow, HTTP Request as Tool, Vector Store as Tool) — кратко перечисли их и опиши, в каких случаях агент должен их вызывать. Например: «Используй инструмент search_kb когда вопрос про внутренние процессы компании. Используй web_search когда нужна свежая информация из интернета.» Без этого блока агент часто ленится и пытается отвечать из общих знаний модели вместо вызова tool'а.
  3. Что делать, если ответа нет. Без явной инструкции модель часто выдумывает. К примеру: «Если в найденных документах нет ответа — скажи об этом честно: "В базе знаний я не нашёл ответа на этот вопрос". Не придумывай факты, не отвечай из общих знаний.»
  4. Тон и формат ответа. «Отвечай коротко, 2-3 предложения. Используй маркированные списки только если в ответе несколько отдельных пунктов. Не извиняйся в начале каждого ответа.»

Шаблон-болванка

Ты — <роль агента>. Помогаешь <кому> с <чем>.

У тебя есть инструменты:
- `<имя_tool_1>` — <когда вызывать>
- `<имя_tool_2>` — <когда вызывать>

Если в результатах поиска нет ответа на вопрос — скажи об этом прямо. Не придумывай факты.

Отвечай <стиль: коротко / развёрнуто / структурированно>. <Опц.: на каком языке.>

Типичные ошибки

  • «Ты лучший помощник в мире» — пустые комплименты модели не работают, экономь токены на полезной информации.
  • Длинный system message без структуры. Если он стал больше ~500 слов — раздели на разделы с подзаголовками или вынеси примеры в отдельный блок «Примеры» в конце. Модель легче следует структурированному тексту.
  • Описание tool'а в system message противоречит схеме параметров tool'а. К примеру, в system message написано «передавай в search_kb сразу полный вопрос пользователя», а tool принимает query (строка) и top_k (число) — модель будет путаться. Описание в system message должно совпадать с реальным контрактом tool'а.

Опция Return Intermediate Steps у AI Agent

Return Intermediate Steps — это опция AI Agent в n8n, которая включает запись каждого шага агента в финальный output. По умолчанию агент возвращает только итоговый ответ, а тут к ответу добавляется массив intermediateSteps со всем, что агент делал по пути.

Зачем это нужно

Когда агент работает с инструментами (Vector Store как Tool, MCP-инструменты, sub-workflow), без промежуточных шагов очень сложно понять, что произошло внутри. Студент видит финальный ответ — и всё. Если ответ кривой, непонятно, агент не вызвал инструмент или вызвал, но с кривым запросом.

С включённой опцией в output попадает массив объектов вида:

intermediateSteps: [
  { action: { tool: 'Qdrant KB', toolInput: '...' }, observation: '...' },
  { action: { tool: '...', toolInput: '...' }, observation: '...' }
]

Каждый элемент — это одна итерация цикла «модель решает что вызвать → вызов инструмента → ответ инструмента → модель решает дальше». По этому массиву видно: какой инструмент, с каким аргументом, и что инструмент вернул.

Как это включить

В параметрах AI Agent нажимают Add Option и выбирают Return Intermediate Steps. После этого появляется тоггл — переключают в on.

К примеру
Что покажет intermediate steps

К примеру, если агент должен был сходить в базу знаний за ответом, в intermediateSteps[0].action.tool появится имя ноды (Qdrant KB), в toolInput — запрос, который агент сформулировал, в observation — кусок найденных документов. Если массив пустой — агент решил, что инструмент звать не нужно (и ответил из головы).

Типичные ошибки

  • Включили в проде — каждый ответ агента таскает за собой массив шагов, объём output растёт, токенов на сериализацию тратится больше; в боевом режиме опцию обычно выключают.
  • Спутали с tracing — Intermediate Steps пишет только steps конкретного вызова; для системного наблюдения за всеми запусками используется отдельный слой (Langfuse, logs в БД), а не эта опция.
  • Ждут логов от sub-нод — если инструмент — это под-workflow, Intermediate Steps покажет, что агент дёрнул инструмент и что вернулось, но не покажет внутренних шагов этого под-workflow.

Когда не подходит

В рабочем режиме опция избыточна — нужен только итоговый output.answer. Включают её на этапе сборки и отладки. Когда тестов хватает и поведение стабильное, тоггл выключают, чтобы не таскать лишний JSON.

Тоггл Use Responses API в Chat Model по умолчанию ВКЛЮЧЁН. Школьный LiteLLM-прокси (через который мы ходим к OpenAI) не поддерживает новый Responses API — он работает только со старым Chat Completions. Если оставить on — все запросы будут падать с ошибкой про неподдерживаемый эндпоинт. Тоггл явно надо переключить в off, прежде чем тестировать.

Если развилка Responses API vs Chat Completions встречается впервые — короткое объяснение, что это за два эндпоинта и почему через прокси работает только один:

Responses API vs Chat Completions: два разных эндпоинта OpenAI

У OpenAI к одним и тем же моделям (gpt-4.1, gpt-4o и др.) есть сразу два эндпоинта-«входа» — старый /v1/chat/completions и новый /v1/responses. Они умеют разное и принимают запрос в разной форме, поэтому интеграции должны явно выбирать, по какому из них идти.

Зачем это нужно

OpenAI добавил Responses API позже, чтобы дать одной API-вызовом то, что раньше требовало нескольких походов: помнить разговор без явной отправки всей истории (stateful conversations), встроенные инструменты (web search, file search, code interpreter), нативные мультимодальные входы. На Chat Completions это всё приходилось склеивать руками снаружи.

Но Chat Completions никто не отключал. Большинство сторонних прокси и SDK к OpenAI-совместимым моделям пока поддерживают именно его — он стабильнее, проще по контракту и понятен всем библиотекам, выросшим вокруг ChatGPT.

Как это работает

Главное различие — структура запроса и ответа.

  • Chat Completions — стандарт «массив сообщений → одно сообщение». На вход массив messages с ролями system / user / assistant / tool, на выход одно choices[0].message. История разговора собирается на стороне клиента.
  • Responses API — модель stateful: можно передать previous_response_id, и OpenAI сам подтянет историю по своему ID. Tool-вызовы и встроенные инструменты описаны в более богатой схеме input / output items.

С точки зрения модели результат похожий, но это разные эндпоинты: запрос на формате Chat Completions, отправленный в /v1/responses, не пройдёт, и наоборот.

К примеру
Где это встречается

К примеру, в n8n-ноде OpenAI Chat Model есть тоггл Use Responses API. Включённый — нода шлёт запрос на /v1/responses, выключенный — на /v1/chat/completions. Иногда — настройка спрятана глубже, например в коде клиента, как параметр функции или флаг класса SDK.

Через OpenAI-совместимый прокси

Если запрос идёт не напрямую в api.openai.com, а через прокси (LiteLLM, OpenRouter, локальный гейтвей) — важна поддержка эндпоинта именно на стороне прокси. Часто реализован только /v1/chat/completions как самый распространённый стандарт; /v1/responses появляется позже и не у всех. В таких случаях интеграция должна явно работать в режиме Chat Completions — иначе все запросы будут падать с ошибкой про неподдерживаемый эндпоинт, хотя сама модель отвечать готова.

При прямом credential на api.openai.com доступны оба эндпоинта — можно выбирать по задаче (Responses ради stateful или встроенных tools, Chat Completions ради совместимости).

Типичные ошибки

  • Не заметили тоггл и оставили дефолт — нода или SDK по умолчанию выбирают Responses, при прокси без поддержки приходит ошибка эндпоинта; человек думает, что «не та модель» или «credential сломался».
  • Перенесли интеграцию с прямого OpenAI на прокси и не переключили режим — раньше работало на Responses, теперь падает; нужно явно переключить на Chat Completions.
  • Смешали схемы запросов — пытаются на /v1/chat/completions передать поля, специфичные для Responses (например, previous_response_id); они игнорируются или вызывают ошибку валидации.

Когда не подходит

Эта развилка не имеет значения, если работа идёт через высокоуровневый SDK или нода, которые сами абстрагируют эндпоинт и контролируют поддержку. Тогда выбор сводится к одному тогглу или вообще не появляется. А вот собирая собственный HTTP-вызов или подключая прокси — выбор эндпоинта стоит зафиксировать осознанно.

Как работает Postgres Chat Memory в n8n

Postgres Chat Memory — sub-нода для AI Agent в n8n, которая хранит историю сообщений сессии в таблице Postgres. На каждый вызов агента она читает последние N сообщений из таблицы по sessionId, передаёт их в контекст и дописывает свежую пару user → assistant обратно.

Зачем это нужно

Без памяти агент каждый ответ начинает с чистого листа: пользователь представился во втором сообщении, а агент в третьем уже забыл имя. Память делает диалог связным. Postgres-вариант удобен тем, что:

  • история переживает перезапуск воркфлоу и контейнеров n8n;
  • к таблице можно ходить SQL'ем — посмотреть, сколько сессий, что отвечал агент, аудит запросов;
  • работает несколько параллельных сессий, разделённых по sessionId.

Как это работает

  1. Нода подключается к AI Agent через разъём Memory (это sub-нода типа ai_memory).
  2. На первом вызове в указанной таблице она автоматически создаёт нужную схему (колонки id, session_id, message, created_at — конкретные имена зависят от версии ноды). Заранее CREATE TABLE руками не нужен.
  3. При каждом вызове агента нода фильтрует записи по текущему sessionId, забирает последние contextWindowLength пар сообщений и подкладывает их в контекст.
  4. После генерации ответа добавляет в таблицу две новые строки — user → ... и assistant → ....

Параметры, на которые смотрят

  • Session ID Type — From Connected Chat Trigger по умолчанию. Если воркфлоу — Sub-WF без Chat Trigger, нужно переключить на Define below и подставить sessionId руками (например, из триггера или из запроса).
  • Session Key — выражение, откуда брать sessionId. Часто ={{ $('Trigger').item.json.sessionId }}.
  • Table Name — имя таблицы в Postgres. Удобно держать в конфиге (DataTable) и подставлять через выражение.
  • Context Window Length — сколько последних пар сообщений подкладывать. Типовое значение — 10.

sessionId — частая ловушка

Если sessionId приходит как число (user_id из Telegram, integer из БД), а Postgres Chat Memory ожидает строку, под капотом ключи могут разойтись — две одинаковые «сессии», одна как 12345, другая как '12345', между ними память не делится. Поэтому в большинстве воркфлоу перед нодой ставят строку: String(sessionId) в Code-ноде или .toString() в выражении.

К примеру
Изолированные сессии

К примеру, если два пользователя пишут одному и тому же боту, у каждого свой sessionId (например, chat_id из Telegram). Память не пересекается: ответ первому не утечёт во вторую сессию.

Типичные ошибки

  • Забыли переключить Session ID Type в Sub-WF без Chat Trigger — нода падает на запуске с ошибкой «No connected Chat Trigger».
  • contextWindowLength слишком большой — каждый ответ подкладывает огромный контекст, токены тают, ответ становится медленнее; для большинства KB-сценариев 10 хватает.
  • Имя таблицы хардкодом в одном Sub-WF, выражением в другом — рассинхрон при переезде; имя таблицы — всегда конфиг, никогда литерал.
  • Несовпадение типов sessionId — два «вида» сессий, путаница в истории.

Когда не подходит

Если история диалога не нужна (каждый запрос — независимый, например, single-shot RAG ответ на одну страницу), память можно не подключать. Если хочется in-memory кэш на время одного воркфлоу — есть Buffer Memory (живёт ровно одно выполнение).

Qdrant Vector Store в режиме Retrieve As Tool

У ноды Qdrant Vector Store в n8n есть несколько режимов работы. Когда она подключается к AI Agent как инструмент, используется режим Retrieve Documents (As Tool for AI Agent). В этом режиме нода превращается в обычный tool, который LLM может вызвать когда сама посчитает нужным.

Чем отличается от Retrieve For Chain

В линейном режиме Retrieve Documents (For Chain) нода всегда выполняется — поиск идёт на каждом запросе, текущий чат-инпут используется как query, результат подмешивается в контекст.

В режиме As Tool поведение другое:

  • Поиск не обязателен на каждом запросе. LLM сама решает, нужна база знаний или ответ можно дать без неё.
  • Query формируется самой LLM через $fromAI(), а не берётся напрямую из чат-инпута. Модель переписывает вопрос пользователя в формулировку, которая лучше зайдёт в KB.
  • Нода появляется у агента в списке tools под именем, которое задаёт Tool Description — текст, который видит модель.

Параметры, на которые смотрят

  • Operation Mode — переключают на Retrieve Documents (As Tool for AI Agent).
  • Tool Description — текст, по которому LLM решает «звать ли инструмент». От качества этого описания зависит, попадёт ли вообще запрос в KB. Описание лучше держать в конфиге и менять без правки воркфлоу.
  • Qdrant Collection — имя коллекции в Qdrant. Если коллекция формируется студентом — переключают By ID и подставляют через выражение.
  • Limit (TopK) — сколько фрагментов вернуть на каждый поиск. Базовое значение — 5.
  • Include Metadata — оставляют включённым, чтобы вместе с текстом приходили метаданные чанка (источник, страница, теги).
  • Embedding — отдельный sub-разъём под нодой; туда вешают ноду эмбеддингов той же модели, через которую коллекция заполнялась. Иначе размерности не совпадут.

Типичные ошибки

  • Разные модели эмбеддингов между этапом загрузки и этапом поиска — векторы окажутся разной длины или из разных пространств, Qdrant вернёт пусто, агент будет отвечать «в базе знаний не нашёл» на любой вопрос.
  • Слабое Tool Description — описание вроде «база знаний» не подсказывает модели, когда инструмент звать. LLM звонит редко или не зовёт совсем. Подробнее — в отдельной статье про структуру Tool Description.
  • Metadata Filter с $fromAI — известный баг n8n: в tool-режиме $fromAI() не работает в Metadata Filter, фильтрация по динамическим значениям через инструмент пока не поддерживается. Если фильтр нужен — задаётся литералом, либо строится отдельный поиск через HTTP Request.
  • Слишком большой TopK — модель захлёбывается контекстом, ответ становится размытым. Базовое 5 подходит для большинства сценариев.
К примеру
Когда модель не зовёт KB

К примеру, на «Привет!» агент обычно отвечает приветствием и в базу не лезет — это нормально. Если включён Return Intermediate Steps, в массиве шагов будет видно, дёргался ли инструмент или нет.

Когда не подходит

Если на каждый запрос нужно гарантированно искать в KB (например, single-shot Q&A без диалога), удобнее линейный режим Retrieve For Chain — он проще и предсказуемее. As Tool — это про разговорного агента с инструментами.

Выражение $fromAI в n8n: как LLM сама заполняет параметры инструмента

$fromAI() — специальное выражение в n8n, которое говорит ноде-инструменту: «значение этого поля при вызове подставит сама LLM». Используется в полях нод, подключённых как Tool к AI Agent.

Зачем это нужно

У классической ноды все поля заполняет автор воркфлоу — либо литералом, либо выражением из предыдущих нод. Когда нода становится инструментом агента, появляется новый источник значений: сама модель. Например, у Qdrant Vector Store в режиме As Tool нужно понять — какой query пользователь сейчас задал, чтобы по нему искать в базе. Этот query модель формирует на лету: переписывает вопрос пользователя в поисковую формулировку.

Как выглядит

={{ $fromAI('query', 'Query text to search in knowledge base') }}

Первый аргумент — имя параметра, второй — описание, которое LLM видит и по которому решает, что туда положить. Часто в нодах-инструментах n8n такие поля помечены автоматически — нажав в режиме выражения, видно подсказку $fromAI(...).

К примеру
Где встречается

К примеру, в Qdrant Vector Store as Tool поле поискового запроса по умолчанию заполнено $fromAI('query', '...'). У других tool-нод (HTTP Request as Tool, MCP tools) — свой набор полей с $fromAI.

Где допустимо

  • В полях нод, подключённых как Tool к AI Agent.
  • Только там, где LLM по описанию tool'а понимает, что подставить.

Где не работает:

  • В обычных нодах вне tool-режима — $fromAI ничего не подставит, нода упадёт.
  • В Metadata Filter ноды Qdrant Vector Store в tool-режиме — известный баг n8n, фильтр не получает значения от LLM. В таких случаях фильтр пишут литералом или делают поиск через HTTP Request.

Типичные ошибки

  • Плохое описание второго аргумента — LLM не понимает, что в это поле положить. Описание должно быть конкретным: «Query text to search in knowledge base» лучше, чем просто «text».
  • Использовали $fromAI снаружи tool-режима — нода падает на запуске с непонятной ошибкой; $fromAI живёт только внутри инструмента агента.
  • Пытаются через $fromAI подставить значение, которое и так есть в воркфлоу — обычное выражение ={{ ... }} тут лучше: предсказуемо и не зависит от того, как LLM сформулирует.

Когда не подходит

Если значение в инструменте всегда одно и то же (или приходит из других нод воркфлоу), $fromAI не нужен. Он только для тех полей, которые именно модель должна формулировать на лету под конкретный запрос пользователя.

Параметризованный SQL в Postgres-ноде n8n

Параметризованный SQL — способ передать значения в запрос отдельным каналом, а не клеить их строкой. В Postgres-ноде n8n значения подставляются через placeholder'ы $1, $2, $3..., а сам массив значений идёт в поле Query Replacement (или Query Parameters — название зависит от версии ноды). Это базовая защита от SQL-инъекций и того, что значения «ломают» запрос специальными символами.

Зачем это нужно

Когда значения вставляются в SQL прямой конкатенацией строк, любая кавычка во входе ломает запрос или, хуже, позволяет переписать его ('); DROP TABLE users;--). С параметризованным запросом значения вообще не парсятся как SQL — Postgres получает запрос отдельно, значения отдельно, и подставляет их безопасно в момент выполнения.

Дополнительно — корректно обрабатываются типы (числа, NULL, JSON, timestamp), кавычки внутри строк не нужно экранировать, длинные значения не выглядят пугающе в логах.

Как это работает в n8n

  1. В поле Query пишется SQL с placeholder'ами $1, $2, ... вместо значений.
  2. В поле Query Replacement пишется одно выражение-массив, каждый элемент которого — значение под соответствующий $N.
INSERT INTO some_log (session_id, question, answer, response_time_ms, created_at)
VALUES ($1, $2, $3, $4, NOW())
={{ [
  $('Trigger').item.json.sessionId,
  $('Trigger').item.json.chatInput,
  $('Agent').item.json.output,
  $('Agent').item.json.responseTimeMs
] }}

Что важно

  • $1, $2, ... работают только для значений, не для имён таблиц/колонок. Имя таблицы или колонки нельзя передать через placeholder — Postgres всё равно его не подставит. Имя таблицы вставляется выражением {{ ... }} ДО отправки запроса в Postgres.
  • В n8n Query Replacement — это одно поле-выражение, которое возвращает массив. Не несколько отдельных полей через запятую.
  • Типы значений важны: число подаётся числом, NULL подаётся null в JS, строка строкой; n8n сериализует массив корректно, если значения — это родные JS-типы.

Типичные ошибки

  • Конкатенация значений в строку запроса ("WHERE chat_id = " + chatId) — это и есть классический путь к SQL-инъекции, плюс ломается на спецсимволах и кавычках.
  • Использовали $1 для имени таблицы — Postgres не подставит, запрос упадёт на синтаксической ошибке. Имя таблицы — через выражение {{ }} (с осознанием, что значение источника должно быть доверенным).
  • Массив-выражение разбили на отдельные поля — нода ждёт одно выражение, возвращающее массив, а не цепочку выражений.
  • Несоответствие количества $N и элементов массива — Postgres возвращает ошибку про «bind message supplies N parameters», debug по логам исполнения.
К примеру
Как ловить лишний $4

К примеру, если в SQL четыре $1..$4, а массив выражения содержит три элемента — нода упадёт на исполнении с сообщением про несовпадение количества параметров. Чинят сверкой запроса и массива по индексу.

Когда не подходит

Если запрос строго фиксированный и значения внутрь не подставляются (SELECT COUNT(*) FROM some_table), параметризация просто не нужна. Также для DDL-операций (CREATE TABLE, ALTER TABLE) placeholder'ы не подходят — DDL не использует параметры.

Динамическое имя таблицы через выражение в Postgres-ноде n8n

В Postgres-ноде n8n значения в запрос пробрасываются через $1, $2, $3, а вот имя таблицы так передать нельзя. Placeholder'ы PostgreSQL подставляют только значения, не идентификаторы. Поэтому имя таблицы вставляется через выражение {{ ... }} прямо в текст запроса — оно рендерится в строку до того, как запрос уходит в Postgres.

Зачем это нужно

Когда воркфлоу работает с несколькими таблицами или одна и та же логика применяется к разным окружениям (dev/staging/prod), имя таблицы удобно держать в конфиге (DataTable, переменные окружения) и подставлять выражением. Тогда смена таблицы — это правка одной строки конфига, без правки воркфлоу.

Как это работает

В поле Query выражение пишется прямо внутри SQL:

INSERT INTO {{ $json.config.log_table }} (col_a, col_b) VALUES ($1, $2)

При выполнении n8n сначала отрендерит {{ $json.config.log_table }} в строку (например, kb_agent_logs) и только потом отдаст полученный запрос в Postgres. Postgres увидит уже готовое имя таблицы.

Значения остаются на параметрах $1, $2 — это два разных канала: имя таблицы рендерится строкой, значения подставляются Postgres'ом отдельно.

Когда это безопасно

Имя таблицы — это не пользовательский ввод. Оно приходит из доверенного источника: DataTable, который заполняет автор, переменные окружения, конфиг проекта. Поэтому риск инъекции, как у значений, отсутствует — значение задано заранее и контролируется.

Если же имя таблицы приходит из ненадёжного источника (например, какой-то параметр запроса от пользователя), параметризация имени не поможет, и решать нужно на уровне дизайна — например, white-list допустимых имён, проверка перед запросом.

Типичные ошибки

  • Попытка через $1: INSERT INTO $1 (...) — Postgres ругается на синтаксическую ошибку, потому что в DDL/DML имена идентификаторов не параметризуются.
  • Жёстко зашитое имя в одном Sub-WF, выражение — в другом — рассинхрон при переезде на новую базу. Имя таблицы либо везде литералом (если оно одно), либо везде выражением.
  • Имя таблицы из user input без проверки — это уже отдельный класс уязвимостей; в простых сценариях лучше не делать.

Почему именно так

К примеру, в одном проекте хочется логировать запросы в kb_agent_logs, в другом — в support_agent_logs. Если имя в конфиге — меняется одна ячейка DataTable, воркфлоу остаётся прежним. Если имя зашито в ноде — каждый раз правят и проверяют SQL.

Если таблица одна и никогда не меняется, литерал в запросе проще читать и не требует выражения. Динамика имеет смысл там, где имя реально варьируется.

Как отредактировать строку DataTable через UI n8n

DataTable в n8n редактируется двумя способами: через ноду Data Table в воркфлоу или прямо через UI — открыть таблицу, кликнуть в ячейку, изменить значение, сохранить. Для разовых правок конфига UI быстрее.

Зачем это нужно

DataTable часто используется как простой конфиг-стор: одна строка — это конфиг для одного канала / окружения / клиента. Когда нужно поменять одно поле (например, обновить системный промпт агента или сменить модель), собирать отдельный воркфлоу ради этого избыточно.

Как это работает

  1. В n8n открыть раздел Data Tables в нужном проекте (он рядом с разделом Workflows).
  2. Кликнуть по имени нужной таблицы — откроется представление со списком строк и колонок.
  3. Найти строку, которую нужно поправить (визуальный фильтр или поиск, если их много).
  4. Кликнуть в ячейку — она перейдёт в режим редактирования. Если значение — длинная строка или JSON, n8n откроет редактор побольше.
  5. Изменить значение и нажать Save (или Enter в зависимости от типа ячейки).
К примеру
UI vs нода в воркфлоу

К примеру, разово сменить модель gpt-4.1-mini на gpt-4.1 для канала telegram — проще через UI. Если же нужно регулярно или массово обновлять строки (например, синхронизировать конфиги при каждом релизе), удобнее делать это через ноду Data Table в техническом workflow.

  • Сломали JSON в ячейке — DataTable не валидирует структуру внутри текстового значения. Если в ячейке хранится JSON-строка, после правки её нужно перечитать читалкой («это валидный JSON?»). Лучше править в редакторе ячейки (он подсветит структуру), не в краткой превью.
  • Не сохранили — кликнули в другую ячейку до сохранения, изменение потерялось. Привычка — сразу нажимать Enter / Save.
  • Несколько строк на один ключ — UI позволяет добавить дубликаты; если строки фильтруются по ключу, ноды могут читать любую из них. Перед правкой проверить, что строка одна.

Когда не подходит

Для регулярных или массовых операций UI неудобен — там собирают технический workflow с нодой DataTable в режиме Update / Upsert. UI хорош для разовых правок одной-двух ячеек.

🚩Publish и Active в n8n — в чём разницастатья · ~2 мин▾

Publish и Active в n8n — в чём разница

В n8n у воркфлоу есть два связанных, но разных понятия: «опубликован» (Publish) и «активен» (Active). Какой из них применить — зависит от типа воркфлоу.

Зачем это нужно

Для обычных воркфлоу с триггерами (Schedule, Webhook, Telegram) важно, чтобы триггер начал слушать события — иначе ничто не запустит. За это отвечает тоггл Active. Для Sub-Workflow триггер запускают не события снаружи, а вызовы из других воркфлоу — поэтому отдельный тоггл Active им не нужен. Их нужно «зафиксировать» как опубликованные, чтобы родительский воркфлоу мог их найти и вызвать.

Как это работает

  • Обычный воркфлоу с триггером (например, Schedule):
  • Publish (сохраняет версию) — фиксирует состояние, по которому n8n исполняет.
  • Active (тумблер) — включает/выключает работу триггера.
  • Чтобы воркфлоу действительно начал работать в проде, обычно нужны оба: и Publish, и Active.
  • Sub-Workflow с триггером Execute Workflow Trigger:
  • Active нет — этот триггер не слушает событий, он реагирует только на вызовы.
  • Publish — обязательно. Только опубликованную версию n8n исполняет при вызове из родителя.

В версиях n8n с разделением Draft / Published «опубликован» = «сохранён и доступен для исполнения». В Draft изменения видны только в редакторе.

К примеру
Sub-WF не активируется тумблером

К примеру, собрали Sub-Workflow, нажали Save, проверили в UI — пин, тестовый запуск, всё ок. Дальше — Publish. Родительский воркфлоу теперь может вызвать этот Sub-WF и получит свежую логику. Никакого Active искать не нужно.

Типичные ошибки

  • Забыли Publish — изменения в Sub-WF остались в Draft, родитель работает с прошлой версией. Проверяется в списке воркфлоу: рядом со Sub-WF не должно быть пометки Draft.
  • Ожидание Active у Sub-WF — путаница, ищут переключатель, которого нет. Sub-WF активируются вызовом, не тумблером.
  • Активировали воркфлоу с Webhook, не нажав Publish — некоторые версии n8n требуют сначала Publish, потом Activate; иначе webhook возвращает 404.

Когда не подходит

Эта пара (Publish / Active) — это про доступность и активацию, а не про деплой кода в другой инсталляции. Перенос между инсталляциями n8n идёт через экспорт/импорт JSON воркфлоу — это уже отдельная процедура.

Что конкретно сделать
  • Открой Sub-Workflow WF2: KB Master Agent.
  • В правом верхнем углу нажми Publish (Save).
  • Проверь, что в списке workflow он отображается как опубликованный (не Draft).

Готово ядро бота: Sub-Workflow с триггером Execute Workflow Trigger, оркестратором AI Agent, тремя sub-нодами вокруг него (модель, память, поиск по базе) и логированием в Postgres. Типичные ошибки
  1. В Chat Model тоггл Use Responses API по умолчанию ВКЛЮЧЁН — это самая частая причина «у меня всё падает». Школьный LiteLLM-прокси работает только через старый Chat Completions API, новый Responses API не поддерживает. Тоггл надо явно выключить.
  2. Если в Embeddings OpenAI под Qdrant KB указать ДРУГУЮ модель эмбеддингов, чем была в первой задаче серии (там — text-embedding-3-small) — Qdrant вернёт пусто на любой вопрос. Векторы окажутся разной длины и просто не сравнятся. Симптом: агент на любой вопрос отвечает «в базе знаний этого нет».
  3. Через школьный LiteLLM имена моделей пишутся БЕЗ префикса провайдера: gpt-4.1-mini (а не openai/gpt-4.1-mini), text-embedding-3-small (а не openai/text-embedding-3-small). С префиксом прокси не находит модель.
  4. В Log Request параметр Query Replacement ОБЯЗАН быть единым выражением-массивом ={{ [v1, v2, v3, v4] }}. Если расписать как четыре выражения через запятую — Postgres разломается на первой же запятой внутри ответа агента (а они там почти всегда есть), и порядок параметров поедет.
  5. Имя ноды-триггера When Executed by Another Workflow НЕ переименовывать. Code-нода Normalize Data ссылается на этот триггер по имени — переименуешь, и Normalize Data сломается.
  6. В Postgres Chat Memory по умолчанию Session ID Type стоит From Connected Chat Trigger — но у нас нет Chat Trigger в Sub-WF. Переключи на Define below и в Session Key подставь выражение из sessionId. Иначе память не найдёт ключ сессии.
  7. В Normalize Data обязательно приводи sessionId к строке через String(sessionId). Telegram отдаёт user_id числом, а Postgres Chat Memory хранит ключи как текст — без конверсии разные сессии могут смешаться.
  8. На ноде Get config row зайди на вкладку Settings (рядом с Parameters) и включи Always Output Data. Без этого тест 5 (несуществующий channel) не сработает корректно — нода тихо остановится, а Normalize Data не получит управление.
  9. В параметрах AI Agent System Message спрятан внутри раздела Options. Студенты обычно ищут его на верхнем уровне и не находят. Нажми Add Option → выбери System Message → пропиши выражение.
  10. В Edit Fields поле Include Other Input Fields по умолчанию ВКЛЮЧЕНО. Если оставить так — Sub-WF вернёт наружу ещё и success-ответ Log Request, что засорит контракт. Выключай — вернётся только {answer: ...}.
